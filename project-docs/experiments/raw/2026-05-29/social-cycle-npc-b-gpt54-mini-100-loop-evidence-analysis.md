## Claim Under Review

Original claim:

> The observation -> free Mineflayer action selection -> long-term settlement /
> social goal achievement loop is not strong yet.

More precise claim after data analysis:

> Observation is producing usable next actions, but those actions collapse into a
> narrow gather/craft/place loop and are not being consolidated into long-term
> settlement or social state.

The weakness is not "the NPC never acts after observing." It does act. The
weakness is that the verified low-level actions do not reliably become durable
settlement/social progress.

Interpretation:

The runtime retry constraint gate is working. It identifies repeated exact
target/args failure and blocks later matching attempts before Mineflayer
execution. That means the runtime is learning a negative action constraint, but
the planner still often needs the runtime to stop it rather than pivoting early
from observation and memory.

## 7. Final Answer To The Claim

The data supports the weak-loop claim, but the reason should be stated precisely.

Supported:

- The run has strong recording integrity.
- Observation produces next actions, often successful ones.
- The chosen actions are concentrated in a small gather/craft/place subset.
- Long-term settlement state does not absorb verified primitive evidence.
- Social behavior was effectively untested: no visible actors, no `say`, no
  relationship events.
- Retry constraints show repeated planner mistakes around exact failed
  target/args pairs.

Not supported:

- It is not true that the NPC cannot act.
- It is not true that observations are useless.
- It is not true that Mineflayer execution is locked down to only action skills;
  most cycle-level actions were direct primitives.

Best current diagnosis:

> The substrate can observe, act, verify, and remember. The weak part is the
> bridge from raw observation and verified local actions into durable
> settlement/social state and broader action selection.

## 8. Next Data-Driven Fix

The next implementation slice should not be another 100-cycle run. It should be
a smaller loop that proves one consolidation rule.

Recommended target:

1. Make `settlement_state` and `settlement_checklist` consume primitive
   `place_block` evidence for `crafting_table`.
2. Add a regression test where verified `place_block crafting_table` changes
   `crafting_table_known_or_placed` from `pending` to `satisfied`.
3. Run a 20-cycle deterministic or low-cost provider experiment and require that
   checklist item to be satisfied from primitive evidence.

That would directly test the missing bridge:

```text
observation -> action -> verifier evidence -> durable settlement state
```
