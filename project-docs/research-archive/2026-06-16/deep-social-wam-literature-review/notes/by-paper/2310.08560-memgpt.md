## Primary-source facts

- **Research question**: LLMs have fixed, finite context windows; extending context is
  quadratic in cost and long-context models "struggle to utilize additional context
  effectively" (cites Lost-in-the-Middle). How can a fixed-context model give the
  *illusion* of unbounded context for long conversations and long documents?
- **Core idea (virtual context management)**: borrow OS virtual-memory paging. Treat the
  context window as constrained physical memory; page data in/out between in-context
  ("main context") and external storage. The LLM manages its own memory through
  function calls (read/write external stores, edit its own context, decide when to reply).
- **Memory hierarchy (two tiers)**:
  - **Main context** = prompt tokens, three contiguous sections: (1) read-only system
    instructions (control-flow + memory-tier usage + function schemas), (2) **working
    context** = fixed-size read/write block of unstructured text (key facts, persona,
    user info), writeable only via function calls, (3) **FIFO queue** = rolling message
    history; its first slot holds a recursive summary of evicted messages.
  - **External context** = out-of-context data in databases: **recall storage** (the full
    message DB) and **archival storage** (read/write store for arbitrary-length text).
    Must be explicitly moved into main context to be seen by the LLM.
- **Queue Manager**: appends incoming messages to the FIFO queue, triggers inference,
  writes messages + outputs to recall storage. Eviction policy: at a "warning token
  count" (e.g. 70% of window) it injects a memory-pressure warning so the LLM can save
  important content to working/archival storage; at a "flush token count" (100%) it
  evicts a chunk (e.g. 50%) and regenerates the recursive summary. Evicted messages stay
  in recall storage and remain retrievable.
- **Function executor (self-editing/self-retrieval)**: memory edits and retrieval are
  "entirely self-directed": the LLM decides when to move items between tiers and updates
  working context to reflect its evolving understanding. The LLM output is parsed; if
  function args validate, the function runs; results (including runtime errors, e.g.
  "main context full") are fed back, forming a feedback loop. Retrieval paginates to
  avoid overflowing context.
- **Control flow / function chaining**: *events* (user messages, system warnings, user
  interactions, scheduled timed events) trigger inference. A `request_heartbeat=true`
  flag returns control immediately to the LLM so it can chain calls (e.g. multi-page
  retrieval); without it, MemGPT yields until the next event. Scheduled events let it
  run "unprompted."
- **Evaluation**: two domains where finite context cripples LLMs. (1) Document analysis:
  MemGPT processes documents far exceeding the underlying LLM window by paging relevant
  context. (2) Multi-session chat (deep memory retrieval + consistency): conversational
  agents that "remember, reflect, and evolve" across long interactions. MemGPT
  outperforms fixed-context LLM baselines in both.

## Interpretation (labeled inference)

- MemGPT is the canonical engineering pattern for **external long-term memory realized
  through tool-calling under a fixed context budget**. The mechanism the project needs
  for "memory commitments across cycles" is exactly this: the actor cannot hold the whole
  history in context, so important facts (an obligation, a debt, a relationship state)
  must be *written out* to an external store and *retrieved* when relevant.
- The self-edit feedback loop ("you are at capacity", retry) is structurally the same
  shape as the project's runtime-error feedback to the Actor Turn. MemGPT already proves
  that returning structured errors to the LLM and letting it re-decide works.
- Caveat: MemGPT's working context is *unstructured text the LLM rewrites*. The project's
  rule is that memory is context, not executable authority. So MemGPT's data-movement
  mechanism transfers, but its "the LLM owns what is true in working context" stance does
  not. In the repo, memory entries should reference runtime artifacts, not be free prose
  that the runtime then trusts.

## Mechanically useful vs research contribution

- **Mechanically useful (engineering, borrow directly)**:
  1. Two-tier memory (in-context working set + external recall/archival store) with
     explicit, schema-validated function calls to move data, mapped onto the actor's
     workspace JSON records the project already uses.
  2. The eviction + recursive-summary pattern for bounding cycle/transcript growth (the
     project lists context compaction as a substrate concern).
  3. Event-driven inference + function chaining via a heartbeat flag (a precedent for
     multi-step retrieval inside one Actor Turn without ceding control to the user).
- **Research contribution it is NOT for this repo**: MemGPT is infrastructure (memory
  management), not a predictive/advisory model. Per the shared contract, memory tooling is
  support, not the research contribution. Adopting MemGPT-style memory does not by itself
  produce any social-material prediction; it only realizes the *substrate* on which a
  remembered obligation can persist.
- **What to avoid / overclaim**: do not claim MemGPT-style memory makes the agent
  "socially consistent" or "trustworthy." It makes facts *persist and be retrievable*;
  whether the agent then honors a remembered obligation is a behavior the runtime must
  *verify* (inventory/container/chat delta), not something memory guarantees.

## Reproducibility

- **Partial**: code and data released (memgpt.ai; MSC-Self-Instruct, function-call-traces
  datasets). Method is clearly specified and widely re-implemented (Letta). But the
  headline chat/doc results depend on a strong function-calling base LLM and specific
  prompts, so exact numbers are model-dependent.

## WAM layer(s) informed

- **Social** and **Institutional** layers: remembered obligations/credit (social), and
  routines/relationship state persisting across cycles (institutional). It informs the
  *memory substrate* the social/institutional layers depend on, not a prediction model.
  No forward-consequence model is present (unlike a WAM); MemGPT manages state, it does
  not predict `o'`.
