## Primary-source facts (LaTeX-verified)

### DSPy
- **Programming model**: LM pipelines as "text transformation graphs" (imperative computation graphs where LMs are invoked through DECLARATIVE modules defined by signatures, not hand-written prompt strings). Modules are PARAMETERIZED: they learn by creating and collecting demonstrations how to apply prompting/finetuning/augmentation/reasoning.
- **The compiler = teleprompters**: an optimizer for DSPy programs that improves quality (or cost) of modules via prompting or finetuning. Typical teleprompters have three stages: (1) Candidate Generation: find all `Predict` modules; for each, generate candidate instructions, field descriptions, or (most importantly) DEMONSTRATIONS. `BootstrapFewShot` runs a teacher (or the zero-shot program) on training inputs, tracks multi-stage traces, and uses the program's METRIC to keep traces that pass and discard the rest -> bootstrapped demonstrations. (2) Parameter Optimization: each parameter now has a discrete candidate set; apply random search or Tree-structured Parzen Estimators (HyperOpt/Optuna), or finetune weights (`BootstrapFinetune`). "Typically, we are optimizing average quality using the metric with cross-validation over the training set or a validation set. This is applicable even with no labels for any stages, depending on the nature of metric." (3) Higher-Order: modify control flow, e.g. ensembles (bootstrap copies, run in parallel, reduce by majority vote).
- **Results (as stated)**: within minutes of compiling, a few lines of DSPy let GPT-3.5 and llama2-13b-chat self-bootstrap pipelines that beat standard few-shot prompting (generally by >25% and >65% respectively) and beat pipelines with expert-created demonstrations (by 5-46% and 16-40%). DSPy programs compiled to small open LMs (770M T5, llama2-13b) are competitive with expert-written prompt chains for proprietary GPT-3.5.

### MIPRO (the successor optimizer)
- **Problem**: optimize prompts for LM PROGRAMS (multi-stage pipelines) to maximize a downstream metric "without access to module-level labels or gradients." Factorizes into optimizing free-form INSTRUCTIONS + few-shot DEMONSTRATIONS of every module, plus credit assignment across modules.
- **Three strategies**: (i) program-and-data-aware instruction proposal; (ii) a STOCHASTIC MINI-BATCH evaluation function used to learn a SURROGATE MODEL of the objective (cheaper than full evaluation); (iii) a meta-optimization that refines how the LM constructs proposals over time.
- **Result (as stated)**: MIPRO outperforms baselines on five of seven diverse LM programs using Llama3-8B by as much as 13% accuracy. Released in DSPy.

## Interpretation (flagged as inference)

- DSPy/MIPRO is the most "engineering-grade" entry of the lane: a working framework where you declare a pipeline and a METRIC, and the compiler optimizes prompts + demonstrations against that metric with cross-validation. The metric IS the verifier in this framing. This is the cleanest blueprint for "make the verifier the optimization target."
- The decisive design choice is that the metric is user-supplied and can be ANY function, including a programmatic check. If the repo supplied the runtime verifier as the DSPy metric, the teleprompter would bootstrap demonstrations and select instructions that pass real physical checks, not an LLM judge.
- MIPRO's surrogate-model-of-the-objective (and AgentSquare's in-context performance predictor) are a recurring cost-reduction pattern AND a recurring risk: the surrogate, not the true verifier, gates candidates during search. If the surrogate is biased, search optimizes the surrogate's error. The honest practice is surrogate for ranking-during-search, true verifier for final acceptance.

## Mechanically useful vs research contribution

- **Mechanically useful**: (1) the declare-pipeline + supply-a-metric + compile pattern maps directly onto the repo: declare the Actor Turn / advisory-WAM as DSPy modules, supply the runtime verifier as the metric; (2) bootstrap-demonstrations-by-keeping-traces-that-pass-the-metric is a label-free way to build few-shot examples from successful verifier-passing cycles; (3) cross-validation over a validation set as the default selection rule; (4) ensembles as a higher-order optimization.
- **What would be overclaim**: DSPy/MIPRO results are on QA/math/multi-hop retrieval with metrics that are answer-checkers. Do not claim it tunes embodied social policy. The borrow is the framework pattern and the "metric = verifier" discipline.
- **Honest bound for thesis**: DSPy's own statement "optimizing average quality using the metric with cross-validation" is exactly the thesis requirement (held-out scoring). The catch is MIPRO's surrogate: when the loop is scaled for cost, the surrogate can become the de facto scorer. Keep the runtime verifier as the FINAL gate; use surrogates only to triage.
