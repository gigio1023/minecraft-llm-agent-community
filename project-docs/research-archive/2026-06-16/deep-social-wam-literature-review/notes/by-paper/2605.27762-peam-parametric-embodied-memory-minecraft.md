## Primary-source facts (what the paper states/shows)

- Problem: LLM embodied agents keep memory non-parametric. Past trajectories, reflections, skills are stored externally and re-injected at inference. The agent's parametric policy never changes even after repeated attempts at the same craft chain. Cost grows with deployment: every recall consumes context budget, retrieval and prompt construction add latency, external experience must be re-introduced on each use.
- Claim framing: this is a "missing consolidation pathway", not a failure of retrieval-augmented memory. The motivation is cognitive (complementary learning systems theory: a fast sparse episodic store plus a slow distributed parametric store, coupled by offline consolidation).
- Architecture: two-tier embodied agent.
  - Slow deliberative LLM pi_slow (Azure GPT-4o): open-ended reasoning, curriculum proposal, code synthesis, outcome verification.
  - External episodic store E: stages successful and corrected trajectories.
  - Fast parametric module pi_fast: multimodal Mixture-of-Experts LoRA over Qwen3-VL-8B-Instruct, with per-category physically isolated adapters {theta_c} for c in {craft, gather, combat}. Executes consolidated skills reflexively.
  - Inference: try fast path first (route task to category adapter, generate code, verifier checks trajectory). If no adapter applies or verification fails, fall back to pi_slow; the slow tier's successful or corrected trajectory is written to E as a future consolidation candidate.
- Three design choices.
  1. Failure as a first-class training signal. Trains on failure-correction trajectory pairs D_cpair = {(x, tau_f, tau_c, c)} (tau_f fails, tau_c later succeeds under matched context x). Loss = BC on successful trajectories + lambda * trajectory-level DPO (corrected = chosen, failed = rejected), restricted to the adapter selected by skill category, lambda = 1.0. The BC term is "load-bearing": DPO supplies a relative preference but no absolute likelihood signal for the required async function name(bot){...} wrapper format.
  2. Per-category parameter isolation (forgetting resistance is structural, not just regularization). Internalizing a craft skill cannot modify the combat adapter.
  3. Formalized what and when to consolidate. Parameterization-worthiness score PV(s) = w1 U_cost + w2 U_stab - w3 P_redun - w4 R_forget (cost saving, stability, redundancy penalty, interference proxy). Self-triggered consolidation STC: a scale-free statistical trigger that fires when the recent-vs-baseline failure-rate z-score Z(s) > z_alpha AND PV(s) in top-q. Each skill is judged against its own historical baseline, no task-specific absolute failure threshold. Hyperparameters W = B = 10, alpha = 0.05, q = 0.5.
- Numbers as stated (main table, 11 tasks, 3 seeds, 33 trials):
  - Task success: PEAM 69.7% (23/33, Wilson 95% CI [0.530, 0.834]) vs VOYAGER 54.5% (18/33), +15.2 pp, McNemar paired p = 0.018. Beats strongest retrieval baseline (Optimus-1-rep. 60.6%) by +9.1 pp.
  - Efficiency: median per-call latency 5.5s (VOYAGER) -> 3.2s (PEAM, -42%); tokens/task ~31,200 -> ~4,600 (-85%). The token/latency drop is the operational consequence of removing per-call skill-library re-injection.
  - Forgetting (sequential consolidate craft -> gather -> combat, re-measure early craft skills): PEAM 0% measurable cross-category forgetting; single shared LoRA loses 32.4%; EWC loses 43.3%; naive full fine-tune loses 78.5%.
  - Ablations: pure-DPO (no BC) generates wrapper-format-correct code 0/12 vs joint BC+DPO 12/12; held-out reward margin +6.51 (DPO-only) -> +37.92 (joint). MoE isolation vs shared LoRA: forgetting 0% -> 32.4% over two consolidations. STC vs fixed schedule: 37% fewer cycles to matched performance, delay 23 -> 7 episodes.
  - Trigger robustness: same STC hyperparameters across craft-heavy and combat-heavy re-sliced distributions; top-10 PV overlap Jaccard 0.538 (synthetic re-slice), 0.61 (real paired collection).
- Execution substrate: VOYAGER's Mineflayer-based execution framework, Minecraft 1.19, code-as-action pipeline, environment-side verifier confirms task completion. The action interface is held fixed; only the memory architecture changes.
- Methodology findings the authors flag: forward-pass preference margin does not predict generate-path deployability; quantized on-device serving has deployment-specific failure modes; navigation yields no usable failure-correction pairs (failures are environmental, not code-level); trajectory re-slicing substitutes for cross-distribution evaluation.

## Interpretation (my inference, labeled)

- This is the single most on-target source for the lane: it is long-horizon memory and continuity, set in Minecraft, on the exact VOYAGER/Mineflayer substrate the repo uses, with the verifier confirming task completion before any trace is allowed into the consolidation pool.
- The verifier-gates-consolidation design is the load-bearing transfer to this repo: PEAM requires environment-side verification before a trajectory becomes training signal. That is the same admissibility gate the repo's deterministic runtime verifier provides, only PEAM uses it to admit experience into parameters rather than into a scored prediction.
- The "failure as a first-class signal" and failure-correction-pair construction (4-stage pipeline: failure identification, correction matching within Delta episodes, discrete pre-action context matching, quality gate) is a concrete recipe for mining the "convincing-wrong" / corrected negatives that an advisory predictor would need.
- The forgetting result (0% cross-category via physical isolation vs 32-78% for shared/regularized) is a clean argument that continuity across many cycles needs structural isolation, not just a growing log. But note: this is a parametric-memory claim. The repo's WAM is advisory and never selects actions, so PEAM's parametric internalization is exactly the side the repo does NOT adopt as runtime authority.

## Mechanically useful vs research contribution

- Mechanically useful: verifier-before-consolidation gate; failure-correction pair mining (the 4-stage matching with discrete pre-action state vector = inventory composition, biome, time-of-day bucket, quantized location); STC's scale-free self-triggered consolidation (judge a skill against its own historical failure baseline, no hand-tuned threshold) as a model for when to update any cross-cycle memory; the efficiency framing (parametric path removes per-call re-injection cost).
- Research contribution it is NOT for this repo: PEAM's contribution is moving experience into parameters of an acting policy. The repo forbids the predictor from selecting actions, so the parametric-skill-execution half is out of scope as runtime authority. PEAM also has no social or material-economic layer: skills are craft/gather/combat physical actions. There is no relationship ledger, obligation, or institutional state. PEAM is a single-agent physical-layer continuity result.
