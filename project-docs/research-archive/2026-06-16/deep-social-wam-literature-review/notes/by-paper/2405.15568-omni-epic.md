- **venue/source**: ICLR 2025; arXiv 2405.15568
- **arxiv_id**: 2405.15568

## Primary-source facts (what the paper states/shows)

- **Research question**: prior open-ended methods (POET, Enhanced POET, OMNI, Voyager) are confined to a narrow, pre-parameterized environment space. How do you generate *any* learnable-and-interesting environment, including its world, its reward, and its termination, toward "Darwin Completeness" (the ability to generate any possible learning environment)?
- **Architecture (the full autoresearch-shaped loop)**:
  - **Task Archive**: an unbounded, ever-growing store of natural-language task descriptions + their environment code, tagging each as successfully learned or attempted-but-failed. Seeded with a few task descriptions.
  - **Task Generator** (LLM): proposes the next "interestingly new" task in natural language. Uses RAG to retrieve the *most similar* archive tasks (successes and failures) as in-context stepping stones (most-similar, not most-different, so prior tasks scaffold the next). Guided by the LLM's internalized MoI.
  - **Environment Generator** (LLM): translates the task description into executable Python implementing the standard Gymnasium API: `reset`, `step`, `reward`, `terminated`. Compilation errors (with traceback) are fed back for up to 5 self-repair iterations; uncompilable tasks are discarded.
  - **Post-Generation MoI** (LLM): a *second* interestingness check, comparing the new task against the most similar archive tasks (RAG), because (a) the generator cannot hold the whole archive in context and (b) it is easier for an FM to judge a solution than to produce one. Uninteresting tasks are discarded.
  - **RL training**: DreamerV3, one specialist agent per task; new tasks continue from the policy of the nearest already-solved task (transfer). R2D2 robot, 6 discrete actions, PyBullet simulator.
  - **Success Detector**: the crux. A "universal reward function" that decides whether *any* task was completed. VLMs were found "not yet accurate enough" as success detectors, so they use **LLM-generated code** `get_success`. Crucially, `get_success` is kept *separate* from the shaped `reward` function: the reward is complex (to drive efficient learning, hence gameable), but the success-checker is simpler and "less susceptible to reward hacking, as it does not affect how the agent learns." Solved tasks enter the archive; failed tasks are iterated up to a max, then archived as failures.
- **Metrics**: (1) task-archive diversity = cell coverage of PCA-reduced task embeddings in a 2-D grid (a MAP-Elites-style coverage measure); (2) **ANNECS-OMNI** = ANNECS (novel + eventually-solved) plus a third criterion: the task must be FM-judged interesting vs the archive.
- **Results**: OMNI-EPIC significantly beats two controls (no archive; no MoI) on both diversity and ANNECS-OMNI (p < 0.05, Mann-Whitney). ANNECS-OMNI rises monotonically "for as long as we could afford to run it," i.e. no stagnation. Code fully open-sourced.
- **Stated limitations**: not yet Darwin Complete (confined to one simulator, PyBullet); cannot rule out that generated environments resemble training data; trains specialists, no generalization evidence; success detector is auto-generated code.

## Interpretation (flagged as inference)

- OMNI-EPIC is the single closest precedent for the wave-4 thesis applied to *this* repo: an LLM auto-generates situations *and the code that checks them*, an archive of successes/failures scaffolds the next situation, and a separate success-checker (not the reward, not the LLM's prose) decides "solved." The architecture is almost isomorphic to the repo's cycle: scenario generation -> environment/scenario spec -> Actor Turn + action -> verifier evidence -> archive of solved/failed.
- The **reward/success separation** is the precise mechanism the repo already enforces, validated here from the open-endedness side: OMNI-EPIC says the success-checker must not be the thing that shapes learning, or it gets hacked. The repo says the LLM proposes and the runtime verifier owns truth. These are the same principle. The repo's position is *stronger*: its verifier is hand-authored runtime code that owns physical truth, whereas OMNI-EPIC's `get_success` is LLM-generated code nobody verified against ground truth.
- The honest bound: OMNI-EPIC's `get_success` being LLM-written is its weak point and the repo's opportunity. If a generated social-material scenario's success function were also LLM-written, the loop could quietly grade itself wrong. The defensible repo design uses the *pre-built, hand-authored* runtime verifier to label generated scenarios, and only generates the *scenario setup* (who has what, what is scarce, who needs whom), not the success oracle.

## Mechanically useful vs research contribution

- **Mechanically useful**: (1) the **archive (successes + failures) -> RAG-most-similar stepping-stone retrieval -> generate next task** loop is directly portable to social-material scenario generation; the repo can keep an archive of verifier-labeled scenarios and condition the generator on the nearest ones. (2) The **reward/success separation** is a design rule to enforce: generate the scenario, but never let the generator author its own success label; use the runtime verifier. (3) **ANNECS-OMNI** is the progress meter (novel + solved + interesting); the repo can keep the first two verifier-grounded and treat the FM-interesting criterion as advisory only. (4) The **self-repair-on-compile-error** loop (feed traceback back, max 5 tries) is a cheap robustness pattern for generated scenario code.
- **Not a contribution to copy / overclaim risk**: the repo must not adopt an LLM-generated success detector; that would be progress laundering by another name. "Darwin Completeness" / "any environment" framing is aspirational; the repo's claim should be bounded to "auto-generate social-material scenarios whose `(state, action, next-state)` the existing runtime verifier can label." Specialist-per-task training and the lack of generalization evidence are also not goals to import.
- **Limitation (theirs + spotted)**: success detector is unverified LLM code; possible training-data leakage; no generalization; the third (interestingness) criterion is FM self-judged, so ANNECS-OMNI's "interesting" axis is contestable. None of OMNI-EPIC's tasks involve possession, obligation, or multi-actor social cost - it is single-agent skill acquisition.

## WAM layer(s) informed

Method-level, cross-layer, but most directly Institutional/settlement (the generation+archive loop) and Social (it can, in principle, generate multi-condition social scenarios as code). Strongest where the success-checker is clean (Physical/Material); the repo's contribution is to keep the success-checker = runtime verifier and generate only the scenario.
