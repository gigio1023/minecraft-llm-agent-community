# EoH and ReEvo: LLM-guided evolution of heuristics for combinatorial optimization

Two LLM-evolution methods covered together: both evolve CODE for a heuristic graded
by a combinatorial-optimization cost, both sit between FunSearch and AlphaEvolve, and
both add a natural-language reflection/"thought" channel to plain code evolution.

- **EoH**: Fei Liu, Tong Xialiang, Mingxuan Yuan, Xi Lin, Fu Luo, Zhenkun Wang, Zhichao Lu, Qingfu Zhang, "Evolution of Heuristics: Towards Efficient Automatic Algorithm Design Using Large Language Model", ICML 2024. arXiv 2401.02051. Code: https://github.com/FeiLiu36/EoH
- **ReEvo**: Haoran Ye, Jiarui Wang, Zhiguang Cao, Federico Berto, Chuanbo Hua, Haeyeon Kim, Jinkyoo Park, Guojie Song, "ReEvo: Large Language Models as Hyper-Heuristics with Reflective Evolution", NeurIPS 2024. arXiv 2402.01145. Code: https://github.com/ai4co/reevo
- **arxiv_id**: 2401.02051 (EoH), 2402.01145 (ReEvo)
- **source availability**: latex (both tarballs; deep-read method sections)

## Primary-source facts (LaTeX-verified)

### EoH (2401.02051)
- Evolves BOTH a natural-language "thought" (a high-level heuristic idea) AND its Python code implementation, jointly, in an evolutionary loop. Each individual = (description, code, fitness). The individual is a HEURISTIC for a problem, not a solution to it.
- Five prompt strategies mimic human heuristic design: E1 (generate a heuristic as different as possible from parents), E2 (share parents' common idea but add new parts), M1 (modify one heuristic), M2 (modify its parameters only), M3 (simplify by removing redundant components). Population N; each strategy called N times per generation (up to 5N new heuristics), then top-N kept. Rank-based selection p_i proportional to 1/(rank_i + N).
- Fitness = run the heuristic on a SET of problem instances (not one), like irace/SMAC-style AHD. Needs no hand-crafted components or model training.
- Result (abstract): on three classic combinatorial-optimization benchmarks, EoH outperforms hand-crafted heuristics and prior automatic-heuristic-design methods INCLUDING FunSearch; on online bin packing, EoH's heuristic significantly beats human baselines at a LOW query budget (fewer LLM calls than FunSearch).

### ReEvo (2402.01145)
- "Language Hyper-Heuristic" with Reflective Evolution. Two LLM roles: a GENERATOR LLM (proposes heuristic code) and a REFLECTOR LLM (gives verbal guidance). Each individual is a code snippet adhering only to a function signature (no fixed encoding).
- Five sequential steps per iteration: selection (random parent pairs, avoid equal-fitness pairs), SHORT-TERM reflection (reflector compares the two parents' relative performance and gives improvement hints), crossover (generator makes an offspring given parents + their relative performance + the short-term reflection), LONG-TERM reflection (reflector summarizes accumulated short-term reflections into persistent hints), elitist mutation (generator improves the current best using long-term reflections).
- The reflections are framed as "verbal gradients" that smooth the fitness landscape; long-term reflections accumulate experience without context blowup. Handles BLACK-BOX combinatorial problems (the reflector infers problem settings and how node/edge attributes correlate with the objective).

## Interpretation (flagged as inference)

- EoH and ReEvo show that adding a NATURAL-LANGUAGE reasoning channel (a "thought" / a "reflection") on top of code evolution improves sample efficiency: the LLM reasons about WHY one heuristic beat another and proposes the next edit accordingly, rather than blind mutation. This is the discovery-loop analog of the wave-4 EvolveR "distill principles from transcripts" idea, applied to code rather than memory.
- Both still grade by an exact combinatorial cost on a set of instances. The reflection improves the PROPOSER; the SCORE is still the exact evaluator. So they reinforce, not relax, the lane's bound: the reasoning channel does not remove the need for a cheap exact evaluator.
- The short-term/long-term reflection split is a concrete, cheap mechanism the repo could borrow for its discovery loop: short-term = compare two candidate advisory-WAM rules on the same transitions and verbalize why one verified better; long-term = accumulate a persistent list of "what tends to make a rule verify" without growing context unboundedly.

## Mechanically useful vs research contribution

- **Mechanically useful**: the thought-plus-code joint representation (EoH) and the generator/reflector split with short-term + long-term reflections (ReEvo), retargeted to evolving advisory-WAM rules or `author_mineflayer_action` skills scored by the runtime verifier; the "evaluate on a SET of instances, not one" rule (EoH) so a discovered rule is not overfit to a single scenario (directly answers the EvoTrace overfitting bound, [[2605.20086-what-evolutionary-coding-agents-evolve]]); the verbal-gradient reflection as a cheaper alternative to RL fine-tuning.
- **Not a contribution to copy**: the specific TSP/bin-packing heuristics and the EoH-beats-FunSearch claim are the contributions. The repo borrows the reflection-augmented loop structure.
- **Bound**: both need an exact cost function on a set of instances. Physical/Material transfer is clean; no social-cost analog.
