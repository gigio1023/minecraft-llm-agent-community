# MineWorld — real-time open-source interactive Minecraft world model

- **title**: MineWorld: a Real-Time and Open-Source Interactive World Model on Minecraft
- **authors**: Microsoft Research
- **year**: 2025
- **venue/source**: arXiv
- **arxiv_id**: 2504.08388
- **urls**: paper https://arxiv.org/abs/2504.08388 ; code + model released (Microsoft); LaTeX at `papers/latex/2504.08388/`
- **source availability**: latex (full); code + weights released per paper

## Primary-source facts

- **Model object**: a pure **world model** (not a WAM): models `p(x_{i+1} | x_{<i}, a_i)` — predict the next visual game state given past frames and the current action. It does NOT jointly select actions; the action is an input condition.
- **Architecture**: visual-action **autoregressive Transformer**. A **VQ-VAE image tokenizer** (16x spatial compression, initialized from aMUSEd, finetuned on Minecraft) turns each frame into discrete tokens; an action tokenizer turns mouse/keyboard actions into discrete tokens; the two are interleaved and trained with next-token prediction.
- **Efficiency**: a **parallel decoding** algorithm predicts spatially-redundant tokens together -> >3x speedup over standard AR decoding -> **4–7 frames/sec** real-time interaction.
- **Efficiency bottleneck stated**: SoTA video tokenizers produce "40k to 160k tokens for 16 frames," causing substantial inference cost — the explicit reason pixel-video world modeling is slow.
- **Evaluation / controllability**: applies an **inverse-dynamics model (IDM, from VPT)** to predicted consecutive frames to infer the executed action, then measures accuracy of inferred vs ground-truth input action — "controllability." Plus standard video-quality metrics. Outperforms SoTA diffusion-based open Minecraft WMs.

## Interpretation

- MineWorld is the canonical **open-source pixel Minecraft world model** and a clean example of `p(o'|o,a)` (a learned simulator), not `p(o',a|o,l)` (a WAM). It predicts the next frame; it never decides what to do. To get a "policy" you would bolt on planning/search over its rollouts.
- Its own stated bottleneck (40k–160k tokens/16 frames; needs a custom parallel decoder just to hit 4–7 fps) is direct evidence that pixel state is an expensive representation. For a Minecraft social runtime, predicting a typed social-material delta is orders of magnitude cheaper than predicting frames.
- The IDM-based controllability metric is the same trick used by the WAM survey's "Wow, wo, val!" — it measures whether the generated video carries enough action information, not whether the underlying (possession/obligation) state is right.

## Mechanically useful vs research contribution

- **Useful**: a ready open-weights **pixel Minecraft simulator** if a visual rollout sidecar were ever wanted (human review, not runtime authority). The IDM-controllability idea is a reminder that "did the predicted future encode the action" is a real check — though the project should check *structured* state consequences, not pixel-decoded actions.
- **Avoid/overclaim**: MineWorld is a pure WM, not a WAM, and is pixel-only; it has no concept of possession, obligation, or social state. Do not treat it as a social predictor or as runtime authority.

## WAM layer(s) informed

Physical (visual); example of a pure WM `p(o'|o,a)` and of pixel-modeling cost.
