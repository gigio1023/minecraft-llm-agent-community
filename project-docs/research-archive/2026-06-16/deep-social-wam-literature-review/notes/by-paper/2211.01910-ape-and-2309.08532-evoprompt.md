# APE (Automatic Prompt Engineer) + EvoPrompt (LLMs + Evolutionary Algorithms)

Two foundational prompt-optimization papers covered together; both treat the prompt
as a "program" searched against a score function.

- **APE title**: Large Language Models Are Human-Level Prompt Engineers
- **APE authors**: Yongchao Zhou, Andrei Ioan Muresanu, Ziwen Han, Keiran Paster, Silviu Pitis, Harris Chan, Jimmy Ba (University of Toronto, Vector Institute)
- **APE year/venue**: 2022 (arXiv 2022-11-03); ICLR 2023
- **APE arxiv_id**: 2211.01910 ; code https://github.com/keirp/automatic_prompt_engineer
- **EvoPrompt title**: Connecting Large Language Models with Evolutionary Algorithms Yields Powerful Prompt Optimizers
- **EvoPrompt authors**: Qingyan Guo, Rui Wang, Junliang Guo, Bei Li, Kaitao Song, Xu Tan, Guoqing Liu, Jiang Bian, Yujiu Yang (Tsinghua, Microsoft Research)
- **EvoPrompt year/venue**: 2023 (arXiv 2023-09-15); ICLR 2024
- **EvoPrompt arxiv_id**: 2309.08532
- **source availability**: latex for APE (abstract + conclusion deep-read); EvoPrompt (tarball extracted, abstract-level read; method cross-confirmed via PromptBreeder's related-work description)

## Primary-source facts (LaTeX-verified)

### APE
- **Method**: treat the instruction as the "program," optimized by searching over a pool of instruction candidates PROPOSED BY AN LLM to maximize a chosen score function. To score a candidate instruction, evaluate the zero-shot performance of another LLM following it. APE explicitly defines its scope: "optimizing the language in a prompt" -- it does NOT chain multiple LLM queries or use external tools (so it is narrower than ADAS/AFlow/GPTSwarm).
- **Results (as stated)**: APE-generated instructions match or beat human-annotator instructions on 24/24 Instruction Induction tasks and 17/21 curated BIG-Bench tasks; they improve few-shot performance (prepended to ICL), find better zero-shot chain-of-thought prompts, and can steer toward truthfulness/informativeness.
- **Framing**: prompt engineering as a black-box optimization problem solved by LLM-guided search.
- **Known limit (reported by later work)**: PromptBreeder cites APE finding "diminishing returns to further selection rounds as the quality seems to stabilize after three rounds," so APE abandoned iterative selection.

### EvoPrompt
- **Method**: connect LLMs with evolutionary algorithms (EAs). The EA proposes the search operators (mutation, crossover) and the LLM executes them on natural-language prompts; fitness is task performance. EvoPrompt is initialized with a whole population of hand-designed task-tailored prompts (vs PromptBreeder's single problem description), and uses fixed mutation/crossover prompts (vs PromptBreeder's self-referentially evolved ones). It includes an operator that asks for a mutant of the DIFFERENCE between two parent prompts.
- **Result (as stated)**: combining LLMs with EAs yields prompt optimizers that outperform human-engineered prompts and prior automatic methods across language understanding and generation tasks.

## Interpretation (flagged as inference)

- APE is the origin point of the "prompt-as-program, search to maximize a score function, evaluate via a held-out LLM" pattern that OPRO, PromptBreeder, EvoPrompt, and DSPy all build on. Its 24/24 + 17/21 results established that automatic prompt search reaches human-level prompt engineering.
- APE's reported "diminishing returns after ~3 rounds" is an early, honest signal that naive iterative prompt search plateaus -- the gap PromptBreeder/EvoPrompt close with diversity-maintaining evolution. It is a useful caution: prompt search is not unbounded free improvement.
- EvoPrompt is the "classical EA operators, LLM as the mutation/crossover engine" point; PromptBreeder generalizes it by evolving the operators themselves.

## Mechanically useful vs research contribution

- **Mechanically useful**: (1) the minimal APE loop (LLM proposes instruction candidates, score each on held-out data, keep the best) is the simplest possible prompt optimizer to apply to the Actor Turn / advisory-WAM prompt; (2) EvoPrompt's EA operators (mutation, crossover, difference-mutant) give more search structure when the simple loop plateaus; (3) the shared discipline: a chosen SCORE FUNCTION on held-out evaluation is the optimization target.
- **What would be overclaim**: both optimize a single instruction string for QA/understanding/generation, with APE explicitly excluding tool use and multi-step pipelines. Do not present them as agent-design or embodied methods. They are the prompt-text layer only.
- **Honest bound for thesis**: APE's own diminishing-returns finding warns that prompt search plateaus and can overfit the score; the discipline is held-out scoring + diversity. For this repo, use these as the cheapest prompt-tuning loop, always scored by the runtime verifier on held-out scenarios, advisory and verifier-gated.
