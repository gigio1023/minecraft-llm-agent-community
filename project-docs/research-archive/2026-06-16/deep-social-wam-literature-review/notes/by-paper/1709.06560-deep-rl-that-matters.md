## Primary-source facts

- Thesis: reproducing deep RL results "is seldom straightforward." Reported results vary widely for
  the SAME baseline algorithm because of both EXTRINSIC factors (hyperparameters, codebases) and
  INTRINSIC factors (random seeds, environment stochasticity). The paper runs controlled experiments
  on policy-gradient methods (ACKTR, PPO, DDPG, TRPO) in MuJoCo (Hopper-v1, HalfCheetah-v1).
- The signature result (random seeds): run 10 trials with identical hyperparameters, differing only
  in random seed, then split into two groups of 5 and average each. On HalfCheetah the two
  same-algorithm, same-hyperparameter groups produce statistically DIFFERENT distributions purely
  from the seed split: a 2-sample t-test across the training distribution gives t = -9.0916,
  p = 0.0016. "It is possible to get learning curves that do not fall within the same distribution at
  all, just by averaging different runs with the same hyperparameters, but different random seeds."
- A run is NOT the same as a fixed seed: "fixing the seed may not be able to control all sources of
  randomness such as non-determinism of ML frameworks with GPUs." (Same point rliable later makes.)
- Selective reporting is called out: it is "not uncommon for the top-N trials to be selected from
  among several trials" or to average over N < 5. The paper shows this is potentially misleading.
- Codebases matter: "default" implementations of the same algorithm diverge; network architecture and
  activation choices (tanh vs ReLU vs Leaky ReLU) and reward scaling each move results substantially
  and inconsistently across algorithms/environments.
- Reward as a misleading proxy (progress laundering analogue): on Swimmer-v1 a local optimum (curl up
  and flail without swimming) yields a return of ~130, so "learning curves can indicate successful
  optimization ... when in reality the returns achieved are not qualitatively representative of
  learning the desired behaviour." Recommendation: show video/behavior, not only returns.
- Recommended statistics: bootstrap confidence intervals; bootstrap POWER ANALYSIS to decide how many
  trials are needed (give the sample a uniform lift, e.g. scale by 1.25, and check what fraction of
  bootstrap simulations come out significant; a low fraction means more trials are required); and
  significance measures (2-sample t-test, Kolmogorov-Smirnov, bootstrap percent differences) used
  WITH confidence intervals. Worked example: ACKTR vs DDPG on Walker2d gives t = 1.03, p = 0.334,
  KS = 0.40, p = 0.697, bootstrap percent difference 44.47 percent (-80.62 percent, 111.72 percent),
  i.e. NOT significant despite a higher mean.

## Interpretation (my inference, labeled)

- This is the founding statement of the few-seed variance problem that
  [[2108.13264-rliable-statistical-precipice]] later operationalizes. For the repo it carries two
  warnings. First, a social run is at least as stochastic as an RL run (LLM sampling temperature,
  partner-agent behavior, world seed), so two batches of 5 social runs can land in different
  distributions for no reason but the seed, and a single mean delta-prediction number is not
  trustworthy. The repo must report variance, CIs, and run count, and should do a power-analysis-style
  check before claiming "predictor A beats predictor B."
- Second, the Swimmer local-optimum result is the RL twin of the repo's central failure mode,
  progress laundering: a metric (return, or "task success") can look like learning while the agent did
  not do the intended thing. The repo's answer is the same in spirit but stronger: the verifier checks
  the actual world delta (inventory/container/ledger), so "success" cannot be faked by a proxy. The
  repo should additionally keep behavior traces (the paper's "show demonstrations") so an evaluator can
  see what happened, not just a score.

## Mechanically useful vs research contribution

- Mechanically useful:
  - The reporting discipline (report all seeds, all hyperparameters, all implementation details; use
    bootstrap CIs and power analysis to set trial count) is a ready checklist for the repo's
    predictor-evaluation protocol.
  - The "show behavior, not just the metric" rule maps onto the repo's existing artifact/trace
    requirement and reinforces it as an evaluation necessity, not a nicety.
- Research contribution (not the repo's to claim): the controlled reproducibility study and its
  seed/codebase/reward-scale findings are 2018 continuous-control results. The repo borrows the
  warnings and statistics; it does not re-derive them. Its own surface is whether the SAME variance
  story holds for stochastic LLM social runs (likely yes, and worth measuring), and whether the
  verifier closes the progress-laundering hole that reward proxies leave open (the repo's claim).
