# EvolveR - Self-Evolving LLM Agents through an Experience-Driven Lifecycle

- **title**: EvolveR: Self-Evolving LLM Agents through an Experience-Driven Lifecycle
- **authors**: Rong Wu, Xiaoman Wang, Jianbiao Mei, Pinlong Cai, Daocheng Fu, Cheng Yang, Licheng Wen, Xuemeng Yang, Yufan Shen, Yuxin Wang, Botian Shi (Shanghai AI Lab and collaborators).
- **year**: 2025 (published 2025-10-17).
- **venue/source**: arXiv preprint.
- **arxiv_id**: 2510.16079
- **urls**: paper https://arxiv.org/abs/2510.16079 ; code https://github.com/Edaizi/EvolveR ; model `Edaizi/EvolveR`; data `Edaizi/EvolveR-NQ-HotpotQA`
- **source availability**: latex (tarball, 10 .tex files). Deep-read.

## Primary-source facts (what the paper states/shows)

- **Research question**: how can an LLM agent learn systematically from the consequences of its *own* interactions across tasks (not just close external-knowledge gaps), without operational amnesia?
- **Model architecture / loop**: a closed-loop experience lifecycle with three components, alternating an Online phase and an Offline phase.
    - **Offline Self-Distillation**: with policy frozen, the agent reviews its own past trajectories and distills each into a concise natural-language *principle* (a "guiding principle" from success, a "cautionary principle" from failure), each paired with structured knowledge triples (Mem0/G-Memory inspired).
    - **Experience base curation**: principles are deduplicated (pairwise semantic-equivalence check by the agent model), integrated (embedding retrieve + binary equivalence judgment; new -> add, else merge trajectory under existing principle), and scored. The score is Laplace-smoothed empirical utility `s(p) = (c_succ + 1)/(c_use + 2)`; principles below a prune threshold are removed.
    - **Online Interaction**: the agent acts in a think-act-observe loop; its distinctive actions are `search_experience` (retrieve own principles) and `search_knowledge` (external search engine), then `answer`.
    - **Policy Evolution**: trajectories collected online update the policy via GRPO. Reward `R = w_o R_outcome + w_f R_format`. `R_outcome` is **exact-match against ground-truth answer** (sparse binary). `R_format` is a dense shaping term that requires structurally complete trajectories (at least one think, one search, one answer).
- **Action space**: `search_experience`, `search_knowledge`, `answer` (knowledge-intensive QA agent).
- **Observation / state representation**: text - retrieved principles and retrieved knowledge are returned as observations.
- **Dataset shape**: trained/evaluated on multi-hop QA (NQ + HotpotQA derived; the released dataset is `Edaizi/EvolveR-NQ-HotpotQA`).
- **Training objective**: GRPO on the composite reward, with the experience base making the trajectories "experience-guided" rather than random; the update "learns a policy of how to effectively utilize its own distilled wisdom."
- **Evaluation protocol + metrics**: complex multi-hop QA benchmarks; reports superior performance over strong agentic baselines. Key ablation: self-distillation is *less* effective on small models but at 3B scale "surpasses distillation by a stronger, external teacher model" (they call this cognitive alignment).
- **Code/data/weights**: all available (GitHub, HF model, HF dataset).
- **Reproducibility status**: partial-to-reproducible - code + data + a released model exist; full RL reproduction needs compute, but the artifacts are public.

## Interpretation (flagged as inference)

- EvolveR is the LLM-agent analog of the ENPIRE loop where "Evolution" (read your own logs, distill lessons) is made concrete as a *curated principle base* rather than code rewriting. It is the cleanest template for the repo's "the agent reads its own transcripts" idea.
- Crucially, the success signal is **exact-match against ground truth**, not the agent's own judgment. The agent proposes principles; an external label decides which trajectories were good. This preserves the proposer-vs-scorer separation even though the agent does the distillation.
- The curation machinery (dedup, merge, utility score, prune) is a direct answer to "how do you keep a self-grown memory from bloating or drifting." The Laplace-smoothed score is a usable formula.

## Mechanically useful vs research contribution

- **Mechanically useful for this repo**: (1) the offline self-distillation step is a near-drop-in for "the agent summarizes its own transcripts + CycleJudgments into reusable cautionary/guiding principles" - and the repo already stores transcripts and verifier evidence. (2) The principle-base curation (dedup/merge/score/prune) is a concrete design for actor-owned memory that does not bloat; it complements the repo's PlanBeads and the memory thread in `research-area-memory-and-verifiers.md` (cited, not re-surveyed). (3) The reward split (sparse outcome from ground truth + dense format from structural completeness) maps onto "verifier evidence for truth, schema/gate compliance for format."
- **Research contribution NOT to copy**: EvolveR's claim is closed-loop self-evolution on multi-hop QA via RL policy updates. The repo does not train model weights (no paid provider RL). Borrow the *lifecycle structure and curation*, not the GRPO weight update.
- **Where it supports the thesis**: it shows a no-human-label loop *can* improve an LLM agent when the outcome reward is a real external check. **Where it bounds it**: the outcome reward here is a crisp exact-match oracle; multi-hop QA has ground truth. The repo's upper (Social/Institutional) layers do not.
