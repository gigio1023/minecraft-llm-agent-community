# ENAS: Efficient Neural Architecture Search via Parameter Sharing

- **title**: Efficient Neural Architecture Search via Parameter Sharing
- **authors**: Hieu Pham, Melody Y. Guan, Barret Zoph, Quoc V. Le, Jeff Dean (Google Brain; Stanford)
- **year**: 2018 (arXiv 2018-02; ICML 2018)
- **venue/source**: ICML 2018
- **arxiv_id**: 1802.03268
- **urls**: paper https://arxiv.org/abs/1802.03268
- **source availability**: latex (tarball extracted, papers/latex/1802.03268/; abstract + intro + experiments + conclusion read directly)

## Primary-source facts (LaTeX-verified)

- **Research question**: NAS's computational bottleneck is training each sampled child model to convergence just to measure accuracy, then throwing the weights away. Can child models SHARE parameters so the search is cheap? (intro)
- **Method**: an RNN controller (policy gradient) selects a subgraph within one large computational graph; the selected subgraph's model is trained on cross-entropy while child models share the graph's parameters. Reward = validation performance (abstract, intro).
- **Cost fact (the 1000x claim)**: "1000x less expensive than standard Neural Architecture Search" in GPU-hours. On a single Nvidia GTX 1080Ti, ENAS finds a recurrent cell in ~10 hours and a conv architecture in <16 hours total. Baseline NAS cited at "450 GPUs for 3-4 days (32,400-43,200 GPU hours)"; for the conv design, ENAS reports "more than 50,000x" GPU-hour reduction vs that NAS run (abstract, intro, experiments, conclusion).
- **Results**: PTB test perplexity 55.8 (SOTA among no-post-processing methods, beats NAS 62.4); CIFAR-10 2.89% error, on par with NASNet's 2.65% (abstract).

## Interpretation (flagged as inference)

- ENAS is the single biggest cost drop in the NAS lineage: parameter sharing turns a 2,000-50,000 GPU-hour search into a <16-hour single-GPU search. It is the proof that the dominant cost in self-improving-a-learning-system was redundant re-training, and that a sharing trick removes most of it.
- The reusable insight for an autoresearch loop: avoid re-paying the expensive evaluation when candidates overlap. ENAS shares weights; AutoML-Zero's FEC dedups equivalent programs; both are "do not re-spend the costly signal" moves. The repo analog: reuse verifier evidence / partial rollouts across overlapping recipe candidates instead of re-running every cycle.

## Mechanically useful vs research contribution

- **Mechanically useful**: the "share state across candidates to avoid re-paying evaluation" principle (reference); a near-$0-verifier loop should cache/reuse rollout evidence across similar candidates.
- **Research contribution (not ours)**: parameter-sharing NAS is a research contribution.
- **What to avoid / overclaim**: ENAS later drew critique for search instability / weak correlation between shared-weight proxy accuracy and true accuracy (not in this paper; flagged as known follow-on). Do not assume a cheap shared-evaluation proxy faithfully ranks candidates.
