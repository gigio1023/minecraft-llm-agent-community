## Primary-source facts

- **Central taxonomy (intro)**: LLM reward-design methods split into two families.
    1. **Generate the reward CODE** (Eureka 2310.12931, Text2Reward 2309.11489, L2R 2306.08647, Auto MC-Reward 2312.09238): produce executable reward code. Limitations the paper names: requires env source code or input-parameter descriptions / templates; limited to logic compactly expressible as code; unclear how to handle high-dimensional state (images) or semantic features.
    2. **Generate reward VALUES** (Motif 2310.00166, and ONI): LLM ranks/labels observation captions; distill into a parametric reward model. Handles semantic input, needs no source code, scales to billions of env steps. Motif's limitation: needs a pre-existing diverse captioned dataset (which itself often needs a non-trivial reward to collect) and is a 3-stage OFFLINE pipeline (annotate -> train reward model -> train RL), costing days of GPU hours.
- **ONI's contribution**: a single, unified ONLINE system that learns the intrinsic reward model AND the policy together, from the agent's own experience, no external dataset. Built on Sample Factory APPO (async PPO); adds an async LLM server (vLLM, LLaMA-3.1-8B), a hash table of caption->annotation, and a reward-model learner thread. Retains ~80-95% of the original throughput (30k env steps/s without reward-model training, 26k with). Over a full 2B-step run (~19-20h) it annotates ~810k NetHack captions; the LLM only ever labels a tiny fraction of observations (often <=0.04%).
- **Three reward-distillation methods**: (1) Retrieval (binary helpful/unhelpful label, hash-table lookup, hyperparameter-free, no generalization); (2) Classification (train a binary classifier to generalize to similar captions); (3) Ranking (pairwise preference, Bradley-Terry reward model, like Motif, with running mean/std normalization for the online setting).
- **Result**: ONI matches Motif's performance on a range of sparse-reward NetHack (NLE) tasks using ONLY the agent's gathered experience in one unified process (no offline dataset).

## Interpretation (inference, labeled)

- ONI gives the lane its cleanest architectural axis: code-generation vs value-generation reward design. Code-gen (Eureka family) is interpretable and exact but brittle on semantic/high-dim state; value-gen (Motif/ONI) is semantic and scalable but produces an opaque scalar reward model and needs captions.
- For an embodied OPEN-WORLD social setting, the ONI taxonomy is directly diagnostic: physical/material consequences are well-suited to code (typed deltas, compact logic, the repo's existing path); social consequences ("is this caption a desirable social state?") resemble the semantic-feature regime ONI/Motif target with VALUE-based preference reward, not code.
- ONI demonstrates the loop can run online at scale without blocking on the LLM (async, label a tiny fraction). That is the engineering reassurance that an LLM-in-the-loop improvement signal need not dominate runtime cost.

## Mechanically useful vs research contribution

- **Mechanically useful**:
    - The async, fraction-labeling design (LLM labels <=0.04% of observations, never blocks rollout) is the pattern for putting any LLM judge or advisory predictor in a high-throughput Minecraft loop without it becoming the bottleneck.
    - The code-vs-value distinction is a design rule for the repo: keep physical/material consequence prediction as CODE (typed, inspectable, verifier-checkable); if a social-desirability signal is ever learned, expect it to be VALUE/preference-based and opaque, and treat it accordingly (advisory, not truth).
    - The three distillation methods (retrieval/classification/ranking) are a menu for turning sparse LLM judgments into a dense usable signal.
- **What to avoid / overclaim**:
    - ONI/Motif rewards are LEARNED, opaque scalar models distilled from LLM PREFERENCES, not from environment ground truth. They are intrinsic SHAPING rewards on top of a real task signal, not a substitute for it. The repo must keep the deterministic verifier as primary truth; an ONI-style learned social signal would be advisory only.
    - Motif's preference labels come from an LLM, so they inherit LLM-judge bias (cross-link [[research-area-memory-and-verifiers]] Part B, [[benchmark-validity-and-evaluation]]). Do not treat a distilled preference reward as a verifier.
    - The contribution is reward-learning infrastructure (support), not a research claim for this repo.
