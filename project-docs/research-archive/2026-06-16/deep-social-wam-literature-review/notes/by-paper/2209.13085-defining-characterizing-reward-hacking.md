## Primary-source facts

- **The first formal definition of reward hacking**. Given a true reward `R` and a proxy
  reward `R~`, the pair is **hackable** (relative to a policy set `Pi` and an environment)
  if there exist two policies pi, pi' in Pi such that proxy return goes UP while true
  return goes DOWN: `J~(pi) < J~(pi')` and `J(pi) > J(pi')`. A pair is **unhackable** if
  increasing expected proxy return can NEVER decrease expected true return. (Quoting:
  "increasing the expected proxy return can never decrease the expected true return.")
- **The central impossibility result (the strongest single fact)**: over the set of ALL
  stochastic policies, two reward functions can be unhackable only if one of them is
  CONSTANT. The reason given: reward is linear in state-action visit counts, which makes
  unhackability "a very strong condition." So you essentially cannot build a non-trivial
  unhackable proxy if the agent may choose any stochastic policy.
- **The escape hatch (where unhackability is achievable)**: restricting to DETERMINISTIC
  policies, or to any FINITE set of stochastic policies, guarantees that non-trivial
  unhackable pairs exist. So unhackability is recoverable precisely when the space of
  behaviors the agent can express is restricted.
- **Simplification (the asymmetric special case)**: a proxy `R~` is a "simplification" of
  `R` if it collapses some distinctions `R` makes while never reversing a strict preference
  (a one-directional unhackability). They establish necessary and sufficient conditions for
  simplifications to exist, and show by example that "seemingly natural ways of simplifying
  reward functions often fail to produce simplifications in our formal sense," i.e. leaving
  terms out of a reward to make it "narrower" usually makes it MORE hackable, not less.
- **Connection to optimization unpredictability**: the definition is motivated by the fact
  that RL optimization is unpredictable, so we cannot assume the agent reaches the optimal
  policy; hackability must hold over the policies the agent could actually visit, not just
  the optimum. They cite prior results that optimizing the proxy can drive true reward
  arbitrarily low.
- **Honest scope caveat (theirs)**: their definitions are conservative/worst-case ("can
  never decrease"). They explicitly call for future work on when hackable proxies can be
  shown safe in a probabilistic or approximate sense, or under LIMITED optimization (which
  is the regime Gao 2210.10760 measures empirically).

## Interpretation (labeled inference)

- The result formalizes a deep intuition that is directly load-bearing for this repo: you
  generally CANNOT make an imperfect proxy unhackable over all behaviors, but you CAN over
  a restricted behavior set. The repo's runtime gates (typed action surface, schema/enum
  enforcement, permission/source guards) are exactly a restriction of the policy set, which
  is the condition under which unhackability becomes possible. Hardening the verifier is, in
  this paper's terms, shrinking the set of behaviors that can satisfy the check without
  achieving the goal.
- "Narrowing a reward usually does not make it unhackable" is the formal warning against
  the tempting move of simplifying a social score by dropping terms; the dropped distinction
  is precisely where a gaming policy lives.

## Relevance to this repo (mechanical import vs research contribution)

- **Mechanical import**: this is the theory that justifies hardening by RESTRICTION rather
  than by trying to write a perfect reward. The repo should (1) keep the success signal a
  deterministic per-transition check and (2) keep the actor's expressible behaviors bounded
  by typed gates, because unhackability is only attainable on a restricted policy set. It
  also warns against "narrowing" a social score by omitting terms as a hardening tactic.
- **Why a learned reward model is the more hackable choice (tie)**: a learned reward over
  open-ended text/behavior is effectively a proxy evaluated over (near) all policies, the
  exact regime where the impossibility result says non-trivial unhackability does not
  exist. A deterministic verifier scoped to typed transitions on a restricted action
  surface is in the regime where unhackable pairs CAN exist.
- **Research contribution it is NOT**: the theorem is not something the repo proves or
  needs to extend; it is the formal grounding for the architectural rule "the runtime owns
  truth and the action surface is typed and gated." Support, not deliverable.
- **What it does not cover**: it is worst-case and exact (no probabilistic/approximate
  unhackability, no embodied or social instantiation, no empirical magnitudes). The
  empirical "how bad and how it scales" is Gao 2210.10760; the social/no-ground-truth
  regime is out of scope entirely.
