## Primary-source facts (LaTeX-verified)

- **Scope**: a position/survey on the bidirectional relationship between AutoML and LLMs, organized as "AutoML for LLMs" (Challenges I-V) and "LLMs for AutoML" (Opportunities I-III), plus five Risks (Section structure).
- **AutoML for LLMs challenges**: (I) cost of pre-training base models, (II) a multitude of different stages (data, pretrain, finetune, align), (III) multitude of performance indicators, (IV) combining learning paradigms and modalities, (V) determining neural architectures for LLMs (Section 2).
- **LLMs for AutoML opportunities**:
  - Opportunity I: improve human-machine interaction (natural-language config instead of code).
  - Opportunity II: LLMs CONFIGURE AutoML, suggesting search spaces, runtimes, termination, multi-fidelity (Hyperband) settings. Notes search-space design is "crucial ... but also hard to set up even for an expert" and that LLMs could even judge whether an AutoML run has converged.
  - Opportunity III: LLMs as COMPONENTS of AutoML systems, replacing hand-designed sub-systems with meta-learned versions (Section 3).
- **The named bridge to autoresearch / coding-agent loops (Opportunity III, the lane-relevant lineage)**:
  - GPT-NAS (Yu 2023): a GPT predicts parts of an architecture used to evolve architectures; pre-trained on NAS-Bench-101.
  - GENIUS (Zheng 2023): replaces the architecture-suggestion step with GPT-4; "iteratively prompts GPT-4 for an architecture, which it evaluates, and re-prompts GPT-4 with the performance asking for a better architecture" until a stopping criterion. This is an LLM hill-climbing a metric from feedback.
  - EvoPrompting (Chen 2023): LLM implements the crossover and mutation operators in an evolutionary NAS; adds all prior architectures + results as prompt context.
  - CAAFE (Hollmann 2023): LLM iteratively proposes feature-engineering CODE for tabular data and feeds back the evaluation.
  - OptFormer (Chen 2022): learns a surrogate model + acquisition function from Google Vizier's optimization history. LLMABO (Liu 2024): LLM improves BO warmstarting/surrogate/sampling.
  - AutoML-GPT (Zhang 2023) and MLCopilot (Zhang 2023): zero-shot AutoML from a textual problem description + knowledge base; AutoML-GPT "never evaluate a single ML pipeline ... only use LLMs to simulate the entire AutoML process." Tsai (2023): one LLM generates AutoML code, another LLM controls it from execution results.
- **Risks stated**: (I) shifts the entry barrier from coding/ML to prompt engineering; (II) evaluation; (III) false facts and misuse; (IV) trust and explanations; (V) resource consumption (Section 4).

## Interpretation (flagged as inference)

- This survey is the explicit hinge between my lane (classical AutoML) and the wave-4/wave-5 autoresearch concept. GENIUS (GPT-4 iteratively proposing architectures from performance feedback) and the AutoML-GPT/MLCopilot/Tsai "one LLM generates code, another controls it" pattern ARE the coding-agent autoresearch loop, applied to AutoML, two-plus years before ENPIRE applied the same loop to robots. ENPIRE's "coding agent edits training code and picks the method against a verified signal" is a descendant of Opportunity III, not a new idea structurally.
- It also surfaces the survey's own warning that maps to the repo's anti-progress-laundering rule: AutoML-GPT "never evaluate a single ML pipeline" and only SIMULATES the process. That is exactly the failure mode the repo guards against, an LLM scoring/imagining success without a real verified evaluation. The lesson: an LLM-driven improvement loop is only trustworthy when a real, held-out evaluation (not the LLM's own simulation) provides the signal.

## Mechanically useful vs research contribution

- **Mechanically useful**: this is the reading map for "what an LLM-driven autoresearch loop tuning the repo's WAM recipe would look like" and which patterns are evaluation-grounded (GENIUS, CAAFE, EvoPrompting all evaluate candidates) versus simulation-only (AutoML-GPT). The repo should adopt the evaluation-grounded shape (every proposed recipe/feature change is actually run against the verifier) and explicitly reject the simulation-only shape.
- **Research contribution (not ours)**: the survey is a field map; the repo does not contribute to AutoML-LLM research.
- **What to avoid / overclaim**: do not adopt simulation-only LLM "AutoML" (LLM predicts which recipe is best without running it). The repo's whole premise is that the runtime, not the LLM, owns truth; a simulation-only loop violates that.
