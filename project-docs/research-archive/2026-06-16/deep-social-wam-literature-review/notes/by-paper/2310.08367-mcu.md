# MCU - A Task-centric Framework for Open-ended Agent Evaluation in Minecraft

- **authors**: Lin, Cai, Wang et al. (Team CraftJarvis)
- **year**: 2023 (rev 2025) - **venue**: arXiv; ICLR 2025
- **arxiv_id**: 2310.08367
- **urls**: https://arxiv.org/abs/2310.08367 ; code https://github.com/CraftJarvis/MCU

## Primary-source facts

- **Research question**: how to evaluate *open-ended* Minecraft agents across
  arbitrary tasks rather than a single objective (ObtainDiamond)?
- **Atom tasks** are the building blocks; arbitrary tasks are generated by
  composition. Each task carries **six difficulty scores**: time consumption,
  operational effort, planning complexity, intricacy, creativity, novelty. The
  scores form a feature space that situates tasks relative to each other.
- **SkillForge**: a curated unified benchmark of representative tasks with
  diverse categories and difficulty distribution; filters let users select tasks
  that probe a specific capability.
- **Claim**: MCU has high expressivity - it can cover all tasks used in recent
  Minecraft-agent literature - and exposes weak spots in creativity, precise
  control, and out-of-distribution generalization.
- **Scoring**: the framework leans on VLM/criteria-based grading of the
  six-dimension rubric (consistent with MineStudio's auto-eval criteria files).

## Interpretation (inference)

- MCU is an **evaluation framework**, not an agent. Its useful contribution is the
  **task-difficulty feature space** - a vocabulary for *why* a task is hard
  (planning vs control vs novelty), which lets you compare agents on matched
  difficulty rather than cherry-picked tasks.
- Its scoring relies substantially on **VLM/rubric grading**, which this repo
  treats as a secondary audit layer, not runtime truth. The MineStudio analysis
  already captured the criteria families (task progress, action control, error
  recognition, creative attempts, efficiency, material usage).

## Mechanically useful vs research contribution

- **Mechanically useful**: the **six difficulty dimensions** are a clean way to
  tag this repo's competence gates and to ensure social scenarios are not
  confounded by raw task difficulty (e.g. hold planning-complexity fixed while
  varying social pressure).
- **What would be overclaim**: adopting MCU's rubric scores as truth, or treating
  "creativity/novelty" scores as social value. Spectacle is not social
  consequence (the repo rule: only count creative acts that change verified
  social/material state).

## WAM layer(s) informed

Physical-WAM (difficulty taxonomy + competence evaluation). Reproducible: code
public; scoring partially VLM-dependent ⇒ partial for truth purposes.
