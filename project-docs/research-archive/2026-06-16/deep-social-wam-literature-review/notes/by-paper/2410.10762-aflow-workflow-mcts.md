## Primary-source facts (LaTeX-verified)

- **Problem formulation**: an agentic workflow W is a series of LLM-invoking nodes N connected by edges E. A node has parameters (model M, prompt P, temperature tau, output format F). Edges can be a graph, a neural network, or CODE; AFlow argues code is the most expressive (code is a superset of graphs; expresses sequences, conditionals, loops). Automated workflow optimization = given task T and evaluation function G, find W maximizing G(W,T) by searching the space S of node-parameter and edge configurations.
- **The method = MCTS over code-represented workflows**: AFlow uses Large Language Models as optimizers inside a Monte Carlo Tree Search variant. Each MCTS tree node is a COMPLETE workflow (not an individual LLM call). To make the vast code space tractable, AFlow FIXES model/temperature/format and searches only over prompts + code edges + a set of Operators (reusable node combinations: Generate, Format, Review&Revise, Ensemble, Test, Programmer, Custom). The loop: Soft Mixed Probability Selection, LLM-Based Expansion, Execution Evaluation, Experience Backpropagation, until N_max iterations or convergence.
- **Selection**: a soft mixed-probability strategy combining uniform and score-weighted distributions over top-k workflows plus the initial blank workflow (so exploration never dies). Constants alpha=0.4 (score influence), lambda=0.2 (exploration/exploitation balance).
- **Expansion**: an LLM optimizer mutates the selected workflow's prompts and/or code edges, conditioned on that workflow's "experience" (all prior modifications and whether they improved or failed, plus precise prediction/expected-output logs).
- **Evaluation (the success signal)**: for reasoning tasks with explicit evaluation functions, AFlow directly EXECUTES each candidate workflow. "We test each generated workflow 5 times on the validation set, computing mean and standard deviation ... it provides more accurate feedback for the optimizer" (algorithm comment: "Multiple runs for robustness"). Data is split 20% validation / 80% test, seed fixed at 42; the blank template is run 5x and high-variance problems are selected to form the final validation set.
- **Backpropagation**: records workflow performance, the modification vs parent, and success-relative-to-parent into experience, propagated to the parent; the score goes to a global record used for selection. Early stopping if top-k average does not improve for n rounds.
- **Results (as stated)**: across six benchmarks (HumanEval, MBPP, MATH, GSM8K, HotpotQA, DROP) AFlow outperforms manually designed methods by 5.7% on average and existing automated approaches by 19.5%. Workflows it generates let smaller/weaker LLMs outperform GPT-4o on specific tasks at 4.55% of the dollar inference cost. On GSM8K it reaches 95.6% (2.5% behind a SOTA that required substantial human effort).
- **Stated positioning vs ADAS**: ADAS also represents workflows in code (a relatively complete representation) "but fails to discover effective workflows due to the efficiency limitations of its linear heuristic search algorithm." AFlow's contribution is the MCTS search + tree-structured experience reuse that makes code-space search efficient.

## Interpretation (flagged as inference)

- AFlow is the strongest demonstration that VERIFIER-SCORED search works: its evaluation function G directly executes the workflow and checks the answer (numerical reasoning tasks), and it runs each candidate 5x for robustness on a held-out validation set. This is the closest analogue to a runtime-verifier-driven search, and it is precisely the part this repo can reuse.
- The deliberate search-space restriction (fix model/temp/format; search only prompts + code edges + operators) is the engineering lesson: an unbounded code space is too big to search honestly, so AFlow narrows to the high-leverage axes. For this repo, the analogue is searching Actor Turn / advisory-WAM PROMPT text and small workflow choices, not the runtime gates.
- The "5 runs on a held-out validation set, pick high-variance problems" detail is a concrete anti-overfitting / anti-noise practice worth copying: a single run can reward a lucky workflow.

## Mechanically useful vs research contribution

- **Mechanically useful**: (1) MCTS-with-experience over a RESTRICTED search space (fix the runtime-owned axes, search only prompt/workflow); (2) execution-based evaluation with multiple runs for robustness on a held-out validation set; (3) the soft-mixed-probability selection that always keeps the blank/initial workflow reachable to escape local optima; (4) operators as reusable building blocks (analogous to the repo's Action Cards / verifier-gated primitives).
- **What would be overclaim**: AFlow's results are on reasoning benchmarks with crisp numerical answer-checkers. A social Minecraft world has no single numerical G for upper layers. Do not claim AFlow-style gains transfer to Social/Institutional layers; the honest borrow is the loop at the Physical/Material layers where the verifier returns a clean score.
- **Honest bound for thesis**: AFlow's whole result depends on G being an execution-based answer-checker. If G were an LLM judge, the same search would optimize the judge's blind spots. This is the lane's central warning instantiated: prompt/workflow search MUST be verifier-scored on held-out scenarios.
