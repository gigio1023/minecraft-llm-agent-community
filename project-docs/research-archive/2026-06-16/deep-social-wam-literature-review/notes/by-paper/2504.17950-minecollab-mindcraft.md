# MINDcraft + MineCollab — Collaborating Action by Action (Multi-agent LLM Embodied Reasoning)

- **authors**: White, Stengel-Eskin et al.
- **year**: 2025 — **venue**: arXiv (2504.17950)
- **urls**: https://arxiv.org/abs/2504.17950 ; code https://github.com/mindcraft-bots/mindcraft ; site https://mindcraft-minecollab.github.io/
- **NOTE**: this is the most directly relevant *multi-agent* source for the
  material/social layers — it uses the same Mineflayer high-level tool family as
  this repo and isolates communication as the bottleneck.

## Primary-source facts

- **MINDcraft platform**: a plug-and-play framework for LLM agents to control
  Minecraft characters. Four components: (1) a server managing agents, (2) a main
  agent loop handling messages from players/agents, (3) a library of **47
  parameterized high-level tools** + observation queries, (4) a prompting/LLM
  layer. Agents may also emit **custom Mineflayer JavaScript** when a tool does
  not exist.
  - Example tool call: `!givePlayer("randy", "oak log", 4)` replaces hand-written
    Mineflayer code to find/travel/identify/drop — i.e. **item transfer is a
    first-class typed action**.
  - Communication: pairwise (only two agents converse at once), scales by
    switching active conversations; agents otherwise unrestricted.
- **MineCollab benchmark**: three collaboration domains —
  - **Cooking**: coordinate ingredient collection + multi-step combine; a
    "**Hell's Kitchen**" variant gives each agent recipes for only a subset of
    needed items, **forcing them to ask teammates** for instructions.
  - **Crafting**: agents hold different resources + a (sometimes blocked) recipe;
    must communicate inventories, share the recipe, and **give each other
    resources** to craft.
  - **Construction**: build from procedurally generated blueprints; scored by an
    **edit-distance metric** vs the blueprint (structured, not VLM).
  - Tasks average **>20 steps**; train/test use disjoint recipes/blueprints.
- **Headline finding**: the primary bottleneck for current SOTA agents is
  **efficient natural-language communication** — performance drops **up to 15%**
  when agents must communicate detailed task-completion plans. The authors
  conclude current LLM agents are ill-optimized for embodied multi-agent
  collaboration and that in-context/imitation learning is insufficient.

## Interpretation (inference)

- MineCollab is the closest existing benchmark to this repo's **material layer**:
  it has explicit **resource sharing, item handoff (`givePlayer`), recipe
  dependence, and forced requests** (Hell's Kitchen). Its construction scoring is
  a clean **structured verifier** (edit distance), not VLM grading.
- But the social model stays at **task-completion**: it measures whether the meal
  was cooked / structure matched, with communication as an *instrumental cost*,
  not whether a **promise was kept, a debt repaid, a claim respected, or a
  relationship/obligation persisted after the task**. The "15% drop when
  communicating plans" is a competence cost of talking, not a social consequence.
- This is the cleanest demonstration of the **social-material transition gap**:
  the field can already make agents *share items to finish a task*; it does not
  yet score the *durable social-material consequence* of that sharing.

## Mechanically useful vs research contribution

- **Mechanically useful (high)**: (a) the **47-tool high-level action library** is
  a direct comparison point / parts list for this repo's Action Cards (esp.
  `givePlayer`, inventory queries); (b) "custom Mineflayer code when no tool
  exists" mirrors `author_mineflayer_action`; (c) **edit-distance blueprint
  scoring** is a reusable structured verifier; (d) **Hell's Kitchen recipe
  asymmetry** is a ready-made way to *force* request/handoff (maps onto
  obligation/credit).
- **What would be overclaim**: presenting MineCollab task success as social
  contribution. It is collaboration-for-task-completion; possession/claim/
  obligation/weak-commons/post-goal continuation are not its measured target.

## WAM layer(s) informed

Material-WAM (resource sharing, handoff, recipe dependence — strong reference) and
Social-WAM (request/communication, but only as instrumental cost). Reproducible:
platform + benchmark public.
