# CARD: LLM-Driven Reward Design via Dynamic Feedback

- **title**: A Large Language Model-Driven Reward Design Framework via Dynamic Feedback for Reinforcement Learning (CARD = Coder-evAluator Reward Design)
- **authors**: Shengjie Sun, Runze Liu, Jiafei Lyu, Jing-Wen Yang, Liangpeng Zhang, Xiu Li (Tsinghua Shenzhen; Tencent IEG)
- **year**: 2024
- **venue/source**: arXiv (NeurIPS 2024 preprint style)
- **arxiv_id**: 2410.14660
- **urls**: https://arxiv.org/abs/2410.14660
- **source availability**: latex (papers/latex/2410.14660/main.tex, deep-read)

## Primary-source facts

- **Goal**: generate and iteratively improve reward function CODE with NO human feedback, NO parallel LLM queries, and crucially NO per-iteration RL training (the dominant cost in Eureka).
- **Architecture**: a Coder (LLM that writes + self-verifies reward code) and an Evaluator (provides dynamic feedback). Three feedback types:
    - **Process feedback**: changes in trajectory return and sub-rewards during RL training.
    - **Trajectory feedback**: per-step parameter detail for successful vs failed trajectories.
    - **Preference feedback** via **Trajectory Preference Evaluation (TPE)**: evaluates a reward function using trajectory PAIRS and preferences WITHOUT running RL. If the reward fails TPE, preference feedback is returned; this is what eliminates per-iteration RL training.
- **Explicit method comparison (Table 1, main.tex lines 121-137)**:
    - L2R (Language to Rewards): 1 LLM query, 0 iterations, no iterative improvement, no evaluator feedback.
    - Text2Reward: iterative, but needs HUMAN feedback, per-iteration RL training.
    - Eureka: iterative, no human, but **16 queries x 5 iterations** with per-iteration RL training.
    - CARD: iterative, no human, **1 query x 2 iterations**, evaluator feedback, NO per-iteration RL training.
- **Results**: on Meta-World and ManiSkill2, CARD balances task performance and token efficiency, outperforming or matching baselines on all tasks; on 10/12 tasks it is >= expert-designed-reward policies, and surpasses the oracle on 3 tasks.

## Interpretation (inference, labeled)

- CARD is the cost-efficiency frontier of reward-code search: it attacks the most expensive part of the Eureka loop (running RL per candidate) by replacing it with a cheaper, learned preference proxy (TPE). The lesson for an autoresearch loop is that the "verify outcome" step does not have to be a full rollout if a cheaper surrogate predicts the rollout's verdict.
- For this repo the parallel is exact and favorable: the repo's runtime verifier already produces the rollout verdict at near-$0 (no GPU RL training needed), so the repo does not even need TPE; it has a real verifier where CARD has a learned preference surrogate. CARD is evidence that the loop is affordable; the repo is in an even cheaper regime for the Physical/Material layers.

## Mechanically useful vs research contribution

- **Mechanically useful**:
    - The Coder/Evaluator split (one LLM proposes code, a separate component scores and feeds back) is a clean two-role pattern for a code-proposing loop, and matches the repo's separation of the proposing LLM from the runtime scorer.
    - TPE (score a candidate by pairwise trajectory preference, no retraining) is a template for cheaply ranking candidate WAM predictors when a full verifier pass is somehow expensive: rank predictors by which better matches observed transition pairs.
    - The dual feedback channels (process = aggregate training trend; trajectory = per-step detail) map onto two granularities of verifier feedback: aggregate cycle outcome vs per-typed-delta agreement.
- **What to avoid / overclaim**:
    - TPE is a LEARNED preference proxy and can be wrong; CARD trusts it to skip RL. The repo must not let a learned proxy stand in for the deterministic verifier on physical truth (the verifier is cheap here, so there is no reason to). Use a TPE-like surrogate only as a pre-filter, never as the final scorer.
    - Like all of this family, the contribution to copy is the loop engineering, not a research claim. CARD's own scope is robot manipulation reward code, not social prediction.
