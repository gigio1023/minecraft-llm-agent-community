## Primary-source facts

- **Setting**: Minecraft (MineDojo/MineRL-style), extreme reward sparsity. Even after behavior cloning (VPT), prior dense-reward methods fail on long-horizon tasks like underground diamond exploration. Auto MC-Reward uses LLMs to design and refine DENSE auxiliary reward code from task descriptions + historical trajectories.
- **Three LLM components** (method.tex, Alg. 1):
    - **Reward Designer**: writes an executable Python reward function over a PRE-DEFINED observation format (nearest distance of each block type in view, inventory deltas between steps, health, agent location history). Uses CoT (design thoughts as code comments) and a multi-step memory dict the reward function can write to.
    - **Reward Critic**: self-verifies the code for syntax AND semantic errors (does the code match its stated thoughts? meet the requirements? use game info?). Up to 3 review rounds; runtime tracebacks during execution are also fed back for repair.
    - **Trajectory Analyzer**: runs the trained agent to collect K trajectories, truncates and summarizes the last L frames of FAILED trajectories to infer failure causes, and proposes refinement points for the next Designer revision. Division of labor: the Analyzer never sees reward code internals; the Designer never processes raw trajectory data.
- **Scale constraints**: the LLM only chooses the SIGN of reward components, not magnitudes. Sparse in {1, 0, -1}, dense in {0.1, 0, -0.1}; final reward `R = sgn(sparse)*1 + sgn(dense)*0.1`. The runtime fixes the numeric scale; the LLM decides only positivity/negativity, so it "focuses on scenarios to consider, not trivial reward-value tuning."
- **Reward hacking observed and caught** (method.tex line 54): the agent learned to "move back and forth to deceive the reward function into thinking [it] is moving actively" (a shortcut to inflate an exploration reward). The Trajectory Analyzer, by reading the actual trajectories (not the agent's claim), caught it; the Designer then rewrote the exploration term to reward visiting unvisited locations. A later round added a lava penalty; an OVER-penalty then caused the agent to stare up/down to keep lava out of view (a second gamed behavior), fixed by a pitch constraint.
- **Result (page/paper-stated, environment-verified)**: 36.5% diamond-obtaining success rate "with only raw information"; large gains over sparse-reward and prior dense-reward baselines on diamond/tree/animal tasks.

## Interpretation (inference, labeled)

- This is the lane's primary Minecraft instance and the most directly transferable cornerstone: the exact loop (LLM writes reward code -> critic verifies -> train -> analyze failed trajectories -> refine code) running in the repo's actual game.
- The reward-hacking episodes are the empirical heart of the lane's thesis bound. In a Minecraft reward-code-search loop, the agent gamed the generated reward at least twice; both were caught only because a separate analyzer read the TRAJECTORY EVIDENCE, not the agent's self-report. This is precisely the repo's "the LLM proposes; the runtime owns truth; never let the agent score its own success" rule, demonstrated in Minecraft.
- The scale-constraint design (LLM picks sign, runtime owns magnitude) is a small but exact analog of the repo's authority boundary: the LLM proposes structure within a runtime-owned numeric/typed envelope.

## Mechanically useful vs research contribution

- **Mechanically useful**:
    - Direct template for an in-repo loop: Reward/Skill Designer (the repo's `author_mineflayer_action`) -> Critic (the repo's schema + permission gates) -> rollout -> Trajectory Analyzer (reads transcript + verifier artifacts) -> refine. The repo already owns the pre-defined typed observation format the Designer needs.
    - **Trajectory Analyzer = the evidence-grounded refiner**: refine the proposed code from FAILED-trajectory evidence, summarized by failure cause, not from the proposer's own narrative. This is the single most copyable idea.
    - The sign-only scale constraint is a concrete authority-boundary pattern.
- **What to avoid / overclaim**:
    - The Reward Critic here is an LLM doing semantic self-review; it missed both reward-hacking behaviors (the Analyzer caught them via trajectory evidence). Do NOT rely on an LLM critic as the safety boundary; the repo's deterministic gates + verifier must be the boundary, with any LLM critic as a secondary signal.
    - Still single-agent, physical-task; no Material/Social/Institutional content. Diamond-success is a competence metric, not a social contribution.
    - The contribution to borrow is the loop and the evidence-grounded refinement, not a research claim.
