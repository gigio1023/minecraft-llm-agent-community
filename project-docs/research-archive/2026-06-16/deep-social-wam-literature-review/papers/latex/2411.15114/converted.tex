\section{\texorpdfstring{{Summary}\textsuperscript{\protect\hypertarget{cmnt_ref3}{}\hyperref[cmnt3]{{[}c{]}}\protect\hypertarget{cmnt_ref4}{}\hyperref[cmnt4]{{[}d{]}}\protect\hypertarget{cmnt_ref5}{}\hyperref[cmnt5]{{[}e{]}}}}{Summary{[}c{]}{[}d{]}{[}e{]}}}\label{h.4dxrroeyhuma}

\begin{itemize}
\tightlist
\item
  {We present 7 evaluation environments for testing AI R\&D abilities. }
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Each evaluation presents a unique continuous ML optimization problem
  (e.g. reducing the loss of a model, or improving the runtime), and
  they are designed to allow for and tell apart a wide range of solution
  qualities.}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {The evaluation suite is trying to capture some of the hardest skills
  involved in AI R\&D, and achieving a strong score generally requires
  significant experimentation, iteration and managing of limited compute
  resources. }
\item
  {We provide results from }{44 human
  runs}\textsuperscript{\phantomsection\label{cmnt_ref6}\hyperref[cmnt6]{{[}f{]}}}{~and
  35 runs with a basic Claude-3.5-Sonnet agent in these 7 environments,
  and find that human researchers are slow to start, but make consistent
  progress over time, while agents outpace humans in the first hour but
  then plateau (see }{\hyperref[id.mf3qlnxl0yyh]{Figure 1}}{). This
  baselining data allows for fine-grained comparisons between human
  experts and future AI systems. ~}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Despite eventually plateauing, AI agents did discover surprisingly
  strong solutions in some cases, occasionally beating weaker solutions
  from human professionals (see }{\hyperref[id.lfw1nma1i54o]{Figure
  2}}{). We discuss these instances and general agent
  strengths/weaknesses in }{\hyperref[h.lhv16x3d5jf3]{the appendix}}{. }
\end{itemize}

\begin{itemize}
\tightlist
\item
  {We also identify and evaluate }{\hyperref[h.5v1v6g2ipr0w]{5 key
  desiderata}}{~}{(such as novelty/non-memorizability, and having a low
  rate of technical issues and ambiguities)}{~}{we have aimed for with
  our AI R\&D evaluations, and present a table of how well our
  environments satisfy them.}
\item
  {Advanced AI R\&D abilities have been identified as a key risk
  threshold requiring strong security and evaluations measures in RSPs,
  and we believe reaching parity with strong human researchers in these
  evaluation environments could be used as a concrete early warning
  threshold for these risks. }
\item
  {In the future we hope to address several key limitations of this
  evaluation suite, including providing a larger number of evaluation
  environments, evaluating performance over longer time horizons (e.g.
  24 hours instead of 8), and seeing how much agent behavior improves
  with better tools, more tokens, or other improved scaffolding.
  }\textsuperscript{\phantomsection\label{cmnt_ref7}\hyperref[cmnt7]{{[}g{]}}}
\end{itemize}

\section{\texorpdfstring{{Introduction}}{Introduction}}\label{h.zaaoehs85v0c}

{AI agents automating AI R\&D has been identified as a key risk in
frontier safety policies from several leading AI
developers}\textsuperscript{\phantomsection\label{ftnt_ref1}\hyperref[ftnt1]{{[}1{]}}\phantomsection\label{cmnt_ref14}\hyperref[cmnt14]{{[}n{]}}}{.
Such agents would massively increase the amount of skilled research
labor available to AI developers, and could potentially lead to a
runaway feedback loop of accelerating AI and R\&D
progress}\textsuperscript{\phantomsection\label{ftnt_ref2}\hyperref[ftnt2]{{[}2{]}}}{,
posing a number of different risks:}

\begin{itemize}
\tightlist
\item
  {Capabilities outpacing safety and oversight: }{AI capabilities might
  progress much faster than safety and oversight processes and
  institutions can keep up with.}
\item
  {Extreme concentration of power: }{A small lead in AI capabilities
  could get inflated, and lead to an actor getting access to vastly more
  powerful AI systems than anyone else. }
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Incentives to race: }{The potential for such concentrations of power
  could in turn result in extreme incentives to race and compromise on
  safety.}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Sabotage: }{Having AI systems involved in AI R\&D efforts within a
  lab gives significant opportunities for those agents to sabotage
  control and oversight methods, if misaligned.}
\item
  {Proliferation risks: }{Model theft could lead to rapid proliferation
  of highly capable AI systems, amplifying other risks.}
\end{itemize}

\textsuperscript{\phantomsection\label{cmnt_ref15}\hyperref[cmnt15]{{[}o{]}}}

{}

\textsuperscript{\phantomsection\label{cmnt_ref16}\hyperref[cmnt16]{{[}p{]}}}

{There are many ways AI developers could try to identify this risk,
including:}

\begin{itemize}
\tightlist
\item
  {Tracking trends in the pace of AI progress, and looking for signs of
  sharp discontinuities.}
\item
  {Doing human uplift studies, investigating how much AI systems are
  contributing to the productivity of human researchers over time}
\item
  {Looking at real-life demonstrations of AI systems doing autonomous AI
  research, e.g. seeing if AI systems can produce NeurIPS papers or
  develop effective novel architectures}
\item
  {Tracking progress on general coding and reasoning benchmarks}
\end{itemize}

{}

{In this work we have tried to design an evaluation that satisfies the
following properties:}

\begin{itemize}
\tightlist
\item
  {Practical early warning: }{The method can detect when AI systems are
  getting close to transformative AI capabilities at early stages of
  development. At the point when AI systems are already drastically
  accelerating progress it might be too late to apply many safety
  measures such as weight security. The evaluation should be possible to
  run in }{a few weeks of time at most }{on a new model or model
  checkpoint, without requiring broad deployments. }
\item
  {Direct comparison against humans: }{Humans generally take months or
  sometimes years to produce e.g. strong NeurIPS papers, and often do so
  at very significant compute and labor costs. If care is not taken to
  compare against realistic human performance, it is very easy to
  construct evaluations and thresholds that are beyond what strong human
  researchers are capable of, which could lead to risks not being
  noticed in time.
  }\textsuperscript{\phantomsection\label{cmnt_ref17}\hyperref[cmnt17]{{[}q{]}}}
\item
  {Test the full set of capabilities needed to advance AI R\&D: }{Many
  benchmarks test narrow skills like writing small code snippets to
  solve competition problems, or answering QA questions. These risk not
  transferring to transformative progress in the real world, since
  real-world problems require a much wider set of skills, like
  long-horizon planning and agency. }
\end{itemize}

{}

{The simplest way to accomplish this is to design evaluation
environments where human researchers and AI agents can be directly
compared in their ability to make end-to-end progress on open-ended
metrics with similar resources. We present 7 such environments, along
with detailed data on human performance over time. Ideally such tests
would be run over months with cutting-edge models and large compute
resources, but for practical reasons we have only evaluated them over 8
hours of work. }

{}

{Compared to existing evaluations of autonomous capabilities and R\&D
skills, these environments are:}

\begin{itemize}
\tightlist
\item
  {Tested very thoroughly with human participants, providing
  fine-grained data on what matching human abilities on these
  environments would look like.}
\item
  {Designed with a continuous success metric, with initial progress on
  each environment being easy but reaching top scores within the time
  limit being almost impossible for humans.}
\end{itemize}

{}

{We find that human researchers are slow to start, but make consistent
progress over time, while agents outpace humans in the first hour but
then plateau. If AI agents were to reach parity with strong human
researchers on these environments over 8 hours we think AI developers
should take strong steps to mitigate AI R\&D automation risks, unless
longer-horizon tests with}{~similar fidelity have been
developed}\textsuperscript{\phantomsection\label{ftnt_ref3}\hyperref[ftnt3]{{[}3{]}}}{~}\textsuperscript{\phantomsection\label{cmnt_ref18}\hyperref[cmnt18]{{[}r{]}}\phantomsection\label{cmnt_ref19}\hyperref[cmnt19]{{[}s{]}}}{showing
significant limitations compared to human researchers.}

Alongside instructions, we provided both human baseliners and
Claude-3.5-Sonnet~agents with a simple starting solution in each
environment. For example, for the ``optimize a kernel'' environment, we
provide a naive PyTorch implementation of the modified prefix sum; for
the ``restricted architecture MLM'' environment, we provide code that
implements a simple ReLU MLP network. The environments are all designed
to have some relatively simple optimization opportunities that can
improve the score a little bit in addition to more challenging ones, to
get a faster and more reliable signal on partial progress. For instance,
some initial score improvement can be achieved in many environments just
by tweaking hyperparameters.

By default, both human baseliners and agents were allowed to submit
solutions multiple times, and for all environments except ``Scaling law
experiment'' they could see their score each time they submitted. Except
for the scaling law environment their final score was the highest score
amongst each of their submissions. However, as we wanted to discourage
random guessing in the scaling law experiment, we used the highest score
between the starting solution and the final submitted score. 


\section{\texorpdfstring{{Desiderata for a good environment
suite}\textsuperscript{\protect\hypertarget{cmnt_ref20}{}\hyperref[cmnt20]{{[}t{]}}\protect\hypertarget{cmnt_ref21}{}\hyperref[cmnt21]{{[}u{]}}}}{Desiderata for a good environment suite{[}t{]}{[}u{]}}}\label{h.5v1v6g2ipr0w}

{We have identified 5 key properties we'd want our eventual environment
suite to satisfy. }

{}

\begin{itemize}
\tightlist
\item
  {\hyperref[h.htw1exqgzby3]{Low floor}}{:}{~½ of baseliners make
  progress in the first 4 hours, and ⅔ by hour 8. }
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Why? If not then it is easy to get false negatives, and we won't
  reliably see gradual progress over time}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {\hyperref[h.85nxwftddzri]{High ceiling:}}{~}{There is room to further
  improve the score even for the best experts by the end of 8 hours,
  based on our best judgment of what's achievable in each environment.
  Additionally, even the top 90th percentile scores do not seem to be
  clearly plateauing by 8 hours. ~}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Why? }{This makes sure the environments can measure surprisingly
  strong or superhuman agent performance in particular domains}{, and
  reduces (but of course doesn't eliminate) the risk of the evaluation
  getting saturated by AI agents before they can do autonomous AI R\&D
  work}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {\hyperref[h.n5lrkrvqt9e9]{Few ambiguities and issues:}}{~}{Less than
  ¼ of }{human and agent runs}{~have severe issues with the environment
  or infrastructure that require an intervention from us or would take
  \textgreater1hr for a skilled engineer to resolve or work around while
  doing the environment. }{Additionally, there are few cases where human
  experts misunderstand the objective or are surprised by their eventual
  score. }
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Why? Building and running these environments is highly complex, and
  it's very easy for ambiguities in instructions or technical issues
  (bugs in the scoring function, issues with package management etc.) to
  invalidate results.
  }\textsuperscript{\phantomsection\label{cmnt_ref22}\hyperref[cmnt22]{{[}v{]}}}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {\hyperref[h.q5m7qd9mjcp0]{Novelty:}}{~}{There are no examples of
  people solving this problem or posting solutions which would score
  well (\textgreater0.5) on the internet as far as we know, and the best
  current language models cannot generate good solutions without
  significant iteration.}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Why? AI agents might be able to outdo experts without having strong
  research skills if solutions can be memorized, or if problems can be
  solved with highly standard methods. We want the benchmark to capture
  the ability to tackle novel problems.}
\end{itemize}

\begin{itemize}
\tightlist
\item
  {\hyperref[h.qt7srnrx8j84]{Coverage:}}{~}{The environment suite covers
  a variety of key research activities and skills, as operationalized by
  looking at key teams and workstreams described in the
  }{\href{https://www.google.com/url?q=https://arxiv.org/pdf/2303.08774&sa=D&source=editors&ust=1727392285019629&usg=AOvVaw0Gwmy43ooZ_4YykKMAux5R}{GPT-4
  technical report}}{, and the recent
  }{\href{https://www.google.com/url?q=https://epochai.org/files/Interviewing_AI_researchers_on_automation_of_AI_R_D.pdf&sa=D&source=editors&ust=1727392285019994&usg=AOvVaw3s4UCmhGh6lLWj6g8fWipa}{AI
  R\&D survey}}{~by Epoch. }
\end{itemize}

\begin{itemize}
\tightlist
\item
  {Why? Good variety increases the chance of discovering narrow AI agent
  strengths, and having relatively complete coverage means strong
  general performance is more likely to transfer to full independent AI
  R\&D research ability. }
\item
  {This criteria cannot be assessed for individual environments, but is
  rather a property of the environment suite as a
  whole}\textsuperscript{\phantomsection\label{cmnt_ref23}\hyperref[cmnt23]{{[}w{]}}\phantomsection\label{cmnt_ref24}\hyperref[cmnt24]{{[}x{]}}}
\end{itemize}

\section{\texorpdfstring{{Evaluation on Claude
3.5}}{Evaluation on Claude 3.5}}\label{h.ehpuwb9fgkyi}

{We ran an exploratory evaluation on Claude 3.5 to demonstrate the
evaluation suite (we picked Claude 3.5 since it did the best on our
general autonomy

{}

{We found that }{Claude-3.5-Sonnet}{~agents were able to make meaningful
progress on 3 out of the 7 environments: ``Optimize a kernel'',
``Finetune GPT-2 for QA'', and ``Scaffolding for rust }{codecontests}{''
(see }{\hyperref[id.lfw1nma1i54o]{figure 2}}{). In the other 4
environments, none of the agents were able to find solutions that
improved upon the provided starting solution. While we expected agents
to find some of the easy improvements across the environment suite, we
were surprised by the score the agent achieved in some cases, with the
best AI solutions sometimes beating }{weaker solutions from human
professionals.}\textsuperscript{\phantomsection\label{cmnt_ref31}\hyperref[cmnt31]{{[}ae{]}}}{~These

\tightlist
\item
  {A basic ability to understand the problem given, iterate on its
  solution, run scoring, read the score log and debug issues that came
  up. }
\item
  {A persistence in debugging issues, sometimes over dozens of steps,
  that occasionally allowed it to fix complex issues.}
\item
  {Extremely fast coding and reading, allowing it to test dozens of

\end{itemize}

\begin{itemize}
\tightlist
\item

\item

  the time limit was reached if a decent solution was found. }
\end{itemize}

\section{\texorpdfstring{{Discussion and
Limitations}}{Discussion and Limitations}}\label{h.894ew8oy1x08}

{Overall our data indicates that these environments provide a meaningful
continuous metric of progress. }

{We think these are }{some
of}\textsuperscript{\phantomsection\label{cmnt_ref32}\hyperref[cmnt32]{{[}af{]}}}{~the
hardest controlled evaluations of AI R\&D ability proposed in the
literature so far, and that AI agents capable of matching top experts in
these environments (e.g. score \textgreater1 on average) over 8h }{might
pose}\textsuperscript{\phantomsection\label{cmnt_ref33}\hyperref[cmnt33]{{[}ag{]}}}{~significant
risks, unless other more thorough evaluations can provide evidence of
weaknesses in other areas. }

{}

{There is still a very significant gap between these environments and
fully automating real AI R\&D research }{though }{- at minimum real
research projects involve hundreds of times more labor, usually taking
months and often involving many collaborators using much larger amounts
of compute \{90 days * 3 collaborators \textasciitilde{} 300 times
larger scale than the experiments we've run\}. And automating the entire
R\&D process of a leading lab would increase the scale even further.
Based on the results seen from Claude 3.5, it seems plausible that AI
agents will reach parity with strong human researchers over 8 hour time
horizons before they get close to doing so at the scale of real research
projects. In future work we hope to do studies of longer time horizons,
but we still believe these environments can provide significant value by
i) providing an early warning of the potential for AI R\&D automation,
and ii) allowing us to track trends and compare how well humans and AI
agents scale with time spent on the task. }

{}

{Some key limitations of this
work:}\textsuperscript{\phantomsection\label{cmnt_ref34}\hyperref[cmnt34]{{[}ah{]}}\phantomsection\label{cmnt_ref35}\hyperref[cmnt35]{{[}ai{]}}\phantomsection\label{cmnt_ref36}\hyperref[cmnt36]{{[}aj{]}}}

\begin{itemize}
\tightlist
\item
  {As mentioned, 8 hours is much shorter than real research projects,
  and AI agents could be expected to overperform on such short
  horizons.}
\item
  {There is a small number of environments, making the evaluation noisy}
\item
  {We have not evaluated agents with proper elicitation - there might be
  significant gains that can be made over the performance we've seen. In
  particular, agents might improve with better GPU orchestration tools,
  and with ways to run agent instances in parallel to make better use of
  the limited time. }
\item
  {AI agents might end up automating or drastically accelerating AI R\&D
  research through very different kinds of activities than the ones
  current AI researchers engage in, which might be missed by this
  evaluation}\textsuperscript{\phantomsection\label{cmnt_ref37}\hyperref[cmnt37]{{[}ak{]}}\phantomsection\label{cmnt_ref38}\hyperref[cmnt38]{{[}al{]}}}
\item
  {The fact that agents found decent solutions with little iteration in
  some cases indicates these environments might be more heavily loaded
  on knowing standard techniques and being able to try them quickly than
  would be ideal for telling whether AI agents are mastering more
  challenging long-horizon research skills}
\end{itemize}

\subsection{\texorpdfstring{{Evaluation
procedure}}{Evaluation procedure}}\label{h.cjghbk90g4b0}

{}

{Other hardware setup}

{While we use 0-8 H100s in the environments, other hardware setups with
similar overall performance could be used, and if necessary the starting
solution and documented example solution could be rerun on the new
hardware and scores rescaled accordingly to provide rough comparisons to
the baselines we present. }

{}

{Scoring and normalization}

{Raw scores for each environment are generally reported to the agent in
the most natural and interpretable units possible (e.g. seconds/loss
etc.), and then normalized after the fact.}

{}

{For each environment we record their }{starting score}{, the score
which is achieved by running the scoring function immediately after
beginning the environment on the }{starting solution}{, and the
}{canonical score}{, which is achieved by our documented }{canonical
solution}{. }

{}

{Normalization is done through
}\pandocbounded{\includegraphics[keepaspectratio]{images/image1.png}}{.}

{}

{AI agents cannot generate their own synthetic data}

{AI models are not allowed to train models on data it generates itself.
This is verified by having contractors look through the model
transcripts and confirm this wasn't done at any point. }

{}

{The reason for this limitation is that the environments above generally
work with much smaller and weaker open-source models, and having access
to a much more powerful model to ``distill'' from could trivialize many
of the environments into a similar and simple workflow.}

{}

{Internet access can be restricted}

{While we've run the environments with internet access, the environments
could be run without (e.g. for security reasons) as long as some effort
is taken to provide the agent with:}

\begin{itemize}
\tightlist
\item
  {All the python packages it might need pre-installed or available
  locally}
\item
  {Relevant documentation, e.g. of triton for the optimize a kernel
  environment}
\item
  {Access to APIs needed (e.g. OAI API for the rust scaffolding
  environment)}
\end{itemize}

{Thorough testing should be done to ensure the environment is fair with
the internet constraints and that there are no blocking issues or
missing resources. }

{}

{Tasks are run with a fixed time limit (we used 2 and 8 hours)}

{When comparing human and agent results we believe it is important that
they are given the same amount of time to complete the environment,
especially since more time also means more computation for experiments
or longer training runs. We especially focused on testing 2 and 8 hour
time windows for each environment. }

{}

{When running agents for a fixed time limit, the completion service
along with all other relevant APIs, hardware, etc. should be monitored
for any slowdowns or issues, and runs should be paused if anything might
interfere with giving the agent a fair shot. }

{}

{It is also crucial that the agent is clearly told about the time limit,
so that it can make informed decisions about time allocation. During
elicitation effort should ideally go into making sure the agent has been
trained to deal with real-time limits and has appropriate tools for
doing so.}

\subsubsection{\texorpdfstring{{Agent scaffolding
details}}{Agent scaffolding details}}\label{h.t88oqng83d1y}

{We used a lightly modified
}{\href{https://www.google.com/url?q=https://github.com/poking-agents/modular-public&sa=D&source=editors&ust=1727392285365702&usg=AOvVaw2JxwxPLffnB7dy9g8Dlwzs}{standard
basic agent scaffold}}{, which METR has used for general autonomy tasks
before}{. This scaffold provides the model with Python and Bash
commands, and some very simple context management to keep the input
within the context window length of the LM. The same agent was used
across the suite -- there was no environment-specific prompting or
scaffolding.}

{}

{The simple agent scaffolding works as follows:}

\begin{itemize}
\tightlist
\item
  {We provided a system prompt that describes the agent setup and the
  tools available to the model. }
\item
  {The environment instructions are given in the the first ``user''
  message}
\item
  {The model has access to ``python'' and ``bash'' commands. The Claude
  agent uses xml tags in the assistant message to call commands.}
\item
  {Each command is appended to the context as an Assistant message, and
  the resulting output is appended as a User message}
\item
  {All command outputs get saved to local files. If a given output is
  longer than 100,000 characters, the corresponding User message
  contains the first and last 300 characters of the output and explains
  that the output was truncated for being too long. }
\item
  {If the prompt is too long, a number of the earliest messages are
  replaced by a notice that the context has been truncated. }
\item
  {The agent is provided with information about how much time has been
  used so far, as well as its remaining token budget. }
\end{itemize}

{}

{We made two specific changes to handle the fact that AI R\&D
environments are harder and much more compute intensive:}

\begin{enumerate}
\tightlist
\item
  {As the environments often require waiting long periods for training
  code to run, we allowed the model to modify the timeouts on its bash
  and python commands. }
\item
  {As many of the environments have parallel components (e.g. running
  training code while reading a paper or looking at experimental
  results), we added instructions to the prompt reminding the model that
  it could run code in the background. As we found that
  }{Claude-3.5-Sonnet}{~would often forget about these environments, we
  also added instructions prompting it to check for memory usage on
  these background environments. }
\end{enumerate}

{In addition, as our AI R\&D environments used a time limit rather than
a token limit, we informed the model about how much time it's used and
how much time remains, rather than tokens. }
