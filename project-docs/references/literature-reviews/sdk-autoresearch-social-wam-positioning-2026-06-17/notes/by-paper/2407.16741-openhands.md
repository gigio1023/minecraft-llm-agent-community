## Primary-source facts (abstract + README + secondary summary; flagged where secondary)

- **What it is** (abstract): "a platform for the development of powerful and flexible AI agents that interact with the world in similar ways to those of a human developer: by writing code, interacting with a command line, and browsing the web." MIT license.
- **Event stream architecture** (secondary summary, EmergentMind): "a flexible event stream architecture" where interactions between users, agents, and environments flow chronologically and form the agent's operational history. This is the platform's core abstraction: actions and observations are a unified, ordered event log that IS the agent state.
- **Agent abstraction** (secondary): "a robust agent abstraction layer, where the state, actions, and observations are clearly defined," so an agent maps state/history to a next action.
- **Sandboxed runtime** (abstract + secondary + README): "safe interaction with sandboxed environments for code execution." Specific action types named in the summary: `CmdRunAction` (shell), `IPythonRunCellAction` (Jupyter), plus browser interaction. README: Docker sandbox is the isolated execution mode; running without it gives the agent "full access to your filesystem" (the warned-against mode).
- **AgentSkills library** (secondary): "utilities that enhance an agent's capabilities like file editing, image parsing, and interaction with IPython notebooks." A toolkit of callable utility functions the agent uses, analogous to a curated action surface.
- **Multi-agent delegation** (abstract + secondary): "coordination between multiple agents"; agents "can collaborate, delegating and receiving tasks from one another," with the example `CodeActAgent` delegating to `BrowsingAgent`.
- **Evaluation** (abstract): agents evaluated over "15 challenging tasks, including software engineering (e.g., SWE-Bench) and web browsing (e.g., WebArena), among others." Benchmark incorporation is a first-class platform feature.
- **Safety** (abstract + README): sandboxed execution is the safety story; the authors flag "security and safety research within AI agents" as future work (i.e. not a solved guarantee). README mentions deployment options ranging from local full-access to Docker sandbox.

## Interpretation (flagged as inference)

- OpenHands is the open-source analog of the Claude Agent SDK / Codex: a builder gets a runtime that owns the sandbox and the event stream, and implements new agents and skills on top. The DESIGN BOUNDARY is the same one this lane cares about: the runtime owns safe execution (the sandbox, the event log, what action types exist); the agent owns choosing actions within that surface.
- The event-stream-as-state design is directly relevant to the repo: the repo's transcripts + verifier artifacts + actor workspace are its event stream. OpenHands shows a mature platform treating the ordered action/observation log as the canonical agent state, not the model's prose memory.
- AgentSkills as a curated utility library maps to the repo's typed action_surface and to the repo's "action skills" the autoresearch loop may author/improve. The lesson is the same as SWE-agent's ACI: capability is shaped by the curated skill/action set, not just the model.

## Mechanically useful vs research contribution

- **Mechanically useful**: the event-stream-as-state architecture (one ordered log of actions+observations is the agent state, not prose); the sandboxed Docker runtime owning safe execution; AgentSkills as a curated, extensible utility surface; explicit multi-agent delegation (CodeActAgent -> BrowsingAgent) as a pattern for a reviewer/sidecar; benchmark incorporation as a platform feature.
- **Not a contribution to copy**: "OpenHands for Minecraft" is a platform import, not a contribution. The platform is a generalist software-developer agent; the repo is a bounded social-Minecraft runtime. Borrow the architecture boundary, not the product.
- **Honest bound**: OpenHands targets a deterministic, testable software environment (shell, files, browser) where the sandbox and tests give clean truth. Its safety story is sandboxing, and the authors themselves defer safety guarantees to future work. It supplies no social-outcome verifier; the repo's upper-layer truth problem is outside OpenHands' design.
