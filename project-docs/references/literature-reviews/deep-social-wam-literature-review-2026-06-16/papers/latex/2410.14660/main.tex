\begin{abstract}
Large Language Models (LLMs) have shown significant potential in designing reward functions for Reinforcement Learning (RL) tasks. However, obtaining high-quality reward code often involves human intervention, numerous LLM queries, or repetitive RL training. To address these issues, we propose CARD, a LLM-driven Reward Design framework that iteratively generates and improves reward function code. Specifically, CARD includes a Coder that generates and verifies the code, while a Evaluator provides dynamic feedback to guide the Coder in improving the code, eliminating the need for human feedback. In addition to process feedback and trajectory feedback, we introduce Trajectory Preference Evaluation (TPE), which evaluates the current reward function based on trajectory preferences. If the code fails the TPE, the Evaluator provides preference feedback, avoiding RL training at every iteration and making the reward function better aligned with the task objective. Empirical results on Meta-World and ManiSkill2 demonstrate that our method achieves an effective balance between task performance and token efficiency, outperforming or matching the baselines across all tasks. On 10 out of 12 tasks, CARD shows better or comparable performance to policies trained with expert-designed rewards, and our method even surpasses the oracle on 3 tasks.
\end{abstract}


\section{Introduction}

Reinforcement Learning (RL) has been successfully applied to various tasks with well-defined reward functions~\cite{mnih2015human,silver2016mastering,berner2019dota}. However, such reward functions do not exist for many real-world scenarios. A common approach is to manually design a reward function, known as reward engineering~\cite{laud2004theory,gupta2022unpacking}, but this requires extensive human knowledge and efforts. To tackle this problem, prior work has explored learning reward functions via inverse RL~\cite{ziebart2008maximum,wulfmeier2015maximum,finn2016guided,GAIL,AIRL} and \pbRL~\cite{christiano2017deep,ibarz2018reward,PEBBLE,Meta-Reward-Net}. However, inverse RL requires high-quality demonstrations and \pbRL often needs a large number of preference labels, both of which limit their use in practical applications.

Recently, Large Language Models (LLMs) have been demonstrated to be effective in generating reward function code for RL tasks~\cite{L2R,Text2Reward,Eureka,li2024auto}. However, the hallucination problem~\cite{rawte2023survey,huang2023survey,liu2024exploring} makes it difficult to generate an effective reward function with a single query. Existing methods require human feedback or incur high costs to improve the quality of generated reward functions. For example, \cite{L2R} relies on humans to manually design Motion Description templates and reward APIs, and the quality of the reward function is directly influenced by the accuracy of these designs. \cite{Text2Reward} requires humans to improve the reward function by analyzing agent trajectories, while \cite{li2024auto} leverages LLMs to analyze trajectories to refine the reward function. \cite{Eureka} identifies the most effective reward function using ground-truth rewards after repeated experiments, which significantly increases token consumption and RL training costs. Reducing human involvement, token consumption, and repetitive RL training while maintaining task performance remains a great challenge in LLM-based reward design.

To address these issues, we propose a LLM-based \textbf{C}oder-Ev\textbf{A}luator \textbf{R}eward \textbf{D}esign framework, named \ourmethod. \ourmethod consists of a LLM-based \designer to generate reward function code and a \critic to evaluate the quality of the code, as illustrated in Figure~\ref{fig:pipeline}.
The \designer queries the LLM using environment description and task goals to generate initial reward function code, which is then verified for successful execution. The \critic dynamically provides feedback based on the quality of the reward function, without requiring additional LLM queries or human involvement. The \designer iteratively improves the reward function based on the provided feedback. We introduce Trajectory Preference Evaluation (\TPE), which evaluates the reward function with trajectory preferences, allowing the \critic to assess reward functions without additional RL training. If the reward function does not pass the \TPE, preference feedback is provided to improve the code, while process and trajectory feedback are provided otherwise. Specifically, process feedback records changes in parameters, such as trajectory return and sub-rewards during RL training. This allows the \designer to better understand the overall trends of the training process and evaluate the effectiveness of sub-reward components. Trajectory feedback provides parameter details for each step of successful and failed trajectories, enabling the \designer to compare the impact of sub-reward components on these trajectories. Preference feedback evaluates the reward function using trajectory pairs and preferences via \TPE without RL training results. The differences between \ourmethod and previous methods are listed in Table~\ref{tab:method_comparison}.
The contributions of this paper can be summarized as follows:
\begin{itemize}
    \item We propose a LLM-based reward design framework that generates and iteratively refines reward code without human feedback or parallel LLM queries.
    \item We propose Trajectory Preference Evaluation, enabling the \critic to dynamically provide feedback for reward function improvement, eliminating repetitive RL training.
    \item Empirical results on multiple tasks of \MW and \MS show that \ourmethod outperforms the baselines and even exceeds the human oracle.
\end{itemize}

\section{Related Work}

\paragraph{Reward Design.}
Reward engineering remains a significant challenge in RL~\cite{laud2004theory,singh2009rewards,sutton2018reinforcement,gupta2022unpacking}.
The quality of the reward function is crucial for RL algorithms, especially for applications in real-world scenarios. Numerous research works investigate the construction of high-quality reward functions from diverse perspectives. In Imitation Learning (IL),~\cite{ziebart2008maximum,wulfmeier2015maximum,finn2016guided,GAIL,AIRL} extract a reward function from expert demonstrations via inverse RL.
In offline IL setting,~\cite{OTR} obtains rewards via optimal transport.~\cite{SEABO} utilizes a search-based method to design a reward function. These methods require expensive expert demonstrations. \PbRL learns from preference relations to obtain reward functions~\cite{christiano2017deep,ibarz2018reward,PEBBLE,Meta-Reward-Net,PEARL}. However, \pbRL often requires a large number of preference labels, which is expensive and time-consuming for human experts to provide.

\ourmethod differs from the above methods, as it does not require high-quality demonstration of preference data. Instead, \ourmethod leverages the impressive comprehension and generation capabilities of LLM to automatically generate and enhance reward functions without human knowledge.


\paragraph{Large Language Models for Reinforcement Learning.}
Many previous studies attempt to use LLM to assist in RL training. \cite{ma2022vip,fan2022minedojo,ma2023liv,du2023vision,karamcheti2023language,ELLM,kwon2023reward} utilize pretrained foundation models to generate reward signals for RL tasks. However, the agent needs a large number of samples from the environment during learning process. Frequent queries to the LLM not only decrease training efficiency but also lead to significant token consumption. Additionally, most of these methods generate scalar reward values, which are difficult to interpret and improve. Other approaches focus on solving robot control~\cite{liang2023code,huang2023instruct2act,wang2023demo2code} and goal planning~\cite{lin2023text2motion,ding2023task,xie2023translating,liu2023llm+,singh2023progprompt,wang2023voyager,silver2024generalized} problems by generating clear and structured program code. However, most of these works focus on executing robotic actions based on known motion primitives instead of learning low-level skills. Recent studies~\cite{L2R,Text2Reward,Eureka,li2024auto} address the challenge of robotic low-level manipulation in RL by creating reward functions using LLMs. However, these methods require human knowledge or lead to significant token consumption. In contrast, \ourmethod can generate reward function code and automatically provide feedback on its quality without human involvement or LLM analysis. Additionally, \ourmethod utilizes \TPE to eliminate unnecessary RL training, which further reduces the cost.

\section{Problem Setup}

We consider the standard RL setting, which is formulated as a Markov Decision Process (MDP). The MDP is defined by a tuple $(\mathcal{S}, \mathcal{A}, \mathcal{P}, \mathcal{R}, \gamma)$, where $\mathcal{S}$ is the state space, $\mathcal{A}$ is the action space, $\mathcal{P}$ is the transition dynamics, $\mathcal{R}$ is the reward function, and $\gamma \in [0,1)$ is the discount factor. The agent interacts with the environment at discrete time steps. At each time step $t$, the agent observes the state $s_t$, selects an action $a_t$ based on the policy $\pi$, and receives a reward $r_t$. The objective of the agent is to maximize the expected return $G = \sum_{t=0}^T \gamma^t r_t$, where $T$ is the horizon.


\section{Method}\label{sec:method}

In this section, we present \ourmethod, which consists of a \designer for generating reward function code and a \critic for evaluating its quality through an iterative process. \ourmethod involves three key steps: (1) Reward Design—the \designer generates the reward function code based on descriptions of the environment and task goals, followed by verification of its correctness; (2) Reward Introspection—the \critic provides various types of feedback on the quality and effectiveness of the trained policy, without relying on LLM queries or human intervention; and (3) Reward Improvement—the \designer refines the reward function code based on the feedback, forming an automatic refinement cycle. We first introduce the reward design process (Section~\ref{sec:reward_design}), followed by the reward introspection process (Section~\ref{sec:reward_introspection}), the reward improvement process (Section~\ref{sec:reward_improvement}), and finally, the practical implementation (Section~\ref{sec:practical_implementation}). The overall pipeline of \ourmethod is illustrated in Figure~\ref{fig:pipeline}, and the pseudo code for the algorithm is provided in Appendix~\ref{app:full_algorithm}.

\subsection{Reward Design}\label{sec:reward_design}

\paragraph{Code Generation.}

At this stage, the \designer queries the LLM to obtain the unchecked reward function code. However, directly generating domain-specific reward functions demands relevant prior knowledge, such as the configuration of the environment and the agent. Therefore, the \designer is provided with an environment description in Pythonic style, as shown in Figure~\ref{fig:pipeline}. Specifically, the environment abstraction shows callable variables, callable functions, and the inheritance relationship between different objects in the form of comments, static variable types, and Python classes.
Then, the \designer is provided with a task description and reward function template, querying the LLM to generate reward code that can guide the agent to complete the task. The \designer requires that the generated reward function not only returns a scalar reward signal but also specifies the value of each sub-reward~\cite{Eureka}. This guides the LLM to design reward code from the perspective of multiple reward components. The values of sub-rewards are used for \critic design feedback, detailed in Section~\ref{sec:reward_introspection}. We refer to the environment abstract description and task description of \MW and \MS in \cite{Text2Reward} and make
modifications to adapt them to our framework. For prompting details, please refer to Appendix~\ref{app:system_and_generation_prompt}.

\paragraph{Correctness Check.}

The reward function code generated by the LLM may occasionally contain syntax errors (e.g., usage of undefined variables) or runtime errors (e.g., variable type mismatch, matrix dimension mismatch), which can hinder downstream RL training. One solution is to provide error information as feedback to the LLM to fix the code~\cite{le2022coderl,olausson2023demystifying}, but this may lead the LLM to alter the original logic to correct the error. Another method is to generate multiple independent reward functions in parallel to ensure that at least one is correct~\cite{Eureka}. However, ensuring correct execution may require repeating the generation process multiple times, leading to unnecessary token usage. Unlike these methods, the \designer provides comments and type annotations for variables in the environment abstraction and reward function template to help the LLM clarify the environment settings. For unchecked reward functions, the \designer uses lightweight dynamic execution tests to check for syntax and runtime errors. If a test reports an error, the result is discarded, and the code is regenerated. The \designer will only re-initiate a query when the test fails, rather than generating a large amount of code at once. This ensures correct code is found without wasting tokens. The design details are in Appendix~\ref{app:implementation_details}. We compare the accuracy of reward design by different algorithms in Section~\ref{sec:execution_error_rate}.

\subsection{Reward Introspection}\label{sec:reward_introspection}


Due to the hallucination problem, the LLM may not always provide the optimal reward function in a single query. Previous methods either use LLM feedback~\cite{li2024auto} or human feedback~\cite{Text2Reward} to refine reward code, or select the best reward function from a set of independent experiments~\cite{Eureka}. To avoid human feedback or additional LLM queries, \ourmethod leverages \critic to automatically generate feedback through introspection. 
Specifically, the \critic generates three types of feedback: process feedback, trajectory feedback, and preference feedback, as illustrated in Figure~\ref{fig:pipeline}. Details of these feedback types are provided in Appendix~\ref{app:introspection_prompt}.


\subsubsection{Process Feedback}

Since the training curve reflects the quality of the reward function, we use the training results as feedback for evaluating the reward function. During RL training, the \critic evaluates the current policy by recording the average return, average trajectory length, and average success rate at each evaluation step, similar to~\cite{Eureka}. Instead of relying solely on the final result, these dense indicators help the \designer understand how the reward function influences the training process and identify areas for adjustment. Additionally, the \critic monitors the average cumulative value of each sub-reward along the trajectory, enabling specific adjustments to ineffective sub-rewards. The \critic then combines data, including average return, trajectory length, success rate, and the cumulative values of sub-rewards, into structured text in a natural language format as process feedback at regular intervals. Further details on the construction of process feedback can be found in Appendix~\ref{app:introspection_prompt}.


\subsubsection{Trajectory Feedback}

The process feedback provides detailed information about the training process. \ourmethod additionally incorporates feedback by analyzing sampled trajectories and such kind of comparison of reward values and states across trajectories is valuable for improving the reward function~\cite{li2024auto}. After RL training, \ourmethod rolls out trajectories using the trained policy and selects those with the highest and lowest returns to generate trajectory feedback. Specifically, the \critic gathers overall information about the trajectory, including return and success flags, while also sampling reward values, sub-reward values, and various observation parameters at intervals. This collected data is then integrated into natural language text as trajectory feedback. Since tasks with continuous state spaces may show minimal state changes per step, the sampling interval is treated as a hyperparameter. Details on the construction of trajectory feedback are provided in Appendix~\ref{app:introspection_prompt}. Unlike~\cite{li2024auto}, the \critic efficiently constructs trajectory feedback offline without relying on an LLM-based \textit{Trajectory Analyzer}, which improves the efficiency of reward introspection and reduces token costs.


\subsubsection{Trajectory Preference Evaluation and Preference Feedback}

Process feedback and trajectory feedback are both constructed based on RL training results, offering guidance for improving the reward function. Iteratively updating the reward function by training RL agents at each iteration is highly inefficient. In addition to repeated RL training, \eureka identifies the optimal reward function by comparing the trajectory return with the ground-truth reward. To address these issues, we introduce Trajectory Preference Evaluation (\TPE), which evaluates the reward function using trajectory preferences without requiring a ground-truth reward or repetitive RL training. The key insight behind \TPE is that some trajectories successfully achieve the goal, while others do not, naturally providing trajectory preference relations. \TPE offers a clear criterion for evaluating the quality of a reward function: if the return of successful trajectories exceeds that of unsuccessful ones when computed using the current reward function, then the reward function is considered order-preserving.

In many goal-conditioned environments, episodes are typically terminated upon task completion. As a result, successful trajectories tend to be shorter than unsuccessful ones. This can lead to longer, unsuccessful trajectories accumulating more rewards simply due to their length, even though they fail to complete the goal within a specified steps. To mitigate the effect of trajectory length, we compare the average per-step return rather than the cumulative return. The following provides a formal definition of order-preserving reward functions.

\begin{definition}

Given a set of trajectories $\tau = \{\tau_1, \tau_2, \dots, \tau_N\}$ where trajectories can have different lengths, a set $\tau^{+} \subseteq \tau$ containing successful trajectories, and a set $\tau^{-} \subseteq \tau$ containing unsuccessful trajectories, a reward function $r(s,a)$ designed by the \designer is order-preserving if the following condition holds:

\begin{equation}
\begin{aligned}
\min_{\tau_{i}^{+} \in \tau^{+}}\left(\frac{1}{|\tau_i^+|}\sum_{t=0}^{|\tau_i^+|} \gamma^{t} r(s_{it}^+, a_{it}^+)\right) > \max_{\tau_{j}^{-} \in \tau^{-}}\left(\frac{1}{|\tau_j^-|}\sum_{t=0}^{|\tau_j^-|} \gamma^{t}r(s_{jt}^-, a_{jt}^-)\right)
\end{aligned}
\end{equation}
where $\tau_{i}^{+}=\{(s_{i0}^+, a_{i0}^+), (s_{i1}^+, a_{i1}^+), \dots, (s_{iT}^+, a_{iT}^+)\}$, $\tau_{j}^{-}=\{(s_{j0}^-, a_{j0}^-), \\ (s_{j1}^-, a_{j1}^-), \dots, (s_{jT}^-, a_{jT}^-)\}$.
\end{definition}


Using \TPE, the \critic can efficiently verify the effectiveness of the improved reward function. Specifically, if the designed reward function is order-preserving, trajectory preference sorting is considered to be reasonable and consistent with human expectations, making it more likely to improve the reward function. 
Since this process avoids redundant RL training and ground-truth rewards, preference feedback reduces both the cost of evaluating rewards and the dependence on prior knowledge. With the results of \TPE, the preference feedback includes the ratio of successful trajectory returns being higher than those of failed trajectories and provides details of two trajectories in the trajectory feedback format. Further details on preference feedback construction are presented in Appendix~\ref{app:introspection_prompt}.




\subsection{Reward Improvement}\label{sec:reward_improvement}

\designer of \ourmethod preserves a historical record of dialogues with the LLM, which includes system prompt, code generation prompt, and feedback prompts for each reward introspection. Through comprehensive context records, LLM gains an exhaustive understanding of the reward function enhancement process and is capable of immediately introspecting when negative optimization occurs.


\designer queries the LLM based on this feedback to improve the reward function. This process does not require direct LLM or human involvement, and only one or a minimal number of independent experiments is needed to achieve comparable or better results than earlier approaches. 

\subsection{Practical Implementation}\label{sec:practical_implementation}

Initially, \critic uses the code generated by \designer, which may not be optimal, to conduct RL training and collect trajectories. Process feedback and trajectory feedback are then constructed to allow \designer to refine the code. Each time \designer improves the reward function, \critic conducts \TPE on the updated code. If the reward function is order-preserving, RL training is performed, and process feedback and trajectory feedback are provided to \designer. \critic also records the generated trajectories for subsequent \TPE analysis. If the reward function is not order-preserving, it is considered unreasonable, and RL training is skipped to avoid unnecessary cost. Instead, preference feedback is provided, informing the LLM that the updated reward function may deviate from the optimal. The whole algorithm of \ourmethod with three types of feedback is shown in Appendix~\ref{app:full_algorithm}.

In implementation, we count the number of successful task trajectories whose average per-step return is greater than that of failed task trajectories to calculate the accuracy of the preference order. If the accuracy exceeds a threshold, the reward function is considered order-preserving. This approach provides flexibility in refining reward functions, allowing ample opportunities for modifications. During the first reward introspection, \critic conducts RL training directly and collects trajectories, as there are no initial trajectories for \TPE evaluation. Starting from the second reward introspection, the \TPE results determine whether RL training is necessary. \TPE significantly reduces unnecessary RL training and shortens the time spent on reward introspection. The results in Section~\ref{sec:results} show that alternating between process feedback, trajectory feedback, and preference feedback improves \designer's understanding of the environment, task goals, and reward functions, leading to the development of more effective reward functions.


\section{Experiments}\label{sec:experiments}

In this section, we conduct experiments to answer the following questions:
(\textbf{Q1}) How does \ourmethod perform on the evaluation tasks compared to the baselines?
(\textbf{Q2}) How does \ourmethod perform with respect to token efficiency?
(\textbf{Q3}) How sensitive is \ourmethod to its key parameters and components?
(\textbf{Q4}) How does \ourmethod perform in terms of execution error rate of the generated reward function code?
(\textbf{Q5}) How does \ourmethod improve the reward function with dynamic feedback?


\subsection{Experimental Setup}\label{sec:experimental_setup}

Following~\cite{Text2Reward}, we evaluate on $6$ robotic manipulation tasks of \MW~\cite{Meta-World} and $6$ robotic manipulation tasks of \MS~\cite{ManiSkill2}. Detailed descriptions of these tasks are provided in Appendix~\ref{app:task_details}.

\paragraph{Baselines.}
To evaluate the performance of \ourmethod, we compare it against the following baselines:
1) \textbf{Oracle}: The reward function is designed by human experts.
2) \textbf{\lr}~\cite{L2R}: \lr generates reward functions by combining human-defined sub-reward components.
3) \textbf{\tr}~\cite{Text2Reward}: \tr generates reward function code using zero-shot or few-shot methods by providing a textual description of the environment and task instructions.
4) \textbf{Eureka}~\cite{Eureka}: Eureka employs an evolutionary search strategy to improve the reward function by selecting the best-performing one from $16$ independent trials.


For fair comparison, all experiments are conducted in a zero-shot setting and \texttt{GPT-4-1106-preview}~\cite{GPT-4} API is used for \tr, \eureka, and \ourmethod. For \lr, we provide detailed task instructions for the Motion Descriptor and organize oracle reward components into separate reward APIs for the Reward Coder. This allows the LLM to perceive almost all relevant environment details, enabling \lr to perform similarly to Oracle. For other methods, we provide the same environment abstraction as \cite{Text2Reward}. Therefore, the range of variables that can be used when generating the reward function is the same, ensuring the fairness of the comparison. For \tr, we use both the human-improved code provided by~\cite{Text2Reward} and the generated code without human feedback (denoted as \textbf{Reproduce} in Section~\ref{sec:results}). We apply the same RL algorithm (PPO or SAC) and set the same hyperparameters as \tr~\cite{Text2Reward} across all methods. For \ourmethod, we set $\delta$ to $0.8$ and run $2$ iterations to refine the reward function code.
The experiments are conducted on a single NVIDIA RTX 3090 GPU with $8$ CPU cores.

\subsection{Results (Q1)}\label{sec:results}

As shown in Figure~\ref{fig:metaworld_comparison} and Figure~\ref{fig:maniskill_comparison}, \ourmethod consistently achieves better or similar performance than other baselines on all tasks, while \ourmethod is also more cost-efficient. Notably, on the \textit{Sweep Into} task, \ourmethod demonstrates significant improvement compared to the baselines. On 3 out of 12 tasks, \ourmethod even surpasses oracle. For the remaining tasks, \ourmethod still performs comparably to oracle. We hypothesis that the reward function code generated by \ourmethod is highly effective given the available information, leaving minimal room for further improvement.


\subsection{Token Efficiency (Q2)}

To further compare the cost of LLM queries of different algorithms, we compute the total input and output tokens consumed. \lr queries the LLM once using the Motion Descriptor and once using the Reward Coder, so we sum the cost of both queries as the overall cost. \tr, without human feedback improvement, performs only one LLM query. \eureka generates initial reward code and refines it for $5$ iterations and each generation contains $16$ parallel sampling. Therefore, the token usage of \eureka is computed by the sum of $96$ queries. For \ourmethod, we measure the token consumption over initial generation and $2$ rounds of reward introspection, which requires $3$ LLM queries. In our experiments, token usage is calculated from 16 independent runs for both \lr and \tr, and we report the average values. Since both \eureka and \ourmethod involve iterations, we measure token consumption from one run due to cost limitations.

Table~\ref{tab:token_usage_simple} summarizes the average token consumption of different algorithms for LLM queries across all evaluation tasks. Although Text2Reward without iterative improvement consumes the least tokens, it still incurs additional token consumption and high labor costs when human feedback is introduced. \lr also consumes a small number of tokens because LLM only needs to select the reward API without requiring details of the environment settings. \subsection{Ablation Study (Q3)}\label{sec:ablation}

\paragraph{Different number of iterations.}

To investigate how different number of iterations influences \ourmethod, we conduct experiments with iterations of $\{1,2,3\}$. The results in Figure~\ref{fig:ablation_iter} indicate that the quality of the reward function improves during the initial iterations and then gradually converges. In particular, on 7 out of 9 tasks, the results after the second iteration are either the best or comparable to those of the other iterations. Significant improvements are only observed in the third iteration for the more challenging tasks, such as \textit{Door Unlock} and \textit{Sweep Into}. However, on \textit{Handle Press Side} and \textit{Turn Faucet}, performance declines considerably after the third iteration. We hypothesis that after generating a well-shaped reward function, further refinement by the LLM may introduce unwanted changes, leading to a drop in performance. Therefore, for more difficult tasks, increasing the number of iterations is beneficial for fully refining the reward function, whereas for simpler tasks, limiting the number of refinement iterations may be more effective.

\section{Conclusion}

In this paper, we propose \ourmethod, a LLM-driven reward design framework that iteratively generates and refines reward function code without relying on redundant LLM queries or human feedback. The framework includes \designer, which is responsible for generating reward functions, and \critic, which evaluates these functions using dynamic feedback. In \ourmethod, we introduce Trajectory Preference Evaluation, which enables \critic to assess the reward functions without requiring RL training at every iteration, thus improving efficiency. The \critic provides multiple types of feedback, guiding the \designer in refining the reward code, creating a self-improving and automated process. Empirical results demonstrate that \ourmethod outperforms or matches the baselines on most tasks, and even surpasses the oracle on some tasks. Future work will explore extending \ourmethod to more complex environments and tasks.

\section{Algorithm}\label{app:full_algorithm}

The full algorithm of \ourmethod is presented in Algorithm~\ref{alg:algorithm}.

\begin{algorithm}[!htbp]
\caption{\ourmethod}
\label{alg:algorithm}
\begin{algorithmic}[1] %[1] enables line numbers
\Require Environment description $M$, task instruction $l$, chat model in LLM $Coder$, offline text formatter $Evaluator$, system prompt $P_{\text{s}}$, instruction prompt $P_{\text{i}}$, feedback prompt $P_{\text{f}}$, preference evaluation function $F_p$, reinforcement learning training function $F_t$, Chat history $\mathcal{L}$, trajectories $\mathcal{T}$, iteration number $N$
\State $\mathcal{L}=\{P_{\text{s}}(M), P_{\text{i}}(l)\}$
\State // Reward Design
\State $R=\text{\designer}(\mathcal{L})$
\State $\text{result}=F_t(R)$
\State $\mathcal{T} \leftarrow \mathcal{T} \cup \{\text{result}\}$
\For{$N$ iteration}
\State // Reward Introspection
\State $\text{preference}=F_p(R)$
\If {$\text{preference}$}
\State $\text{result}=F_t(R)$
\State $\mathcal{T} \leftarrow \mathcal{T} \cup \{\text{result}\}$
\State $\mathcal{L} \leftarrow \mathcal{L} \cup \{P_{\text{f}}(\text{result})\}$
\Else
\State $\mathcal{L} \leftarrow \mathcal{L} \cup \{P_{\text{f}}(\text{preference})\}$
\EndIf
\State // Reward Improvement
\State $R=\text{\designer}(\mathcal{L})$
\EndFor
\Ensure Reward function code $R$
\end{algorithmic}
\end{algorithm}

\subsection{Implementation Details}\label{app:implementation_details}

Since the \texttt{GPT-4-0314}~\cite{GPT-4} API used by~\cite{Text2Reward} has been deprecated~\footnote{\url{https://platform.openai.com/docs/models}}, we use \texttt{GPT-4-1106-preview}~\cite{GPT-4} API in all methods. And we use \texttt{GPT-3.5-turbo-1106} in Section~\ref{sec:ablation} to investigate the impact of different LLM APIs on \ourmethod. The temperature of sampling is set to $0.7$ in all tasks. For each generated reward function, we use dynamic execution to detect runtime errors. In the event of an error in the reward function code, we initiate a re-query to the LLM. The hyperparameter \textit{max\_try\_num} serves to prevent excessive costs associated with querying the LLM by either generating an executable code or ceasing the process after \textit{max\_try\_num} attempts. Throughout our experiments, we set \textit{max\_try\_num} to $10$.

Process feedback includes the average evaluation results of the agent, such as reward, episode length, success rate and each reward term. Preference feedback provides the information of sorting the reward of successful trajectories and failed trajectories.     total_reward = distance_reward + lifting_reward + action_penalty + grasp_reward
    # Return the total reward and the dictionary of individual reward components
    return total_reward, {
        'distance_reward': distance_reward,
        'lifting_reward': lifting_reward,
        'action_penalty': action_penalty,
        'grasp_reward': grasp_reward,
    }

\section{Limitations and Future Work}

Our research substantiates the efficacy of employing LLMs for the enhancement of dense functions in RL while eliminating human feedback. We design three distinct feedback categories, enabling the LLM to formulate reward functions composed of sub-rewards and to achieve introspection using the available information. Therefore, the reward improvement process exhibits high interpretability. Simultaneously, we remove invalid RL training and significantly improve the efficiency of introspection. 
However, as indicated in Table~\ref{tab:token_usage_detailed}, the token consumption of our method grows linearly with increasing iterations. Consequently, tasks necessitating extensive iterations to refine reward codes may incur substantial token usage. Moreover, our method presumes that the perception of environmental information is known a priori.
For future work, we consider utilizing Vision-Language Models (VLMs) to generate reward function with both vision and language information.
