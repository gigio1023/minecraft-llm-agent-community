## Primary-source facts (what the paper states/shows)

- Problem ("write-before-query barrier"): long-horizon agents compress interaction histories into write-time summaries. The compression decision is made before the system knows what a future query will hinge on. Result: unverifiable omissions. A decisive constraint (the paper's running example: a "severe peanut allergy" compressed into generic "dietary preferences") can be dropped, leaving the agent unable to trace or cite the specific fact. The paper's claim: any fixed-budget summary admits a worst-case query it cannot support with traceable evidence. This is framed as a structural risk of lossy compression, not occasional summarizer error.
- The other extreme (always ground on raw logs) is auditable but wasteful: long inputs inflate tokens and latency and can degrade context utilization (lost-in-the-middle), and most queries do not need raw evidence.
- TierMem: a provenance-linked two-tier memory that treats retrieval as inference-time evidence allocation (answer with the cheapest sufficient evidence).
  - Tier-2 (M_R): immutable paged raw log, fixed-size pages (~1000 tokens) each with a stable identifier. The authoritative source of truth. Pages are immutable so evidence does not shift under later reorganizations, enabling stable citation.
  - Tier-1 (M_S): summary index. Each entry = (compact summary, dense embedding, provenance links rho pointing to the Tier-2 page identifiers that support it). Construction is synchronous: when a raw page is sealed, a summary entry is synthesized and its links recorded, so no summary exists without a traceable raw source.
  - Router policy pi_theta (lightweight "miss detector"): given query + retrieved Tier-1 summaries, outputs Answer (Tier-1 only) or Escalate. Trained to escalate when the summary lacks precise constraints (negations, exact values, attribution).
  - Escalation is provenance-guided: prioritize reading the Tier-2 pages explicitly linked by the retrieved summaries (a high-recall warm start), then bounded multi-hop retrieval over Tier-2 (up to T_max = 3) if still insufficient.
  - Verified write-back: after a successful escalation, distill only grounded findings into new Tier-1 units linked back to supporting raw pages (retrieve-and-edit: ADD / UPDATE / SKIP). The write-back is query-triggered and evidence-backed.
- Router training: hindsight labeling (compare a summary-only answer vs a raw-grounded answer with an LLM judge; label Escalate only when the summary answer is wrong/vague but raw recovers it). Two stages: SFT distillation from a GPT-5 teacher rationale, then cost-aware GRPO with reward R = R_acc - lambda_cost * R_cost - lambda_waste * R_waste (penalizes both escalation cost and false-alarm escalations).
- Numbers as stated:
  - LoCoMo: TierMem (router) 0.851 overall accuracy vs 0.873 raw-only upper bound, while reducing average input tokens by 54.1% and latency by 60.7% (3,396 vs 7,398 tokens; 6.76s vs 17.18s). Summary-only drops to 0.755 (15.4% unverifiable-omission rate). Summary-centric baselines have UOR 14.7-23.3%.
  - LongMemEval: raw-only 0.808 vs router 0.752; summary-only methods show UOR up to 29.6% (Mem0). Always-summary baselines degrade sharply under long-range updates.
  - Router (SFT+GRPO): escalates 39.0% of queries, recalls 71.7% of oracle-hard cases, overhead ~678 tokens/query (14.7% of input tokens). Matches the GPT-4.1-mini router's 0.851 accuracy.
  - Provenance-pointer ablation: Linked 85.1% vs No-Linked 83.6% overall (+1.5 pp), with the gain concentrated on escalated queries: Acc@R 77.5% -> 81.7% (+4.2 pp), Answer-path accuracy unchanged. The intended causal role: pointers do not change what summaries contain, they improve how fast and reliably the system finds decisive Tier-2 evidence.
  - Consolidation over replay epochs: with verified write-back, summary-path correct answers rise 32% (819 -> 1,083) and Tok_avg drops 3,958 -> 2,419, latency 5.14s -> 3.39s, while overall accuracy stays stable. Retrieve-and-edit (ADD/UPDATE/SKIP) keeps Tier-1 more coherent than blind ADD.
  - Error analysis: summary-only makes 378/1540 errors (24.5%); root causes are information missing (41.3%) and over-generalization (28.8%), both direct manifestations of write-before-query. Of the router's Answer decisions on these failures, many are "false cache hits" (router answered from Tier-1 when it should have escalated), the dominant remaining router failure mode is insufficient strictness in sufficiency checking.

## Interpretation (my inference, labeled)

- This is the strongest academic instance of the repo's "evidence-linked memory" rule. The repo requires that every memory cite a concrete artifact rather than be free-form prose; TierMem makes that an architectural invariant (no Tier-1 summary exists without provenance links to immutable Tier-2 pages) and shows it is not just hygiene but measurably improves grounding (+4.2 pp on the cases that need raw evidence).
- The "write-before-query barrier" is the precise failure mode the repo avoids by keeping the deterministic runtime artifacts (inventory/container/chat/verifier ids) as the immutable source of truth. A free-prose social-memory entry ("Bob is trustworthy") is exactly a write-time summary that drops the decisive evidence; TierMem's argument is that you must retain the immutable raw event and link to it.
- The immutability + stable-identifier discipline is directly portable: Mineflayer world/inventory snapshots and chat logs are the natural Tier-2 (immutable, stably addressable), and any distilled social/material ledger value is a Tier-1 summary that must carry a pointer back to the verified event id.

## Mechanically useful vs research contribution

- Mechanically useful: provenance links as a hard invariant (no summary without a pointer to an immutable raw artifact); immutable paged raw store with stable ids as the source of truth; query-triggered, evidence-backed write-back (only commit what was needed and validated); a sufficiency router that escalates to raw evidence when the cheap distilled view is underspecified. All map onto an actor-workspace where distilled memory must cite verifier ids.
- Research contribution it is NOT for this repo: TierMem is a conversational-QA memory system (text histories, LLM-judge accuracy). It has no physical/material/social typed state, no transition prediction, no multi-agent grounding. Its contribution is the accuracy-efficiency frontier for retrieval; the repo's contribution is verifier-grounded prediction. TierMem is evidence-plumbing the repo can adopt, not the predictive object.

## WAM layer(s) informed

- Cross-layer plumbing (the memory-faithfulness invariant). Most relevant to Social and Institutional layers, where distilled values (trust, norms, routines) most tempt free-prose summarization and most need a pointer back to the verified physical/material event that justifies them.
