## Primary-source facts

- **Research question**: how well can AI agents do autonomous end-to-end machine learning engineering (MLE), measured against real human performance.
- **Benchmark shape**: 75 hand-curated Kaggle competitions (22 Low / 38 Medium / 15 High complexity; +7 dev split), spanning NLP, CV, signal processing. Each sample = a competition description + dataset (usually a new train-test split) + local grading code + a snapshot of the competition's Private leaderboard.
- **Success signal (external)**: submissions are graded LOCALLY by per-competition grading code, then ranked against the real human Private leaderboard. Headline metric = percent of attempts awarded any Kaggle medal (bronze+). Medal thresholds are leaderboard-relative (Table). The metric is deliberately hard: only nine humans in history have ever medaled in 75 different competitions.
- **Setup**: agents run in an Ubuntu Docker container with the dataset, a local validation server (checks submission validity, gives no score, unlimited calls), and ML packages; 24h per competition; 36 vCPUs, 440GB RAM, single Nvidia A10 GPU; 3 seeds default.
- **Headline results**: best = o1-preview + AIDE scaffold = 16.9% any-medal (pass@1); GPT-4o+AIDE 8.7%. Pass@k roughly doubles the rate (o1-preview pass@1 16.9% -> 34.1% at higher k per intro; GPT-4o pass@6 ~= o1-preview pass@1). o1-preview averages 7 gold medals (Kaggle Grandmaster needs 5), with caveats.
- **Scaffold matters more than expected**: AIDE (purpose-built tree search over Kaggle solutions) beats general-purpose MLAB (0.8%) and OpenHands (4.4%) with GPT-4o; general scaffolds often end runs early or fill context inspecting huge files. More time helps (GPT-4o 8.7% at 24h -> 11.8% at 100h); extra GPU does not (agents did not use the 2nd GPU).
- **Anti-cheating rules and detectors**: submissions must come from a model the agent trains (no recalling memorized labels); viewing online solutions is forbidden. Enforcement = a GPT-4o log-analysis tool (flags manual submission writing, external API calls, unauthorized access) + Dolos source-code plagiarism detector (disqualify >60% similarity to top-50 public notebooks).
- **Contamination study**: (1) familiarity = mean token probability the base model assigns to the competition page + 5 winner discussion posts; found NO correlation between GPT-4o familiarity and performance. (2) Obfuscated all 75 descriptions (remove Kaggle/name references); performance essentially unchanged (8.5% -> 8.4%). Plagiarism detector and log-analysis found no rule-breaking on medal-winning runs. Conclusion: no systematic inflation for GPT-4o, with honest caveats.
- **Stated limitations**: cannot rule out subtle high-level-strategy contamination or future models having memorized the test set (GPT-4 base could reproduce Titanic dataset rows); MLE-bench selects clean, well-specified problems unlike open-ended real R&D; new train-test splits differ from the original competitions; very resource-intensive (1800 GPU-hours per main run; o1-preview+AIDE ~127.5M input + 15.0M output tokens per seed).

## Interpretation (flagged as inference)

- This is the lane's cleanest "external grader" exemplar: the agent never scores itself; a local grader plus a frozen human leaderboard supply the truth. The contamination study is unusually rigorous and is the right template for the repo's worry that headline gains can be memorization, not capability.
- The rule "submissions must be produced by a model, not by the agent recalling labels" is the MLE analog of the repo's "the LLM proposes; the runtime owns truth": it structurally forbids the agent from short-circuiting the task with stored knowledge.
- The scaffold-dominates-model result (AIDE >> general scaffolds; ENPIRE's harness-matters finding) confirms that the agent-computer interface and search structure are first-order, which maps to the repo's action_surface/gates being where capability is shaped.

## Mechanically useful vs research contribution

- **Mechanically useful**: the held-out external grader + frozen human baseline as the only success signal; the validity-not-score validation server (an agent can check whether an output is well-formed without learning its score, exactly what the repo's schema/gate layer does); the two-pronged anti-cheat (automated log analysis + plagiarism/similarity detector); the familiarity + obfuscation protocol for testing whether a benchmark gain is contamination.
- **Not a contribution to copy**: building a benchmark or a grader is support infrastructure per the shared contract, not the research contribution. MLE-bench's own contribution is measuring MLE capability; the repo should borrow the grading/anti-contamination discipline, not reframe itself as "MLE-bench for Minecraft."
- **Honest bound**: MLE-bench works precisely because each task has a clear objective, clean data, and a crisp metric. The repo's Physical/Material layers have that (typed deltas, deterministic checks); the Social/Institutional layers do not, so the clean-grader guarantee does not transfer up the WAM stack.

## WAM layer(s) informed

Method-level, primarily Physical/Material (where a clean external grader exists). Directly informs how to score an autoresearch loop that improves an advisory WAM predictor: use a held-out deterministic grader, forbid the agent from supplying the answer from memory, and run a familiarity/obfuscation check before trusting a gain.
