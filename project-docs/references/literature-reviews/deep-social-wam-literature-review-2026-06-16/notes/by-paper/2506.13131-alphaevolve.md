## Primary-source facts (LaTeX-verified)

- **What it is**: an "LLM code superoptimization agent". An evolutionary loop where SOTA LLMs (ensemble of Gemini 2.0 Flash + 2.0 Pro) propose code edits to an initial program, the edits are scored by a user-supplied automated evaluator, and high-scoring programs re-enter a database that resurfaces them in later prompts. Direct successor to FunSearch (2406.08414's predecessor; see [[funsearch]]).
- **The evaluator is the ground truth**: the user must provide a function `h` (typically a Python `evaluate`) mapping a solution to a dict of scalar metrics, maximized by convention. "This evaluation mechanism allows AlphaEvolve to avoid any incorrect suggestions from the base LLM." The paper is explicit that the main limitation is that it "handles problems for which it is possible to devise an automated evaluator" (math, CS, systems), not natural-science domains needing physical experiment.
- **What is evolved**: whole code files (not single functions), in any language, via `# EVOLVE-BLOCK-START/END` markers; the rest of the file is a fixed skeleton calling the evolved blocks. The initial program may be rudimentary (single-line functions returning constants).
- **Diff-based edits**: the LLM emits `<<<<<<< SEARCH / ======= / >>>>>>> REPLACE` blocks for targeted edits (or full rewrites for short code).
- **Database**: MAP-elites (Mouret and Clune 2015) + island population models. Evaluation cascade (test cases of increasing difficulty; cheap stages first to prune faulty programs early). Optional LLM-generated feedback score for properties hard to encode in `h` (e.g. simplicity). Multi-objective optimization supported, and optimizing multiple metrics often helps the single target metric (diversity injection).
- **Flexibility of abstraction**: can evolve the solution directly, a constructor function, a bespoke search algorithm, or co-evolve solution+search. Symmetric problems favor constructor functions; non-symmetric favor evolved search algorithms.
- **FunSearch vs AlphaEvolve (paper Table 1)**: FunSearch evolves a single Python function up to 10-20 lines, needs <=20min eval on 1 CPU, uses millions of small-LLM samples, single metric, minimal context. AlphaEvolve evolves entire multi-file codebases in any language, can evaluate for hours in parallel on accelerators, thousands of frontier-LLM samples suffice, multi-metric, rich context.

### Results (paper-stated)

- **Matrix multiplication / tensor decomposition**: improved SOTA rank for 14 targets with m,n,p<=5. Flagship: a rank-48 algorithm for multiplying two 4x4 COMPLEX-valued matrices, the first improvement over Strassen's rank-49 (1969) for any field of characteristic 0 in 56 years. (Note: distinct from AlphaTensor's rank-47, which holds only over the 2-element field; see [[alphatensor-alphadev]].)
- **Open math problems**: applied to >50 problems across >5 branches; matched best-known constructions on ~75%, surpassed SOTA on ~20%. Named new results: improved Erdos minimum-overlap upper bound; kissing-number lower bound in 11 dimensions raised to 593 (from 592). Method: evolve heuristic SEARCH algorithms (given a time budget e.g. 1000s and the previous best construction), producing multi-stage adaptive search (early heuristics make big gains, later ones fine-tune). External mathematicians (Gomez-Serrano, Tao) suggested/formulated problems.
- **Google infrastructure**: (a) Borg datacenter scheduling heuristic recovers on average 0.7% of fleet-wide compute (stranded resources); chosen over deep RL for interpretability/debuggability/deployability; "correct by construction" because the heuristic only re-ranks machines Borg already deemed valid. (b) Gemini training kernel (Pallas/JAX tiling heuristic): 23% average kernel speedup, 1% total Gemini training-time reduction, optimization time cut from months to days; correctness preserved by construction (tiling does not change the math). (c) TPU Verilog RTL: a verified bit-removal simplification (also caught by synthesis tools). (d) XLA-compiled FlashAttention IR: 32% kernel speedup + 15% pre/post-processing speedup, each modification checked against the reference on random inputs and confirmed correct by experts.

## Interpretation (flagged as inference)

- AlphaEvolve is the strongest existence proof that LLM-guided evolution graded by an EXACT, CHEAP evaluator discovers genuinely new artifacts (a 56-year-old matmul record; provably better math constructions; deployed infra heuristics). Every headline result rests on a machine-checkable score, which is the lane's central claim and its bound.
- The recurring "correct by construction" framing (heuristic only re-ranks already-valid options; tiling/IR rewrites checked numerically) is the same authority boundary this repo enforces: the proposer changes only what cannot break truth, and an external check owns correctness. This is the cleanest industrial precedent for an advisory-only WAM rule generator gated by a verifier.
- The discussion's "distill AlphaEvolve-augmented performance back into the base model" and "AlphaEvolve improves its own infrastructure/base LLM" is a measured, months-long self-improvement loop, not recursive self-improvement; the authors call the gains "moderate" and the loop slow. This is the sober end of the wave-5 self-improvement spectrum.

## Mechanically useful vs research contribution

- **Mechanically useful**: the loop skeleton (initial program with marked evolve-blocks + `evaluate` returning a scalar dict + MAP-elites/island database + evaluation cascade + diff-format edits + multi-metric diversity); the "evolve a search algorithm given a time budget and the previous best" pattern for problems with cheap evaluation; the "correct by construction" discipline (let the generated code only re-rank/retune within a space the verifier already constrains).
- **Not a contribution to copy**: AlphaEvolve's research claims are the matmul/math/infra results. The repo must not reframe itself as "AlphaEvolve for Minecraft". The defensible borrow is the loop + the verifier-as-ground-truth separation, applied to proposing advisory-WAM consequence rules or gated `author_mineflayer_action` skills scored by the runtime verifier's agreement on already-emitted (state, action, next-state) transitions.
- **Where it bounds the thesis**: AlphaEvolve only works where an automated evaluator exists; the authors say so plainly. This is exactly the Physical/Material layer in this repo (clean verifier) and exactly NOT the contested Social/Institutional layer.
