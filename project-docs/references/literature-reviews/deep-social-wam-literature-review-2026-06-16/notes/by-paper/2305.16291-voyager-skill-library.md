## Primary-source facts (skill-library mechanism only)

- **Skill = executable code**. Each skill is GPT-4-generated code that scaffolds temporally-extended Mineflayer actions for a curriculum-proposed task. Generation prompt includes: code-gen guidelines ("make it generic and reusable, it will be reused for more complex functions"), control-primitive APIs + retrieved relevant skills, last round's code + environment feedback + execution errors + critique, the agent's current typed state, and CoT.
- **Iterative prompting / self-improvement via three feedback types**:
    1. Environment feedback (e.g. "I cannot make an iron chestplate because I need 7 more iron ingots"), via `bot.chat()`.
    2. Execution errors from the program interpreter (bug fixing).
    3. **Self-verification for task success**: a SEPARATE GPT-4 instance acts as critic, told the current state + task, and reports whether the program achieved the task; if failed, it gives a critique. The paper claims this is "more comprehensive than self-reflection [Reflexion]."
- **Library management**: a validated skill is added to the library and INDEXED BY THE EMBEDDING of its description; retrieval queries the library with the embedding of the self-generated task plan + environment feedback. If stuck after 4 code-gen rounds, the curriculum proposes a different task.
- Result (paper-stated): Voyager obtains more unique items, travels farther, and unlocks tech-tree milestones faster than prior LLM-agent baselines; skills transfer to a new world.

## Interpretation (inference, labeled)

- For this lane, Voyager is the SKILL-code instance of "LLM authors the improvement code." Where Eureka authors reward code, Voyager authors SKILL code, refined by an iterative-prompting loop. The repo's `author_mineflayer_action` path is the direct descendant of Voyager's skill-code generation + trial.
- The decisive difference, and the lane's thesis bound made concrete: Voyager's success signal is an LLM SELF-VERIFYING (a GPT-4 critic judging whether the task was achieved). That is exactly the "agent scores its own success" pattern the repo names as progress laundering. Voyager's skill library accumulates skills validated by an LLM's judgment, not by a deterministic verifier; a wrong self-verification silently adds a bad skill or marks a non-achievement as done.
- The repo's design replaces Voyager's LLM-critic with runtime gates + a deterministic verifier on the SAME skill-code-authoring mechanism. So the repo can adopt Voyager's library/retrieval/iterative-refine machinery while removing its weakest link (the self-judge).

## Mechanically useful vs research contribution

- **Mechanically useful**:
    - **Skill-as-code + embedding-indexed library + retrieval-by-plan-embedding**: a ready pattern for a repo skill store of validated `author_mineflayer_action` candidates, retrieved by task/plan similarity.
    - **Iterative refine from environment feedback + execution errors**: the repo already emits these (verifier output, tracebacks); reuse them to refine a proposed skill candidate, exactly as Voyager does, but gate promotion on the verifier.
    - **Reusability guideline in the prompt** ("make it generic and reusable") is a cheap, effective instruction for any skill-authoring prompt.
- **What to avoid / overclaim (the central one for this lane)**:
    - **Do NOT adopt Voyager's LLM self-verification as the success/promotion signal.** It is the canonical progress-laundering risk. Skill promotion into the repo's library must be gated by the deterministic verifier and runtime trial, with any LLM critique advisory only.
    - Voyager is single-agent, item/tech-tree-driven; its success metric is competence (unique items, milestones), not social contribution. Do not frame its skill library as social.
    - The skill-library engineering is support; the repo must not reframe its work as "Voyager with a verifier" as a research claim.
