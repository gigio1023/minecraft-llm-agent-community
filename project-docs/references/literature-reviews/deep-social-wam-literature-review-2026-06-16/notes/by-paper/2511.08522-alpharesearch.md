## Primary-source facts (LaTeX-verified)

- **What it is**: an autonomous research agent for OPEN-ENDED algorithmic problems. Loop: (1) propose a new idea, (2) program to verify, (3) optimize. Its novelty is a DUAL environment combining two rewards: an execution-based verifiable reward AND a reward from a simulated peer-review environment.
- **The two rewards (the key design)**:
    - Program-based execution reward: a code executor `E` with a Verification module (does the program satisfy the problem constraints?) and a Measurement module (what score?). This is the exact, cheap oracle of the FunSearch/AlphaEvolve family.
    - Peer-review reward model (RM): a Qwen2.5-7B-Instruct fine-tuned on 24,445 ICLR peer-review records (2017-2024 train, 2025 test, dates disjoint to avoid contamination); input = paper abstract, target = average overall rating. Binarized at the ICLR accept/reject midpoint (rating > 5.5 = positive). The RM gates IDEAS: if `RM(idea) < threshold`, the round is skipped before any coding.
- **RM evaluation (held-out ICLR 2025 ideas)**: AlphaResearch-RM-7B reaches 72.0% binary accuracy, vs human annotators 65.0%, GPT-5 (medium) 53.0%, Qwen2.5-7B-Instruct 37.0%, random 50.0%.
- **Results**: stronger discovery than other agentic discovery systems on 6 of 8 open-ended problems in their AlphaResearchComp benchmark. On the "packing circles" problem, the discovered algorithm achieves best-of-known performance, surpassing human researchers and AlphaEvolve.

## Interpretation (flagged as inference)

- AlphaResearch is the most direct attempt in the lane to handle a problem where the evaluator is NOT a single exact oracle. Its answer is instructive and exactly the pattern this repo should adopt for the social layer: keep an EXACT executor as the score, and add a LEARNED model only as an advisory FILTER ON PROPOSALS (idea feasibility), never as the success metric. The learned RM decides what to TRY; the executor decides what SUCCEEDED.
- This is the proposer/scorer separation generalized: even when you want a soft, judgment-laden signal (is this idea any good?), you confine it to gating proposals, and you keep a non-learned check as the authority. For the repo: a learned social-plausibility model could prioritize which advisory-WAM rules to propose, but the runtime verifier on held-out transitions must remain the only thing that promotes a rule.
- The RM beating human annotators (72% vs 65%) is a measured, honest result, but note its scope: it predicts ICLR accept/reject from abstracts, a much narrower and more data-rich task than "is this a good social outcome in Minecraft". Do not over-read it as evidence that a learned model can judge open social quality.

## Mechanically useful vs research contribution

- **Mechanically useful**: the dual-reward architecture (exact executor owns the score; a learned model only filters proposals) is a concrete blueprint for using a learned social-plausibility signal WITHOUT letting it own success; the contamination-safe RM training split (disjoint dates) as a discipline for any learned scorer; the verification+measurement module split mirrors the repo's schema/permission gate (verification) plus verifier (measurement).
- **Not a contribution to copy**: the circle-packing result and the benchmark are the contributions; the repo borrows the dual-reward separation.
- **Bound**: the executor is still exact (a geometry check); the RM only filters ideas. AlphaResearch does NOT show a learned model can SCORE an open-ended outcome, and the social layer's missing piece (an exact outcome oracle) remains missing.

## WAM layer(s) informed

Method-level; bridges Physical/Material to Social by showing how to add a learned advisory filter (proposal gating) while keeping an exact verifier as the score. The pattern the repo should use if it ever introduces a learned social-plausibility model.
