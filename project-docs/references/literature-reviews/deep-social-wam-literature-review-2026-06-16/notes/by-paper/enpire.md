## Primary-source facts (PDF-verified)

- **What "autoresearch" means here**: the paper adopts the term from Karpathy (ref 22, "AI agents running research on single-GPU nanochat training automatically"). Recent autoresearch is DIGITAL (a coding agent automates algorithmic improvement). ENPIRE's contribution is extending autoresearch to the PHYSICAL world (real-robot policy improvement).
- **ENPIRE = four modules**: Environment (EN: automatic reset + automated verification interfaces), Policy Improvement (PI: the coding agent edits training code and picks the method), Rollout (R: single or parallel real robots), Evolution (E: agents read logs, do literature review, improve infra and algorithm code). Two phases: phase 1 EN constructs the environment+verifier from a one-time human-guided setup; phase 2 PIRE is fully autonomous improvement from real-world verification.
- **The coding agent chooses the improvement regime**: heuristic learning, tool calling, behavior cloning, offline RL, online RL, code-as-policy, or hybrids, and tunes hyperparameters (batch size, BC-term weight, actor update rate). RL is ONE selectable regime, not the framing. The outer loop is an LLM coding agent searching over training recipes; an inner loop may be RL.
- **Three coding agents benchmarked**: Codex (GPT-5.5 xhigh), Claude Code (Opus 4.7 High), Kimi Code (Kimi K2.6 thinking). Codex solved fastest; the agent HARNESS matters as much as the model (a Codex harness driving the Opus 4.7 API was the least efficient configuration; native-vision Codex reached success first).
- **The autoresearch prompt (Fig 10, verbatim shape)**: "/goal Achieve a 100% success rate over each window of 50 consecutive episodes. Read @auto_research.md and fan out a subagent team to study different algorithms (RL, BC, hybrid). You may reuse checkpoints or data buffers across hypotheses. Create your own branch from autorl ... Do not directly work on or push to autorl. You must actively monitor other branches and commits ... push your branch ... pull from origin. Collaborate with other agents via Git." This is a parallel-subagent design coordinated entirely through Git.
- **Fleet**: 8 hardware-identical bimanual YAM robot stations, decentralized; coordination is entirely via Git (branch, cherry-pick, merge), not shared state; each station owns its arms, cameras, compute (1x RTX 5090 32GB), and its own coding agent; FastAPI control endpoints (`/start`, `/restart`, `/home`, plus `/avoid` and `/resume` for occlusion). Agent sandbox bypasses per-action permission prompts and has unrestricted internet; transfer between tasks is via a distilled markdown summary, not raw prior state.
- **Automated verification (the success signal)**: the coding agent synthesizes a binary reward from a few minutes of labeled success/failure demos (video + proprioception), inference < 150ms; the agent "may study failures but cannot train on the test set or alter metric computation" (held-out, sandboxed). Examples: zip-tie reward = two-camera crop + SAM3 segmentation AND-fused; pin-insertion reward = visual alignment + insertion depth (proprioception) + contact force. Hard safety constraints trigger auto-reset; torque-limiting compliant grasp makes unattended operation safe.
- **Automated reset**: procedural tool calls inspired by CaP-X (ref 15) compose SAM3 segmentation, pose tracking, and cuRobo collision-free planning to restore the scene to the onset of the hardest phase.
- **Results (paper-stated)**: ~99% success on PushT, pin insertion (4mm clearance; target 100% over 50 consecutive episodes; converges faster than a frontier human-in-the-loop method, ref 48), GPU insertion, zip-tie cutting. Pin-insertion hill-climb milestones: BC regularization +10.8pp, tweak BC-term weight +8.4pp, tune batch size 1024->512, controller compensation +1.3pp, over ~3h research wall-clock.
- **Simulation**: on RoboCasa365 (ref 34), ENPIRE outperforms an end-to-end VLA (GR00T N1.5, ref 5) and zero-shot agentic tool use without autoresearch (CaP-X, ref 15). It discovers a hover-before-grasp strategy and transfers it to the real zip-tie task.
- **Scaling metrics**: Mean Robot Utilization (MRU) and Mean Token Utilization (MTU). Scaling 1 to 8 agents reaches success sooner (pin insertion: 1.5h to ~40min) but token cost grows super-linearly (Push-T total tokens 80M at 1 agent to 209M at 4 agents); MRU falls with fleet size (more time merging peer Git branches), GPU utilization rises.

## Interpretation (flagged as inference)

- ENPIRE is the PHYSICAL instance of a DIGITAL pattern. The same loop (a coding agent hill-climbs a target against a verified signal) is what Karpathy's autoresearch does for nanochat training and what AI-Scientist systems do for ML papers. The "self-improvement" is the agent rewriting training code, choosing the method, and tuning hyperparameters, with the environment, not the agent, scoring success.
- On the user's "is it RL?" question: RL is an inner-loop regime the agent may select and tune; the outer loop is LLM-agent search over training recipes coordinated by Git. So ENPIRE is "an agentic autoresearch loop that can invoke RL," not "an RL method."
- The verified-signal separation is the same principle this repo enforces (the runtime owns truth; the LLM proposes). ENPIRE's explicit guard, "the agent may study failures but cannot train on the test set or alter metric computation," is exactly the repo's anti-progress-laundering rule, stated for a reward function.
- The Git-coordinated subagent team is itself a parallel-subagent autoresearch design, the same shape used to run this literature review.

## Mechanically useful vs research contribution

- **Mechanically useful**: the two-phase split (one-time human-guided environment+verifier construction, then fully autonomous improvement); decentralized Git-as-coordination for a fleet; the idea-tree + hill-climb logging (Fig 12) as the record of what was tried; the held-out-verifier guard (study failures, never train on or alter the metric); the autoresearch prompt structure (goal, read a method doc, fan out subagents, branch, merge peers).
- **Not a contribution to copy**: the research claim is real-robot dexterous-manipulation autonomy. Do not reframe the repo as "ENPIRE for Minecraft." The defensible borrow is the loop structure, the verifier-as-ground-truth separation, and the two-phase setup, applied to autonomously improving the advisory social-material WAM or actor policy.
- **Where the analogy breaks (honest bound)**: ENPIRE has a crisp, resettable, near-binary success metric and a one-time-constructable verifier. An open-ended social Minecraft world has no single 99% metric, social scenarios are costly to reset cleanly, and upper-layer success is contested. The loop transfers cleanly at the Physical and Material layers, where the verifier gives clean labels.

## WAM layer(s) informed

Method-level, cross-layer. Directly informs how to autonomously improve a Physical/Material WAM predictor or policy against verifier evidence. Social/Institutional improvement is meaningful only after physical prediction is reliable.
