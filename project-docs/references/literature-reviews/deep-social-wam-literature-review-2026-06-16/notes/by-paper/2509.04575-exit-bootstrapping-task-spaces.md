# ExIt - Bootstrapping Task Spaces for Self-Improvement (Exploratory Iteration)

- **title**: Bootstrapping Task Spaces for Self-Improvement (method name: Exploratory Iteration, ExIt)
- **authors**: Minqi Jiang, Andrei Lupu, Yoram Bachrach (Meta / FAIR-lineage authors).
- **year**: 2025 (published 2025-09-04).
- **venue/source**: arXiv preprint; LaTeX uses the TMLR template.
- **arxiv_id**: 2509.04575
- **urls**: paper https://arxiv.org/abs/2509.04575
- **source availability**: latex (tarball, 13 .tex files; main is `tmlr.tex`). Deep-read.

## Primary-source facts (what the paper states/shows)

- **Research question**: how to train an LLM to reliably *self-improve over many inference-time steps* without committing to a fixed, costly maximum iteration depth K up front?
- **Core idea**: ExIt is a family of **autocurriculum RL** methods. It trains for K-step self-improvement using **only single-step self-improvement transitions**. It grows a task space by treating each intermediate, partial solution history encountered during an episode as a *new* self-iteration task instance, sampled into a buffer by a learnability score.
- **Model architecture**: standard LLM (DeepSeek-R1-Distill-Qwen-7B and others) fine-tuned with GRPO. No new architecture; the contribution is the training procedure / task-space construction.
- **Action / task structure**: a "turn" is a POMDP timestep; a "step" is one self-iteration (the LLM rewrites a previous response). ExIt seeds new single-step tasks from prior step endpoints.
- **Reward (verifier-grounded, central to this lane)**: per domain, an explicit verifier defines the reward.
    - Competition math: `math-verify` exact-answer check, reward 1/0.
    - Multi-turn tool use (BFCLv3): a trajectory is correct iff its tool-call sequence reaches the ground-truth final program state AND contains the ground-truth call sequence as a subset; this state-equivalence check is the verified reward.
    - ML engineering (MLE-bench, real Kaggle tasks): execute the generated code in a sandbox, run `mlebench grade`, normalize the metric (e.g., AUC) to [0,1].
    - Self-improvement step reward: `max(0, Delta/Delta_max)` where Delta is the quality gain over the previous iterate (so improving an incorrect solution is rewarded; regressing is penalized).
- **Learnability / prioritization**: tasks are prioritized by **group return variance** (extends the binary-outcome "learnability" measure to scalar outcomes). High variance = the policy sometimes succeeds and sometimes fails = high learning potential. Because variance is just a function of GRPO's group rollouts, the autocurriculum is "compute-equivalent."
- **Exploration**: two mechanisms - epsilon-greedy *self-divergence* (with prob epsilon, switch the task from "improve" to "find a valid but substantially different approach") and a centroid-distance novelty bonus in embedding space.
- **Evaluation protocol + metrics**: held-out math sets (MATH500, AMC, AIME, Minerva, OlympiadBench via math-verify), BFCLv3 tool-use split, MLE-bench. Result: ExIt strategies improve K-step inference-time self-improvement on held-out instances, with K extending *beyond* the average depth seen in training, and also raise initial (step-0) solution quality. Emergent autocurriculum: sampled task complexity (e.g., mean solution steps, starting turn depth) rises over training.
- **Code/data/weights**: not released in the source examined (built on a "heavily-modified fork of open-instruct"). Reproducibility: claim-only-to-partial (method is detailed; no released checkpoints/code in source).

## Interpretation (flagged as inference)

- ExIt is the formal bridge between *self-improvement loops* and *autocurriculum / open-endedness* (it explicitly relates itself to Prioritized Level Replay and evolutionary task mutation). For my lane the key transfer is the **learnability-driven scenario sampling**: do not waste loop budget on tasks the policy already aces or always fails; sample where outcome variance is highest.
- It also names a documented caution: "RL fine-tuning based on verified rewards can reduce output diversity, making it harder to find effective improvements" - i.e. even an honest verifier can collapse diversity, which exploration must counter.

## Mechanically useful vs research contribution

- **Mechanically useful for this repo**: (1) the **learnability = outcome-variance** heuristic is a cheap, model-agnostic way to *choose which Minecraft scenarios to re-run* in an autoresearch loop - prioritize scenarios where the actor/WAM is neither reliably right nor reliably wrong against the verifier. (2) "Upcycle the agent's own intermediate states into new task instances" is a way to generate scenario variety without an external generator - relevant because clean social-scenario reset is the repo's missing piece (the partial states are free). (3) self-divergence (force a substantially different valid approach) is a usable anti-stagnation move.
- **Research contribution NOT to copy**: ExIt's claim is an autocurriculum RL *training* method for inference-time self-improvement. The repo does not train weights. Borrow the *task-space and prioritization logic*, applied to scenario selection, not the GRPO update.
- **Where it supports the thesis**: it gives a principled, near-zero-extra-cost (compute-equivalent) way to keep a loop from stagnating, and it is explicitly grounded in verified rewards. **Where it bounds it**: every reward here is a clean per-domain verifier (math-verify, state-equivalence, mlebench grade). The variance-learnability signal needs a *reliable* scalar success measure to be meaningful; the repo has that at Physical/Material, not at the contested Social/Institutional layers.

## WAM layer(s) informed

Method-level, cross-layer. Most useful for the *control logic* of an autoresearch loop (which scenarios to sample, how to avoid both trivial and impossible tasks). Its reliability inherits the verifier's reliability, so it lands cleanly only where the verifier gives clean scalar labels.
