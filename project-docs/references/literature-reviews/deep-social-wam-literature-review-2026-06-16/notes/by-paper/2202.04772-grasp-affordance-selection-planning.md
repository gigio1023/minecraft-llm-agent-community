## Research question
Planning with a learned model in CONTINUOUS action/option spaces is hard for tree search: you cannot enumerate every action even at the root node. Can an agent LEARN a small set of affordances (state-and-goal-conditioned mappings to a few useful actions/options) so the planner only branches over those few, and can those affordances be discovered by backpropagating gradients THROUGH the planning procedure?

## Primary-source facts (what the paper states/shows)

Definition of affordance used (the part this lane needs):
- The paper grounds the term in Gibson's ecological psychology: affordances are "directly perceived high-level actions that environment states afford, such as object interactions (e.g., chairs afford sitting, cups afford grasping) or navigation actions (doorways afford passage), all restricted subsets of the vast space of possible actions the organism may perform" (intro, citing gibson1977concept).
- Their operational definition: an affordance is a PARAMETRIC MAPPING from states (and goals, in multi-task settings) to actions/options. Selecting K affordances = learning K such mappings; then there are AT MOST K choices at each node of the planning tree. The action the i-th affordance yields depends on the current node's state.
- They also study "unconditional affordances": just K actions/options available in all states, to measure how much state-conditioning matters.

Method (GrASP):
- Affordances are functions f^afford parameterized by theta^afford. GrASP computes gradients THROUGH the planner (a value-equivalent learned model, MuZero-style) to update the affordance parameters, jointly with learning the value-equivalent model that predicts the value-consequences of those affordances.
- Two coupled learning problems run simultaneously and nonstationarily: (1) learning to SELECT affordances, (2) learning the value-equivalent model used to plan with them.

Evaluation + claims:
- Empirically shows it is feasible to learn to select BOTH primitive-action affordances and temporally-extended OPTION affordances.
- Headline claim (abstract + intro): simultaneously learning to select affordances and planning with a learned value-equivalent model "can outperform model-free RL" and is "often more sample efficient than model-free approaches in benchmark RL control problems."
- Explicit modesty: "We do not claim or show that GrASP is state of the art for model-based RL in continuous domains ... A comparison with state of the art awaits further development of this new idea." (intro.)

Related-work genealogy of affordances-in-RL (useful map, all from related.tex):
- Abel et al. 2014: affordances = a subset of actions available in a state when the state meets preconditions defined by propositional functions; learned in a scaffolded multi-task curriculum and shown to reduce planning time. Abel et al. 2015 extends to goal-based priors (distributions over each action's optimality for a state-goal pair). Both rely on Object-Oriented MDPs (objects/classes/attributes in the state).
- Khetarpal et al. 2020 ("What can I do here? A theory of affordances in RL"): defines affordances for MDPs; hand-specified affordances give faster planning by shrinking the action set at a state AND enable efficient learning of PARTIAL transition models.
- Action-pruning / action-prior lineage: Sherstov & Stone 2005 (transfer a reduced action set), Rosman & Ramamoorthy 2012 (Dirichlet action priors from related tasks speed planning), and others (Even-Dar 2006, Cruz 2014/2016/2018, Zahavy 2018 "learn what not to do") all show pruning the action space helps scale to large action spaces.

## Interpretation (my reading, labeled inference)
- GrASP is the cleanest statement, in an RL-planning frame, of the exact phrase this lane owns: "future action opportunities." An affordance IS the formal object "which small set of actions is worth considering in this state." That is what the query means by an action making opportunities newly available or blocked.
- The Abel/Khetarpal "precondition -> available action subset" definition is the one that maps best to this repo, because the repo's runtime ALREADY computes typed readiness/eligibility for Action Cards (per SPEC: "typed readiness/eligibility contracts" decide whether a tool is visible). That is a hand-authored, rule-based affordance function: state preconditions -> visible action set. So the repo already HAS affordances in the precondition sense; the open question is whether a WAM should PREDICT how a candidate action changes the future affordance set.
- GrASP's specific contribution (gradient-through-planner discovery of K continuous affordances) is largely NOT applicable to the repo: the repo's action set is a discrete, schema-bound tool menu chosen by an LLM, not a continuous space needing tree-search pruning. The useful import is the CONCEPT (affordance = state-conditional action availability) and the genealogy, not the gradient method.

## Mechanically useful vs research contribution (for this repo)
- Mechanically useful (engineering to borrow):
  - The precondition-based affordance framing (Abel 2014, Khetarpal 2020) as the vocabulary for the repo's existing typed readiness/eligibility gates: "state meets preconditions -> this Action Card is afforded." This aligns affordance theory with the repo's "the runtime owns eligibility" rule WITHOUT adding any planner.
  - The idea that affordances let you learn PARTIAL transition models (Khetarpal 2020): a WAM need only predict consequences for the small AFFORDED action set in a state, not the whole tool menu, which keeps an advisory predictor cheap.
- Research contribution (novel if done here):
  - Predicting the DELTA in the affordance set as a first-class WAM output: "if I place a crafting table, crafting-X becomes afforded; if I lend my only pickaxe, mining becomes UN-afforded for me and afforded for the borrower." No surveyed affordance work predicts affordance-set transitions across a SOCIAL/material state; that is the repo's own contribution surface.
  - A "social affordance" notion (a returned tool restores the borrowing relation; an accepted promise affords a future request) is not in this literature and would be a research claim, not a borrow.
- Avoid / overclaim guard:
  - Do NOT adopt GrASP's gradient-through-planner mechanism; it solves a continuous-action tree-search problem the repo does not have. Borrowing it would add a planner the repo's design explicitly forbids ("the runtime must not become a hidden Minecraft planner").
  - Affordance prediction must stay ADVISORY: predicting "crafting becomes afforded" must not auto-unlock the tool. The runtime's typed eligibility remains the authority; the WAM only forecasts what the eligibility WILL be.

## WAM layer(s) informed
- Physical WAM (primary): a placed station / a mined block / a repaired tool changes which physical actions are afforded next. Affordance-set deltas are the literal "newly available/blocked affordances" in the Physical layer definition.
- Material + Social WAM (as research extension): possession/claim changes alter who can use a station (material affordance), and social acts alter who may request/borrow next (social affordance). Transporting affordance theory there is unproven and is the repo's contribution, not a citation.
- Institutional: stable affordance patterns across a settlement (who routinely may use the shared furnace) connect to the institutional layer, but the surveyed works stop at single-agent planning.
