## Primary-source facts

- **Research question**: when a coding agent repeatedly extends its OWN prior solution (iterative, long-horizon), does code quality degrade, and how much?
- **Benchmark shape**: 36 problems, 196 checkpoints; agents repeatedly extend their own solutions under evolving specifications that demand architectural decisions but leave internal structure to the agent (unlike prior iterative benchmarks that over-constrain the design space).
- **Two degradation measures**: structural erosion (concentrated complexity) and verbosity (redundant code).
- **Results (15 agents, open + closed)**: no agent fully solves any problem end-to-end; best agent passes only 14.8% of checkpoints. Quality degrades across checkpoints: structural erosion rises in 77% of trajectories, verbosity in 75.5%. Compared to 473 open-source Python repositories, agent code is 2.3x more verbose and 2.0x more eroded, and human repositories degrade LESS often and by smaller margins across their git histories. Explicit quality guidance reduces INITIAL verbosity/erosion by up to a third but does NOT change the degradation RATES.
- **Conclusion**: first measurement of code degradation under iterative extension; agents "pass checkpoints while producing code that erodes and bloats with each turn."

## Interpretation (flagged as inference)

- This is the lane's strongest evidence for a silent failure mode of self-iteration: the agent keeps passing the checkpoint (the verifier says OK) while the artifact quietly rots. Passing the gate at each step does not prevent cumulative degradation, because the gate scores task success, not artifact health.
- It generalizes AIDE's mild "code complexity grows over steps" observation into a measured, cross-agent law (erosion in 77%, verbosity in 75.5% of trajectories; 2x worse than humans).
- For the repo: an autoresearch loop that iteratively edits advisory-WAM code or skill candidates will, by default, bloat and erode them while still passing the verifier. The loop needs an explicit quality/regression signal (not just task-pass), and the finding that quality guidance changes the starting point but not the RATE means a one-time prompt is insufficient; the loop needs per-step structural checks.

## Mechanically useful vs research contribution

- **Mechanically useful**: the two degradation metrics (structural erosion, verbosity) and the human-repo baseline as a template for an artifact-health check the repo's loop should run alongside the verifier; the finding that pass-the-checkpoint does not imply healthy code (so a WAM/skill loop needs a separate regression/complexity gate); the measured insufficiency of one-shot quality guidance.
- **Not a contribution to copy**: the benchmark is support; its contribution is measuring degradation. The repo borrows the "watch artifact health across iterations, not just task-pass" discipline.
- **Honest bound**: SlopCodeBench measures CODE artifacts. The repo's loop also edits prompts/principles/WAM parameters where "erosion" is less defined; the principle (cumulative degradation despite passing) likely carries, but the specific metrics may not.
