# WebEvolver: Enhancing Web Agent Self-Improvement with Co-evolving World Model

- **title**: WebEvolver: Enhancing Web Agent Self-Improvement with Co-evolving World Model
- **authors**: Tianqing Fang, Hongming Zhang, Zhisong Zhang, Kaixin Ma, Wenhao Yu, Haitao Mi, Dong Yu (Tencent AI Lab)
- **year**: 2025
- **venue/source**: arXiv preprint (COLM-style template); code released
- **arxiv_id**: 2504.21024
- **urls**: paper https://arxiv.org/abs/2504.21024 ; code https://github.com/Tencent/SelfEvolvingAgent
- **source availability**: latex (tarball extracted, `papers/latex/2504.21024/acl_latex.tex`)

## Primary-source facts (what the paper states)

- Problem: LLM-agent **self-improvement** (agent trains its own backbone LLM on self-sampled, rejection-filtered trajectories) plateaus in web environments. The paper attributes the plateau to (1) shrinking exploration diversity as the policy overfits familiar trajectories and (2) inference-time search (tree search, MCTS) being too costly in real web interactions.
- Core idea: add a **co-evolving World Model LLM** that predicts the next observation (a web page, as an accessibility tree) given the current observation and an attempted action: a learned text transition model `o_{t} ~ p_theta_w(. | I_w, c^w_t)`.
- The world model serves **two roles**:
    1. **Virtual web server** for offline trajectory synthesis. Replace the real transition function with the WM, alternate WM-predicts-observation and policy-predicts-action to roll out synthetic multi-step trajectories (depth up to 7), rejection-sample, and add the survivors to the policy's SFT data. Explicitly: the WM may produce hallucinated (non-realistic) pages, and the paper argues this is "not critical during training, as the agent's goal is to learn flexible action prediction" under noisy conditions.
    2. **Imagination / look-ahead engine** at inference (named WMLA, World Model Look-Ahead). For each candidate action the WM rolls out d-step look-ahead; an LLM evaluator (GPT-4o) scores each rollout in {0, 0.5, 1.0}; the policy executes the highest-scoring action. This is the advisory predict-then-pick mode.
- Architecture: both policy and WM are `Llama-3.3-70B` fine-tuned. Both trained by supervised next-token objectives (Eq. for policy J(theta) and for WM J(theta_w)), iteratively, on the same self-collected trajectory pool. The WM is co-trained alongside the policy each iteration (WM iter-1, iter-2).
- Key insight (stated): pretrained LLMs already encode "website dynamics, user intents, task workflows," so an LLM is a good prior for a learned web transition model.
- Evaluation protocol: live web benchmarks WebVoyager (473 queries), Mind2Web-Live (53), plus GAIA-web and SimpleQA. Success scored end-to-end by GPT-4o (binary 0/1). Runs repeated twice in the same time window, averaged.
- Results (page-stated): self-improvement gives +6% on WebVoyager at iter-1, then plateaus at iter-2; adding WM-synthesized data to iter-1 SFT data gives a further +4% (WebEvolver total ~10% over the OpenWebVoyager self-evolving baseline). WMLA look-ahead (depth 1-2) adds further gains (WebVoyager all up to ~51%, Mind2Web-Live up to ~28%). An intrinsic WM-quality evaluation (Structural correctness / Similarity / Overall) shows iter-2 WM best.
- No distillation from stronger closed models is used; the loop is self-contained on Llama-3.3-70B.

## Interpretation (flagged as inference)

- WebEvolver is the cleanest published precedent for the wave-4 thesis's advisory role: an LLM world model `p(o'|o,a)` that (a) self-improves from the agent's own collected transitions and (b) doubles as a look-ahead engine that scores candidate actions without committing them. That is exactly "advisory WAM": it predicts and evaluates consequences; it never acts directly, and the real web transition still owns ground truth at execution.
- The two roles map onto two distinct uses this repo could borrow: WM-as-data-generator (cheap synthetic transitions to refine a predictor or policy) and WM-as-look-ahead (advisory scoring of Action Card candidates before the runtime executes one).
- Honest caution the paper itself raises: the WM hallucinates, and they treat that as acceptable BECAUSE the downstream signal (rejection sampling against real success, and look-ahead scoring by a separate evaluator) re-grounds training. The hallucination is tolerable only because a non-self WM-judged signal filters it. If you removed the real-success filter and trained the policy purely on WM-imagined trajectories scored by the same model, you would be laundering.

## Mechanically useful vs research contribution

- **Mechanically useful for this repo**: the co-evolving loop (collect verifier-labeled transitions, fit a predictor on them, use the predictor for synthetic rollouts AND advisory look-ahead, repeat) is directly transposable to the Minecraft cycle, where the runtime verifier already auto-labels `(state, action, next-state)` at near-$0. WMLA's "score each candidate rollout, pick the best, runtime executes" is a concrete advisory-WAM inference pattern that respects "the LLM proposes, the runtime owns truth."
- **What to avoid / overclaim**: WebEvolver's measured contribution is a +10% web-navigation success bump from one model family on browser benchmarks; do not reframe the repo's work as "WebEvolver for Minecraft." The borrow is the loop and the dual-role world model, not the numbers. Also avoid copying the "hallucination is fine" stance into the SOCIAL layers without the re-grounding filter: WebEvolver tolerates hallucination because GPT-4o re-scores against task completion. The repo's analog of that re-grounding is the runtime verifier; the WAM's predicted social deltas must be scored against verifier evidence, never against the WAM's own confidence.

## Limitations (theirs + spotted)

- Theirs: no standalone planning module (orthogonal, by their framing); live websites drift over time so exact replication is hard; WM intrinsic quality (esp. for real-time content like BBC/Hugging Face pages) is bounded and hallucination is not directly measured.
- Spotted: the WM is a free-text page generator, not a structured-state predictor; its "correctness" is judged by surface similarity, which is exactly the surface-form-fidelity trap that RWML (2602.05842) warns leads to model collapse. The repo's structured-state setting (typed Minecraft+social deltas) sidesteps this but loses the pretrained-web-knowledge prior WebEvolver leans on.

## Reproducibility status
- partial: code released, model is open (Llama-3.3-70B), but the live-web benchmarks are non-stationary (the authors note exact replay is impossible), so numeric reproduction is approximate.
