## Primary-source facts

- **Research question**: how to get (1) scalable, real-world interactive SWE training data and (2) a contamination-free benchmark, given that static SWE benchmarks become outdated and contaminated as models improve.
- **Method**: an automated, scalable pipeline that continuously extracts real-world interactive SWE tasks from diverse GitHub repositories (interactive = the agent must use a dev environment, execute code, and adapt to outcomes, not one-shot generation).
- **Dataset**: SWE-rebench, a public dataset of over 21,000 interactive Python-based SWE tasks, suitable for RL of SWE agents at scale.
- **Decontaminated benchmark**: a continuous supply of FRESH tasks builds a contamination-free benchmark for agentic SWE.
- **Key finding**: comparing LLMs on this fresh benchmark vs SWE-bench Verified shows that "performance of some language models might be inflated due to contamination issues."

## Interpretation (flagged as inference)

- SWE-rebench is the operationalization of SWE-bench's own contamination defense (continually-updatable, post-cutoff tasks), turned into a standing pipeline. It is the lane's positive answer to "how do you keep a benchmark honest as models improve": keep the held-out set fresh.
- Together with 2503.15223 (verifier too weak) it brackets the two ways a headline coding-agent number inflates: (a) the test/verifier under-specifies correctness (2503.15223), and (b) the test set leaked into training (SWE-rebench). The repo cares about both: a Minecraft verifier must be strong AND its held-out scenarios must be fresh enough that the agent has not effectively memorized them.

## Mechanically useful vs research contribution

- **Mechanically useful**: the continuous fresh-task pipeline as the template for keeping a held-out evaluation set uncontaminated over time (the repo can regenerate/perturb social scenarios so a loop cannot overfit to a fixed held-out set); the explicit fresh-vs-static comparison as a contamination diagnostic (run the agent on fresh held-out scenarios and on familiar ones; a gap signals contamination, the same logic as MLE-bench's obfuscation test).
- **Not a contribution to copy**: the dataset/pipeline is support; its contribution is decontaminated SWE evaluation. The repo borrows the freshness discipline.
- **Honest bound**: SWE-rebench freshness is cheap because GitHub continuously supplies new issues. The repo's social scenarios are not free; generating fresh, comparable social scenarios at scale is itself an open problem (the wave-4 sibling's "clean social reset" gap).
