# MineNPC-Task - Task Suite for Memory-Aware Minecraft Agents

- **authors**: user-authored benchmark (GPT-4o used for the initial snapshot)
- **year**: 2026 - **venue**: arXiv (UIST/acmart style)
- **arxiv_id**: 2601.05215
- **urls**: https://arxiv.org/abs/2601.05215
- **NOTE**: the brief flags this as the **closest existing benchmark to this
  repo's evidence stance**. Read closely. It is the single most aligned source.

## Primary-source facts

- **Research question**: how to evaluate memory-aware, mixed-initiative LLM NPCs
  on *real player tasks* in open-world Minecraft, fairly and reproducibly?
- **Task construction**: tasks are **elicited from formative + summative co-play
  with expert players** (not synthetic prompts), then **normalized into
  parametric templates with explicit preconditions and dependency structure**.
- **Validators**: each task is paired with **machine-checkable validators under a
  bounded-knowledge policy that forbids out-of-world shortcuts**. Judgment is made
  **only from in-world evidence**.
- **Harness** (model-agnostic): routes intents, plans with **single-turn
  clarification**, executes via **public Mineflayer APIs**, and captures
  **plan/act/memory events** - plan previews, targeted clarifications, memory
  reads/writes, precondition checks, repair attempts.
- **Outcome metric**: success reported **relative to the total number of
  attempted subtasks**, derived from in-world evidence (not a binary task pass,
  not a VLM score).
- **Snapshot**: GPT-4o evaluated on **216 subtasks across 8 experienced players**.
  Recurring failure modes: code execution, inventory/tool handling, referencing,
  navigation. Recoveries came from mixed-initiative clarification + lightweight
  memory. Participants wanted **stronger memory persistence across tasks**.
- **Release**: full task suite, validators, logs, and harness released.

## Interpretation (inference)

- MineNPC-Task and this repo share four design commitments almost verbatim:
  1. **Mineflayer high-level execution** (not pixels/camera-buttons);
  2. **machine-checkable validators / in-world evidence as truth** (not VLM
     scoring, not prose);
  3. **bounded knowledge** = no out-of-world shortcuts (parallels "the LLM
     proposes, the runtime owns physical truth");
  4. **plan/act/memory event capture** as the analyzable artifact (parallels the
     repo's transcript + memory + CycleJudgment trace).
- The gap MineNPC-Task does **not** cross: it is **single-NPC + one human
  player**, task-completion framed. It measures memory persistence and repair,
  not durable *social-material consequence between multiple actors* (possession,
  claim, obligation, weak commons, post-goal continuation). It is the right
  evidence *mechanism* but not the social target.

## Mechanically useful vs research contribution

- **Mechanically useful (high)**: (a) "parametric templates with explicit
  preconditions + dependency structure" is exactly the scenario-manifest shape
  this repo wants; (b) "report success / attempted-subtasks from in-world
  evidence" is a metric this repo can adopt directly; (c) the plan/act/memory
  event schema is a concrete checklist for what the repo's trace must capture;
  (d) bounded-knowledge policy is a ready-made fairness rule.
- **What would be overclaim**: claiming social evaluation. MineNPC-Task validates
  the *competence + memory + repair* layer with great rigor; the social-material
  layer is this repo's net-new contribution, not borrowable from here.

## WAM layer(s) informed

Physical-WAM (validator/evidence mechanism - strongest reference) and the
*method* for Social/Institutional evaluation (event capture + subtask accounting).
Reproducibility: harness + validators released ⇒ reproducible mechanism; the
GPT-4o numbers are a single snapshot.
