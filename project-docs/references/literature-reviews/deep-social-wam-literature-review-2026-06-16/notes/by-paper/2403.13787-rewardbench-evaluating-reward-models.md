## Primary-source facts

- **Research question**: reward models (RMs) are central to RLHF but are rarely EVALUATED
  directly. Build the first broad benchmark and leaderboard for RMs, to make these "opaque
  technologies" and the values embedded in them legible.
- **Method (how you evaluate a verifier)**: the dataset is a collection of
  prompt-chosen-rejected trios. For each trio, the RM scores prompt-with-chosen and
  prompt-with-rejected; the trio counts as a WIN if the chosen score exceeds the rejected
  score. Subset accuracy = percent of wins. This is the canonical "does the scorer rank the
  correct answer above the wrong one" protocol.
- **Crucially, the pairs have "subtle, but verifiable reasons" (bugs, incorrect facts) why
  chosen should beat rejected**, so a correct verdict requires recognizing a real defect,
  not a stylistic preference.
- **Structure**: five sections, four of which are the new RewardBench data: Chat (open
  conversation), Chat Hard (subtle conversation), Safety (abstain when appropriate),
  Reasoning (code and math), plus Prior Sets (existing preference test sets, weighted 0.5
  for noise). Reasoning up-weights a PRM-Math subset so code and math count equally.
- **Length-bias control (a robustness design choice worth importing)**: prior work shows
  RMs and RLHF have a length bias (longer = preferred). RewardBench is built so the chosen
  response is similar-length or SHORTER than the rejected, so a model cannot win by a
  length heuristic; the AlpacaEval Length subset isolates this. (This is the same spirit as
  reWordBench's no-confound design: remove the surface proxy so you test the real signal.)
- **Findings on RM behavior**:
  - DPO-trained implicit RMs vs trained classifiers: results "flatter the recent DPO
    methods" except on Prior Sets. But the DPO reward is sensitive to the reference model:
    using the "wrong" (similar-but-different) base model as reference "reduced the DPO
    trained RM performance to similar levels as the random baseline." A verifier whose
    score depends on an undocumented reference is fragile.
  - They report propensity for refusals, reasoning limitations, and instruction-following
    shortcomings across RMs (per-subset accuracy tables; many models do poorly on Chat Hard
    and Reasoning).
- **Honest limitations (theirs, important)**: (1) outside specific subsets they lack human
  preference labels and use semi-automatic chosen-rejected construction (then manually
  validated); (2) reasoning-domain formats "might potentially include spurious correlations"
  (artifacts a model could key on); (3) "whether and how the benchmark results correlate
  with downstream training" is UNRESOLVED; (4) possible contamination if models were trained
  on AlpacaEval/MT-Bench data.

## Interpretation (labeled inference)

- RewardBench is the field's "how do you measure whether your scorer is any good" answer:
  build held-out, defect-bearing pairs with the confound (length) removed, and measure
  ranking accuracy. The "subtle but verifiable reasons" design is the right shape for a
  social verifier's eval too: pairs that differ by a real, checkable property.
- The unresolved limitation (does benchmark accuracy predict downstream training quality)
  is the honest ceiling: a good RM-eval score does not guarantee a good RL outcome. This is
  the evaluation-side version of the gap the whole lane studies.

## Relevance to this repo (mechanical import vs research contribution)

- **Mechanical import**: (1) to TRUST the runtime verifier (or any advisory predictor
  scored against it), evaluate it on held-out, defect-bearing pairs the way RewardBench
  does: cases that differ by a real, checkable world property, with surface confounds (the
  repo analog of length: response verbosity, action count, phrasing) controlled out. (2)
  The DPO-reference fragility is a direct caution: a learned scorer can silently drop to
  random if a hidden dependency (reference model, prompt template) shifts; a deterministic
  verifier has no such hidden dependency.
- **Why a learned reward model is the more hackable choice (tie)**: RewardBench exists
  because learned RMs are opaque and need evaluation to even know what they reward; many
  fail on subtle, verifiable distinctions and are reference-fragile. A deterministic
  verifier is legible by construction (its rule IS its behavior) and needs no leaderboard to
  know what it checks.
- **Research contribution it is NOT**: building an RM leaderboard is not the repo's
  contribution; the import is the EVAL PROTOCOL (defect-bearing held-out pairs,
  confound-controlled) for validating the repo's own success signal. Support, not deliverable.
- **What it does not cover**: text-domain preference RMs only; no embodied/social verifier;
  and the paper itself flags that benchmark accuracy may not predict downstream training,
  so a high "verifier eval" score is necessary, not sufficient, for trusting the loop.
