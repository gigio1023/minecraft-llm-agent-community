## Primary-source facts (LaTeX-verified)

- **What it is**: LLM-driven objective DISCOVERY. An LLM (GPT-4) writes PyTorch code for a new offline preference-optimization LOSS function; each candidate is unit-tested, used to finetune an LLM, then scored on a downstream metric; the performance feeds back into the LLM to refine its next proposal. The discovered algorithm (the best one) is called DiscoPOP.
- **The loop (Algorithm 1)**: (1) burn-in: seed the LLM context with established losses (DPO, SLiC, KTO) given as CODE plus their performance; (2) propose a candidate loss `f_i` as JSON/code; (3) run unit tests (valid output shape, no NaN in loss or gradients) and resample with the error message on failure; (4) evaluate by finetuning (build on the alignment-handbook, replacing the DPO step with `f_i`, same hyperparameters as Zephyr-7B-Gemma) and scoring on MT-Bench (GPT-4 judge); (5) feed performance back, iterate. ~100 objectives evaluated.
- **What the loop does (not random search)**: the paper documents that LLM discovery ALTERNATES between exploring new objective concepts, tuning components (e.g. a label-smoothing temperature), and COMPOSING two conceptually different objectives, "does not perform a random search ... but instead composes various concepts in a complementary fashion." Discovered objectives generalize to other architectures and longer training.
- **The discovered DiscoPOP / LRML loss**: a dynamically weighted blend of logistic (DPO) and exponential loss, weight set by a sigmoid of the log-ratio difference rho with temperature tau=0.05. MT-Bench 7.916; several discovered objectives beat DPO (7.888); the best, DBAQL, reached 7.978. Held-out wins on Alpaca Eval 2.0, TL;DR summarization, IMDb.
- **Documented limitations (the lane's key bound)**:
    - The discovered LRML has a NON-CONVEX segment with negative gradients at rho=0. The authors flag this as possibly useful (curriculum/stochasticity) but also dangerous.
    - LRML FAILS to converge when beta is outside the discovery range (beta <= 0.01 or beta >= 2.5), "likely because beta != 0.05 was never seen or used during the discovery process." At beta=0.01 it gets stuck in a local minimum (predominantly negative reviews, reward ~0.15); at beta in {2.5, 5.0} the model collapses (loss spikes then NaN), attributed to the large gradient in the non-convex part.
    - Uses closed-source GPT-4 (limits reproducibility, costly).

## Interpretation (flagged as inference)

- DiscoPOP extends program discovery from math/algorithms to a MACHINE-LEARNING artifact (a loss function), with the evaluator being "finetune-then-judge". This is a weaker, noisier, more expensive evaluator than a math oracle, and the failure mode shows it: the discovered loss is calibrated to the exact conditions (beta=0.05) it was discovered under and breaks outside them.
- This is the single cleanest demonstration in the lane of a deep thesis bound: a discovered artifact inherits the distribution of its evaluator/discovery conditions and can silently fail off-distribution. It is the program-discovery analog of the wave-4 "loop converges to the verifier's knowledge center" finding (2510.16657) and the Q-Evolve in-distribution-only warning. For this repo: a verifier-discovered advisory-WAM rule will be valid only under the scenario distribution it was scored on; it must be re-verified before reuse in a new regime, never trusted as a universal law.
- The non-convex, never-validated-off-range loss also shows that an evaluator that is itself noisy/learned (GPT-4 judge on MT-Bench) admits artifacts that pass the metric but are fragile. This is one notch worse than FunSearch/AlphaEvolve's exact oracles, and exactly the regime the social layer would live in.

## Mechanically useful vs research contribution

- **Mechanically useful**: the discovery-loop structure for a typed artifact (burn-in with known artifacts + code + scores, propose code, unit-test, evaluate, refine); the unit-test gate on every candidate (validity before expensive evaluation, the repo's schema/permission gate analog); and, as a cautionary control, recording the exact regime (here: beta) a discovered artifact was scored under so off-regime use is flagged.
- **Not a contribution to copy**: DiscoPOP the loss is the contribution; the repo would not import it. The loop and the off-distribution caution are the borrows.
- **Bound**: the evaluator (finetune+judge) is noisy and the artifact overfits its discovery regime. This is precisely why the repo must not let a learned/judge signal own promotion, and must re-verify any discovered rule per scenario distribution.

## WAM layer(s) informed

Method-level; bridges Physical/Material (exact oracle) toward the noisy-evaluator regime the Social layer would face. Strongest single source for the "discovered artifact is calibrated to its discovery conditions and fails off-distribution" bound.
