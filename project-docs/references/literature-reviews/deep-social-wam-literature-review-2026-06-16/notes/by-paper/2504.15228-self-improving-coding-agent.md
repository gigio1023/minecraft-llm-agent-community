# A Self-Improving Coding Agent (SICA)

- **title**: A Self-Improving Coding Agent
- **authors**: Maxime Robeyns, Martin Szummer, Laurence Aitchison (University of Bristol; Aitchison Lab)
- **year**: 2025 (NeurIPS 2025 formatting in source)
- **venue/source**: arXiv 2504.15228
- **arxiv_id**: 2504.15228
- **urls**: paper https://arxiv.org/abs/2504.15228 ; hf https://huggingface.co/papers/2504.15228 ; code referenced (Aitchison Lab SICA)
- **source availability**: latex (deep-read methods + conclusion + safety sections)

## Primary-source facts

- **Research question**: can an agent system (code that orchestrates LLM calls and tools) autonomously EDIT ITS OWN code to improve its benchmark performance, without any weight updates.
- **Loop (self-referential, ADAS-style)**: keep an archive of prior agents and their benchmark results. Each step: take the best agent so far as the meta-agent, instruct it to inspect the archive, identify an improvement, and implement it (by editing the codebase); evaluate the new agent on the benchmark set; store results. Unlike ADAS (fixed meta-agent), SICA selects the meta-agent FROM the archive (Algorithm: A_{i+1} generated by A_{argmax p}). No weight updates; the numeric score only picks the next meta/base agent.
- **Utility function (the success signal it optimizes)**: U = 0.5 * p_score + 0.25 * (1 - min(1, cost/$10)) + 0.25 * (1 - min(1, time/300s)), with a 0.5 timeout penalty. So benchmark score is half the objective; cost and wall-clock time are the other half.
- **Initial agent**: tools to open/close/overwrite files, run shell, a calculator, submit-answer, sub-agent return/early-exit, and an archive-analysis tool. Sub-agents (coding, problem-solver, reasoning) both enable autonomous decomposition AND manage context (a sub-agent's context is discarded on return, leaving only a summary). An asynchronous LLM OVERSEER runs every 30s in a concurrent thread, watching the agent callgraph + event stream for pathological behavior/loops, and can notify or CANCEL a deviating/stuck agent. Context is laid out to preserve the KV cache (file edits appended as diffs, periodically re-consolidated).
- **Results**: self-improvement works, especially on longer-horizon agentic tasks: 17% -> 53% on a random subset of SWE-Bench Verified, with a slight decrease in average time per problem. Gains also on LiveCodeBench and on two synthetic agent benchmarks the agent itself curated.
- **Limitations (honest)**: hard to get the LLM to propose "truly novel, innovative, feasible and interesting" improvements (echoes open-ended-learning literature). Path dependency: a bad early idea (e.g. "fixating on caching open files") lowers the quality of later ideas and raises variance. The short 5-minute timeouts depress initial scores, and "much of the initial improvement... arises from features speeding up file editing and agent cost," not traditional SWE-Bench correctness.
- **Safety (explicit section)**: two mitigations. (1) Observability: a web interface plus the async LLM overseer (which may use a different LLM than the running agent) give humans insight into chain-of-thought, actions, sub-agent calls; because weights are NOT updated, this is argued to be strong mitigation. The authors state that for recursively self-improving systems that ALSO update weights, "observability mechanisms, on their own, are clearly inadequate." (2) The iterative evaluation-based update can include safety evaluations in the benchmark set to validate each agent before it becomes the next meta-agent.

## Primary-source facts vs interpretation boundary

The numbers above are paper-stated (no independent re-run). The 17% -> 53% gain is on a random subset of SWE-Bench Verified, and the paper explicitly attributes much of it to file-editing/cost speedups rather than new correctness.

## Interpretation (flagged as inference)

- SICA is the lane's strongest DIGITAL analog of ENPIRE/DGM: a coding agent that edits its own scaffold against an external benchmark, with an archive and a meta-agent. It is the "what does the team improve" answer for the digital case: prompts, tools, sub-agents, oversight, context management - i.e. the agent's own software, not its weights.
- Two findings map straight onto the repo. (1) The utility function is multi-objective (score + cost + time), a useful reminder that a real loop optimizes more than accuracy; but it also means the agent can improve "utility" by getting cheaper/faster without getting more correct (their own caveat). For the repo, the verifier-defined correctness must be the dominant term, or the loop drifts toward cheap-but-wrong. (2) The safety stance is exactly the repo's: observability + an external overseer is adequate ONLY because there are no weight updates; once a system can change its own learned policy, observability alone fails. The repo's advisory-WAM rule (the WAM never overrides the verifier) keeps it in the safer, no-weight-authority regime.
- Path dependency (bad early ideas poison later ones) is a concrete argument for keeping diverse stepping stones (DGM's archive + non-greedy sampling) rather than greedily following the latest improvement.

## Mechanically useful vs research contribution

- **Mechanically useful**: the self-referential archive + meta-agent loop as the template for a coding agent improving the repo's advisory-WAM code / prompts / gated skills; the async overseer that can CANCEL a deviating or stuck agent (a clean watchdog the repo could mirror for its loop); sub-agent-context-discard for bounded context; the explicit safety practice of putting safety evals INTO the benchmark set so each candidate is validated before promotion.
- **Not a contribution to copy**: SICA's contribution is demonstrating scaffolding-level self-improvement; building such a loop is engineering support for the repo. Do not claim "SICA for Minecraft."
- **Honest bound**: the gains are scaffolding-level and partly cost/speed, not new capability; the authors say genuinely novel improvements are hard to elicit. A WAM-improvement loop should expect tuning/recombination, not invention, and must keep verifier-correctness dominant over cost/time terms.
