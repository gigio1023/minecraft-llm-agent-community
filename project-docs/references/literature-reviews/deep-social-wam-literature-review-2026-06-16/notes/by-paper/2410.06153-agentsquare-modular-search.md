## Primary-source facts (LaTeX-verified)

- **Problem named**: Modularized LLM Agent Search (MoLAS) -- automatically optimize an LLM agent design by reusing knowledge from previously published/evaluated modules.
- **Modular design space**: abstracts existing LLM agent designs into FOUR fundamental modules with a uniform IO interface: Planning, Reasoning, Tool Use, Memory. Any agent is a choice of module implementations.
- **Search framework = AgentSquare**: two mechanisms over the modular space: (1) module EVOLUTION (an LLM proposes new module implementations); (2) module RECOMBINATION (combine modules from different agents). To cut search cost, a PERFORMANCE PREDICTOR uses in-context surrogate models to skip unpromising designs before full evaluation.
- **Results (as stated)**: across six benchmarks (web, embodied, tool-use, game applications) AgentSquare beats hand-crafted agents by an average 17.2% over the best-known human designs. It also produces interpretable design insights.

## Interpretation (flagged as inference)

- AgentSquare sits between ADAS (search arbitrary code) and a fixed pipeline: it constrains the search to a four-module schema with uniform IO. This is a useful middle point: less open-ended than ADAS (so cheaper and more interpretable), more flexible than a fixed workflow. The four modules (Planning, Reasoning, Tool Use, Memory) are a reasonable decomposition that maps loosely onto the repo's actor (planning ~ PlanBeads, reasoning ~ Actor Turn selection, tool use ~ Action Cards, memory ~ actor memory).
- The performance predictor is the AgentSquare instance of the recurring surrogate-scorer pattern (also in MIPRO). It speeds search but introduces the same risk: an in-context surrogate, not the verifier, decides which designs are even evaluated. If the surrogate is biased, good designs get pruned and the search optimizes the surrogate's view, not real task performance.

## Mechanically useful vs research contribution

- **Mechanically useful**: (1) constraining agent search to a small typed module schema with uniform IO is a cheaper, more interpretable alternative to arbitrary-code search, and the Planning/Reasoning/Tool-Use/Memory decomposition is a candidate schema for this repo's actor; (2) module recombination as a way to reuse prior good components; (3) the explicit acknowledgement that vast search spaces need triage.
- **What would be overclaim**: the 17.2% gain is over six QA/web/game/tool benchmarks with their own metrics, not embodied social tasks. And the surrogate predictor is a cost hack, not a validated stand-in for a verifier. Borrow the modular schema, not the result, and never let the surrogate be the final gate.
- **Honest bound for thesis**: same as the lane. AgentSquare's gain is real only against its benchmarks' scores; for this repo, any recombined/evolved module must be scored by the runtime verifier on held-out scenarios (surrogate to triage only).
