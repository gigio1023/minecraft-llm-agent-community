# ROCKET-1 - Mastering Open-World Interaction with Visual-Temporal Context Prompting

- **authors**: Cai, Mu, Liu, Liang (Team CraftJarvis)
- **year**: 2024 - **venue**: CVPR 2025
- **arxiv_id**: 2410.17856
- **urls**: https://arxiv.org/abs/2410.17856 ; code https://github.com/CraftJarvis/ROCKET-1 ; weights https://huggingface.co/CraftJarvis/MineStudio_ROCKET-1.12w_EMA (also ROCKET-3-1.5x)

## Primary-source facts

- **Research question**: how should a high-level reasoner (VLM) communicate
  *spatial* intent to a low-level Minecraft policy, given that language under-
  specifies which object to act on?
- **Method - visual-temporal context prompting**: the VLM marks **object
  segmentation masks** + an **interaction-type id** on past/current frames; the
  low-level policy `π(a_t | o_{1:t}, m_{1:t}, c_{1:t})` reads observation +
  segmentation to act. ROCKET-1 is described as "the first **segmentation-
  conditioned policy** in Minecraft."
- **Architecture**: concatenate observation `o_t` (3-ch) and mask `m_t` (1-ch)
  into a 4-channel image (first conv extended 3->4 ch, new weights zero-init);
  fuse via visual backbone + attention pooling; a **Transformer-XL** models
  temporal dependencies and folds in interaction-type tokens to predict `a_t`.
- **Data - backward trajectory relabeling**: place a box/point at frame-center
  just before an interaction event, run **SAM-2 in reverse** to propagate the
  mask back in time, label the interaction type consistently. This auto-generates
  segmentation-annotated trajectories (offline BC; rewards discarded).
- **Inference plug-in**: SAM-2 provides real-time object tracking; **Molmo**
  supplies point grounding; a GPT-class VLM does reasoning.
- **Pipeline taxonomy (Fig. 2, useful)**: (a) end-to-end token VLA, (b) language
  prompting, (c) latent prompting, (d) future-image prompting (build a world
  model - flagged as suffering hallucination / temporal inconsistency / limited
  scope), (e) ROCKET-1's visual-temporal prompting.
- **Result**: +76% absolute improvement in open-world interaction; enables tasks
  prior methods could not do (e.g. nether-portal block placement).

## Interpretation (inference)

- ROCKET-1 is a **visual policy with a structured, non-language interface**
  (masks + interaction ids). It is the cleanest example in the Minecraft
  literature of "replace ambiguous prose with a typed, spatial channel between
  reasoner and executor" - conceptually parallel to this repo's insistence that
  the LLM proposes via **typed tool-call params**, not prose the runtime parses.
- The paper explicitly rejects future-image world models (their Fig. 2d) as the
  interface because predicting pixels is unreliable - an independent argument
  that a **structured-state** predictive layer beats a pixel one for control.

## Mechanically useful vs research contribution

- **Mechanically useful**: (a) the *principle* "communicate intent through a
  typed structured channel, not prose" validates this repo's Action-Card schema;
  (b) backward-relabeling with SAM-2 is a cheap-label pattern for a visual lane;
  (c) ROCKET-1 is a strong **interaction executor baseline**.
- **What would be overclaim**: importing segmentation-conditioned pixel control
  as runtime authority. It is the wrong altitude (Mineflayer high-level actions
  already name the target object); and its evidence is task success, not social
  consequence.

## WAM layer(s) informed

Physical-WAM (interaction executor). Reproducible: weights + code public.
