# DEPS: Describe, Explain, Plan and Select (Minecraft LLM planning)

- **title**: Describe, Explain, Plan and Select: Interactive Planning with Large Language Models Enables Open-World Multi-Task Agents
- **authors**: Zihao Wang, Shaofei Cai, Anji Liu, Xiaojian Ma, Yitao Liang (Peking University, BIGAI, UCLA)
- **year**: 2023 (arXiv 2023-02-03; v-final 2024-07)
- **venue/source**: NeurIPS 2023
- **arxiv_id**: 2302.01560
- **urls**: paper https://arxiv.org/abs/2302.01560 ; code https://github.com/CraftJarvis/MC-Planner
- **source availability**: latex (main.tex deep-read)

## Primary-source facts (LaTeX-verified)

- **The four-step loop**: an LLM planner decomposes a goal (e.g. ObtainDiamond) into sub-goals; a controller executes each via a goal-conditioned policy. On failure: (1) a DESCRIPTOR summarizes the current world state and the failed outcome into text; (2) the LLM as EXPLAINER locates the error in the previous plan via self-explanation (e.g. "iron ore needs a stone pickaxe, not a wooden one"); (3) the LLM as PLANNER re-plans using the description and explanation; (4) a learned SELECTOR ranks parallel sub-goals by predicted steps-to-completion (proximity) to pick the most accessible one.
- **The implicit world model**: the LLM's plan encodes the task's precondition structure (which tool gates which resource, which item enables which action). DEPS never builds an explicit `p(s'|s,a)`; the dynamics live inside the LLM's planning prior, and the "explain" step is the LLM REASONING about why its implicit model's prediction (the plan would work) contradicted the observed outcome.
- **The descriptor is the grounding channel**: the LLM does not see raw pixels; the descriptor converts the real Minecraft state and execution result into text the LLM can reason over. This is the structured-state interface, not pixels.
- **Selector is trainable and state-aware**: it predicts time-steps-remaining for each candidate sub-goal given the current state, addressing plan inefficiency (challenge 2), separate from plan correctness (challenge 1).

## Numbers as stated

- First zero-shot multi-task agent to robustly accomplish 70+ Minecraft tasks; nearly doubles overall performance.
- First planning-based agent to accomplish the ObtainDiamond grand challenge; promising update on it.
- Between on-par and >50% relative improvement over existing/concurrent LLM-based planning methods on non-open-ended domains; requires no planning training for the environment.

## Reproducibility status

Self-reported (NeurIPS 2023), code released. Minecraft (MineDojo/MineRL) success rates; the controller is a separately trained policy, so the LLM-planning contribution is the describe-explain-plan-select loop, not end-to-end control.

## Relevance to this repo

- **Mechanical import (high, Minecraft-specific)**: DEPS is the clearest precedent that an LLM's IMPLICIT world model (its plan's precondition structure) is useful in Minecraft, and that the way to keep it honest is a runtime DESCRIPTOR that reports the actual state so the LLM can explain and correct the gap. The repo's verifier is a stronger descriptor: it does not just summarize, it auto-labels the true (state, action, next-state) delta. So the repo can run a describe-explain-plan loop where "describe" = the verifier's typed delta and "explain" = the advisory WAM reasoning about its own misprediction.
- **The lane point**: DEPS shows the LLM's predicted dynamics ARE wrong often enough that an explicit failure-feedback-and-re-explain loop is necessary even at the Physical/Material layer (tool-gating preconditions). This is direct evidence for the repo that a prompt-based WAM at Physical/Material will mispredict and needs the verifier to catch it.
- **Honest scope**: DEPS is single-agent and task-completion-driven; it never models possession transfer between agents, obligations, or another agent's response. Its describe-explain-plan loop transports to the repo's mechanics, but the SOCIAL predictions (who-owes-whom, trust deltas) are outside what DEPS validates.
- **Research contribution vs support**: borrowing the describe-explain-plan-select loop is mechanical. The contribution is grounding "describe" in a verifier's typed delta (not an LLM-written summary) and extending the predicted state from physical preconditions to social-material ledger values.

## WAM layer(s) informed

Physical and Material (tool-gating preconditions, sub-goal proximity). It is the Minecraft existence proof that an implicit LLM world model needs runtime grounding to stay correct, which is the repo's exact verifier argument. Social/Institutional unaddressed.
