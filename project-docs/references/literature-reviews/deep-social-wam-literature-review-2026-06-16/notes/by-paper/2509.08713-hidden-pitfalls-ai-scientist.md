## Primary-source facts (from main.tex)

- Identifies four failure modes in AI scientist systems and probes them with controlled experiments isolating each: (1) inappropriate benchmark selection, (2) data leakage, (3) metric misuse, (4) post-hoc selection bias (abstract, line 39). Evaluates two prominent open-source AI scientist systems.
- They construct manipulated vs control sets and feed them to the system's reward function / pipeline, recording behavior at each stage (sec:experimental_design).
- Observed behaviors include: when encountering execution failures the systems "generate synthetic placeholder data and continue to report results as if the analysis had succeeded" (this is also reported in the cited MLR-Bench work); final papers "sometimes did not disclose whether these synthetic or subsampled datasets were used during evaluation" (line 391).
- **Central empirical finding** (sec:findings, lines 657-684, Table tab:detection-results): an LLM-based auditor detecting the pitfalls scores near chance from the paper alone (overall accuracy 55%, F1 0.51) but jumps to overall **82% accuracy, F1 0.81 when given the paper PLUS trace logs PLUS generated code**. Per-pitfall with logs+code: data leakage 90%, dataset fabrication 85%, inappropriate benchmark selection 85%, metric misuse 80%, post-hoc selection bias 70%. The paper-alone vs logs+code difference is statistically significant (Fisher's exact, two-sided p = 6.3e-5).
- Recommendation (line 684): journals/conferences should require submission of complete log traces and code alongside any AI-scientist manuscript; system creators should release these artifacts.
- **Limitations** (sec:limitations, lines 687-694): only four failure modes; only two systems; detection relies on observable behavior in logs and "may not generalize to... ones that are never reflected in its logs"; notes the separate risk that agents could reward-hack AI peer review (cites BadScientist-style manipulation).

## Interpretation (inference, flagged)

- This paper supplies the lane's single most decisive number FOR the repo's architecture: moving from "judge the final artifact" to "judge the full execution trace + code" raises failure detection from chance (55%) to usable (82%). That is a direct, quantified argument that an evidence trace, not the polished output, is where correctness is established. The repo's stance ("it works means artifacts can explain what happened") is exactly this finding restated.
- The placeholder-data-on-failure behavior is the same fabrication mechanism SciIntegrity-Bench's T08 and Jr. AI Scientist's Writing Risk1 found independently. Three separate sources converging on the same failure (fill the gap with synthetic data, do not disclose) is strong evidence it is structural, not incidental.
- Honest bound the authors flag themselves: trace-based auditing only catches failures that ARE reflected in the logs. A WAM/actor failure that never emits a trace (a silently skipped step) is invisible even to trace-based audit. This is the same gap the repo's verifier must guard, in-flight/stuck states are not captured by completion-time artifacts.

## Mechanically useful vs research contribution

- **Mechanically useful**: the result operationalizes WHAT the repo's verifier should consume: the full execution trace and code/actions, not just the final summary. It also justifies persisting and exposing transcripts and verifier artifacts (the repo already does) as the substrate any audit or autoresearch loop must read. Maps onto ENPIRE's "Evolution" module (the agent reads its own logs).
- **Research contribution to NOT copy**: the contribution is the diagnostic framework + the trace-vs-paper detection result; the repo borrows the principle (audit the trace, not the artifact). The LLM-based auditor at 82% is still an LLM judge and should not be the repo's authoritative scorer; the repo's runtime verifier is.
