# MindForge: Empowering Embodied Agents with Theory of Mind for Lifelong Cultural Learning

- title: MindForge: Empowering Embodied Agents with Theory of Mind for Lifelong Cultural Learning
- authors: Mircea Lica, Ojas Shirekar, Baptiste Colle, Chirag Raman
- year: 2024 (arXiv); NeurIPS 2025
- venue/source: NeurIPS 2025; arXiv 2411.12977
- arxiv_id: 2411.12977
- urls: paper https://arxiv.org/abs/2411.12977 ; code https://github.com/tapri-lab/mindforge ; hf https://huggingface.co/papers/2411.12977
- source availability: LaTeX (`papers/latex/2411.12977/neurips_2025.tex`), read abstract + method (ToM structure, communication) + experiment setup.

## Primary-source facts (what the paper states)

- Research question: can embodied LLM agents in Minecraft learn over a lifetime by
  *cultural learning* (learning from other agents via perspective-taking), so that
  weak open-weight models improve through communication rather than fine-tuning?
- This is the most Minecraft-grounded ToM paper in the lane: built on the same runtime
  family the project uses (MineDojo + **Mineflayer** for motor control; chat via the
  in-game chat system). It extends Voyager with three additions:
  1. A **structured ToM representation** using the BigToM causal template (context,
     desire, percept, belief, causal event, action), operationalized from the classic
     Belief-Desire-Intention (BDI) framework. Beliefs are split into four kinds:
     perception-related (from the Minecraft API), task-related, interaction-related
     (from chat), and **partner-related** (tracking others' mental states). Each belief
     type is a separate LLM call.
  2. **Natural-language inter-agent communication**: when an agent fails a task it opens
     a communication round and uses its partner-BigToM model to ask for help; a dedicated
     LLM instance interprets the conversation back into the BDI template (it does not
     directly drive any other module).
  3. A **multi-component memory** (Soar-inspired: semantic + episodic), feeding beliefs.
- Theory of Mind specifically: each agent maintains a *partner-specific BigToM graph*
  inside its own beliefs and uses it to "simulate what the partner knows, believes, or
  needs," updating partner beliefs as new chat arrives. The paper frames self-modeling of
  one's own beliefs/desires/intentions as the prerequisite for *recursive social reasoning*
  (model others by analogy to modeling oneself).
- Results:
  - Instructive setting (GPT-4 expert teaching open-weight novices): MindForge agents get
    3x more Minecraft tech-tree milestones and 2.3x more unique items than Voyager; >40%
    improvement on two "seemingly trivial" tasks.
  - The paper explicitly identifies Voyager's two failure cases it fixes via perspective-
    taking and communication: (1) **false beliefs** and (2) **code generation** errors.
  - Collaborative setting (two equally-weak agents): one round of GPT-4 seeding + seven
    turns of peer-to-peer dialogue lifts population success from 62% to 79%, which they
    frame as a Condorcet-Jury-Theorem "many-minds" boost from richer interaction, not from
    a bigger model.
- Reproducibility aids: code released (github.com/tapri-lab/mindforge); built on public
  Voyager/MineDojo/Mineflayer stack.

## Interpretation (my reading, labeled inference)

- For this project this is the most transferable single source on the lane, because it is
  ToM *on the project's own runtime* (Mineflayer, in-game chat, Minecraft API percepts).
  It demonstrates that a structured belief representation (percept -> belief -> desire ->
  action) over Minecraft state is buildable and improves embodied task outcomes.
- The partner-related belief category is exactly the Social WAM target: a typed, updatable
  model of another actor's mental state, fed by observed percepts and chat. The repo could
  hold an analogous "what I currently believe Bob believes / wants" record per actor.
- Caveat: MindForge's gains are largely *cooperative knowledge transfer* (expert helps
  novice) and tech-tree progress, NOT the project's harder questions (possession, claims,
  weak commons, obligation). It shows ToM helps task learning; it does not show ToM predicts
  social-material consequences (cost to others, trust shifts) in the project's sense.

## Mechanically useful vs research contribution

- Mechanically useful for this repo:
  - The BigToM/BDI belief schema (context, desire, percept, belief, causal event, action),
    with the four belief categories, is a ready typed structure for an actor's own state and
    for a partner model. It maps onto the repo's existing typed `current_state` + memory.
  - "Open a communication round on failure, then fold the exchange back into typed beliefs"
    is a clean, loggable pattern for request/handoff/repair events in the Social layer.
  - Because it already uses Mineflayer + in-game chat, the integration surface is the closest
    of any lane source to the repo's runtime.
- What this repo should avoid / overclaim risk:
  - Many separate LLM calls per belief type is expensive; the repo's cost discipline (mock ->
    small dry-run -> one medium) and advisory-WAM minimalism argue against a belief call per
    category per step.
  - Do not treat the BigToM "intention == action" equivalence as runtime authority. The paper
    itself flags that intentions (internal commitments) can diverge from actions (observable);
    the repo must keep physical success defined by the verifier, never by a declared intention.
  - Do not generalize the cooperative tech-tree gains into a claim that ToM solves the project's
    contested social-material questions; that is unproven here.

## WAM layer(s) informed

- **Social WAM** (partner-belief modeling, perspective-taking, communication-driven belief update).
- **Institutional WAM** secondarily (cultural transmission / expert-novice knowledge transfer is
  proto-role and proto-convention behavior).
- Depends on **Physical WAM** (percept-related beliefs come straight from the Minecraft API; the
  whole structure sits on Mineflayer physical truth).

## Reproducibility

- Partial-to-good: code released and built on the public Voyager/MineDojo/Mineflayer stack, but
  the strongest results depend on a GPT-4 expert (provider-gated) and open-weight novices.
  Transfers as schema + runtime-integration pattern and as a reproducible-in-principle baseline.
