## Primary-source facts (what the paper states/shows)

- **Research question**: do standard decontamination methods (n-gram/string overlap, embedding similarity) actually protect benchmarks, or can semantically-equivalent "rephrased samples" of test items evade them and inflate scores?
- **The mechanism (rephrase attack + LLM decontaminator)**:
  - **Rephrase attack**: use a strong LLM (GPT-4 or Claude) at non-zero temperature to reword a test prompt, then loop (re-rephrasing up to MaxRetry) until an n-gram/embedding detector no longer flags it. Non-deterministic, so no simple reversible transform. For text it reorders words / swaps synonyms; for code it changes style, naming, and language (Python -> C/Java/JS/Rust/Go) plus dead-code injection. Translation maps identical meaning to very different embeddings, evading both n-gram and embedding search.
  - **LLM decontaminator (the fix)**: a two-step detector. (1) embedding similarity finds the top-k most similar training items per test case; (2) a strong LLM (GPT-4) judges each candidate pair as same-or-not, returning True/False. Catches rephrased pairs n-gram misses and embedding-only over-flags, at ~k times the test-case count in cost.
- **Numbers / results as stated** (quoted by sub-agent):
  - Inflation, MMLU: "a Llama-2-13B trained on a rephrased test set can reach 85.9 accuracy on MMLU while being undetectable by n-gram overlap." 7B rises 45.3 -> 88.5; 13B 54.8 -> 89.9.
  - Inflation, HumanEval: CodeLlama 7B 32.9 -> 67.7, 13B 36.0 -> 81.1; "In contrast, GPT-4 can only achieve 67.0 on HumanEval" (a rephrased 13B beats GPT-4).
  - Inflation, GSM-8k: Llama-2 7B 14.6 -> 86.7; 13B 28.7 -> 95.3.
  - Detector failure (F1, lower = more evasion): 10-gram F1 drops to 0 on all rephrased and translated MMLU subjects; multi-qa BERT embedding search "completely ineffective against translated samples."
  - Real-world rephrased overlap found by the LLM decontaminator (HumanEval has 164 cases): StarCoder-Data subset 26 = 15.9%; CodeAlpaca 21 = 12.8%; RedPajama-Data-1T subset 14 = 8.5% (abstract: "8-18% of the HumanEval benchmark overlaps"). FLAN CoT vs MMLU 76 = 0.5%; MATH self-rephrase 79 = 1.58%.
  - Unintentional contamination found in CodeAlpaca, a synthetic dataset generated by GPT (Davinci-003): GPT-generated training data can carry rephrased benchmark items.
- **Reproducibility status**: SELF-REPORTED experimental results (fine-tuning runs, F1 tables); tool publicly released (github.com/lm-sys/llm-decontaminator). The decontaminator itself depends on a GPT-4 judge, so detection is not deterministic.
- **Stated limitations**: (1) detecting contamination "without access to training data remains an open problem" (the decontaminator needs the candidate training set). (2) defining contamination is unresolved: the number-substituted-only case (same problem, different numbers) is borderline. (3) the real remedy is not better detection but "fresh one-time exams" that iterate as fast as model development (e.g. CodeForces weekly competitions); any static, unhidden test set can eventually be rephrase-contaminated.

## Interpretation (flagged as inference)

- This paper transfers to the repo as the CONTAMINATION-CHECK GATE on synthesized scenarios. The danger is concrete: if held-out social scenarios are LLM-synthesized (the SOTOPIA-pi path) from the same models in the training loop, they can be paraphrase-near-duplicates of training scenarios that n-gram and embedding checks miss, and the paper shows this inflates scores to GPT-4 level on a 13B (HumanEval 36.0 -> 81.1, beating GPT-4's 67.0). So scenario admission needs a paraphrase-aware check, not just string overlap.
- The strongest stated remedy, "fresh one-time exams," is the same conclusion LiveCodeBench (2403.07974) reaches structurally: against a self-improving loop, the durable defense is fresh, one-time, content, not better post-hoc string matching. For the repo this means genuinely new scenarios over re-scored old ones.
- The part that transfers to a deterministic-verifier setting is the principle, not the GPT-4 judge: because the repo verifies state-action-next-state transitions deterministically, contamination of the scenario PROMPT/SETUP is the residual leak, so a decontamination check belongs at scenario-admission time, while time-gating remains the cheapest structural defense.

## Mechanically useful vs research contribution

- **Mechanically useful**: (1) the rephrase-attack finding as the reason an LLM-synthesized held-out social set can be silently stale, motivating a paraphrase-aware admission check against the training-scenario bank; (2) the top-k-embedding-then-judge decontaminator shape as that admission check (with the caveat that the judge is an LLM, so it is a heuristic gate, not the success label); (3) the "fresh one-time exams" recommendation as the principle behind authoring genuinely new scenarios rather than re-scoring; (4) the CodeAlpaca finding (GPT-generated data carries rephrased benchmark items) as a warning that LLM-generated scenarios specifically risk leakage.
- **Not a contribution to copy / overclaim risk**: the decontaminator is support tooling, not a research contribution, and its GPT-4 judge is non-deterministic. The repo should use it only as an admission-time heuristic and keep the success label on the deterministic verifier; the defensible point is that fresh, time-gated, deterministically-verified content beats post-hoc string decontamination.
