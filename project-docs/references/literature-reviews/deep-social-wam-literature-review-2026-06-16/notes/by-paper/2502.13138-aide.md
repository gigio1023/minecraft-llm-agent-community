## Primary-source facts

- **Research question**: can ML engineering be automated by framing it as code optimization and searching the solution space with an LLM.
- **Core method (tree search over solutions)**: a solution s is code; an evaluator h: S -> R gives a scalar score (the validation metric); all solutions form a solution tree T (nodes = scripts, edges = improvement attempts); a search policy pi(T) picks the next base node; a summarization operator Sigma(T) compresses tree history into the prompt; a coding operator f(s, Sigma(T)) drafts/debugs/improves. The loop: propose s_n = f(s, Sigma), evaluate v_n = h(s_n), record (node, score), pick next base via pi, return argmax score (Algorithm 1).
- **Search policy (hard-coded)**: draft if not enough initial solutions yet; debug if a buggy node is within a debug depth; otherwise improve the best non-buggy solution. So: explore diverse drafts first, then hill-climb the best, with bounded debug attempts per broken solution.
- **Coding operator entry points**: drafting (plan + single-file Python program from scratch), debugging (repair from error logs/traces while preserving approach), improving (refine a promising solution).
- **Results**: state-of-the-art on the Weco-Kaggle benchmark, OpenAI's MLE-bench (it is the scaffold behind MLE-bench's best o1-preview result), and METR's RE-Bench.
- **Code-complexity growth (relevant to degradation)**: aggregated code complexity (LOC, LLOC, Volume, N1, MI) increases steadily as iteration steps grow; "solutions tend to become more elaborate" over steps.
- **Cost**: per-task LLM cost mostly under $1.50 (up to ~$2.50) with GPT-4 Turbo, far below human experts or AutoML.

## Interpretation (flagged as inference)

- AIDE is the lane's reference implementation of the inner search a coding-agent autoresearch loop runs: hill-climb code against a scalar validation score, keeping a tree of attempts. It is exactly the "Policy Improvement" search ENPIRE describes, instantiated for ML code, and it is why MLE-bench's and RE-Bench's best results both used AIDE.
- Two design choices matter for the repo. (1) The evaluator h is an EXTERNAL validation metric, not the agent's judgment, preserving proposer-vs-scorer separation. (2) The tree + summarization avoids the context-accumulation pathology RE-Bench measured (the Modular agent degraded when it kept appending history; AIDE keeps whole-solution nodes and a compressed summary instead). For a WAM-improvement loop, this argues for episodic candidate-level memory over raw transcript accumulation.
- The steady code-complexity growth is an early, mild version of the degradation SlopCodeBench (2603.24755) measures at scale: iterative LLM editing tends to bloat code unless something pushes back.

## Mechanically useful vs research contribution

- **Mechanically useful**: the solution-tree + scalar-evaluator + draft/debug/improve search policy as a concrete loop the repo could mirror to improve an advisory-WAM predictor or a gated skill candidate (each candidate = a node, the runtime verifier = the evaluator h); the summarization-operator pattern to keep the prompt bounded; the bounded-debug-depth rule as a loop-effort cap.
- **Not a contribution to copy**: AIDE's contribution is the code-optimization-as-tree-search framing for ML engineering; building such a search is engineering support for the repo, not a research claim.
- **Honest bound**: AIDE depends on a clean scalar validation metric per task. The repo has that at Physical/Material layers; Social/Institutional layers lack a single scalar h, so the tree search transfers cleanly only at the lower layers.
