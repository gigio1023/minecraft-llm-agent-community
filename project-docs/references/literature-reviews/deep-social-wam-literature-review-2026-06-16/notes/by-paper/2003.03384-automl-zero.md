## Primary-source facts (LaTeX-verified)

- **Research question**: can AutoML go beyond constrained search spaces (architecture-only, learning-rule-only, augmentation-only) and discover WHOLE ML algorithms (model + optimizer + initialization + loss) from only basic mathematical operations, with minimal human design? (intro, paper_part2_introduction.tex)
- **Why prior AutoML was constrained**: existing AutoML constrains search to "isolated algorithmic aspects" using "sophisticated expert-designed layers as building blocks." Two stated drawbacks: (1) human-designed components bias results toward human-designed algorithms, reducing innovation; (2) constructing constrained spaces "creates a new burden on researchers," undermining the time-saving goal (intro).
- **Search space**: algorithms are programs over a small virtual memory (scalar / vector / matrix address spaces, all float, dimension F = task feature size). Programs are three component functions: `Setup`, `Predict`, `Learn` (init / predict / learn). 65 ops total, chosen by a "typically learned by high-school level" criterion; ML concepts, matrix decompositions and derivatives are purposefully excluded. Op-specific args are memory addresses or searched real constants. Component-function size is itself a free parameter (methods, paper_part4_methods.tex).
- **Search-space sparsity (the key cost fact)**: the space is "so generic that it ends up being quite sparse." "Good algorithms to learn even a trivial task can be as rare as 1 in 10^12." Existing AutoML spaces were "constructed to be dense with good solutions," which is why on those spaces "advanced techniques are often only marginally superior to simple random search." Random search (RS) cannot make enough progress here; they "move away from RS" (intro).
- **Search method**: regularized evolution (aging evolution). Population of P empty programs; each cycle: tournament-select best of T random, copy + mutate to a child, remove oldest. Mutations: (i) insert/remove a random instruction, (ii) randomize a whole component function, (iii) modify one argument (methods).
- **Compute scale / engineering**: throughput 2k-10k algorithms/second/CPU core via small proxy tasks (input projected to 8 <= F <= 256 dims), functional equivalence checking (FEC, 4x speedup by hashing predictions after 10 train + 10 valid steps), and "hurdles" (5x). Distributed across W worker processes (typically 100 < W < 1000), each its own P-sized population on a commodity CPU core, exchanging models via random migration. Run time: 5 days. Meta-params (P, T) were "not tuned at all" / decided in small experiments / taken from prior work (methods; "electricity ... matched with purchases of renewable energy" footnote).
- **Results**: starting from empty programs with teacher-labeled data, evolution rediscovers gradient-descent-trained 2-layer neural networks. On projected binary CIFAR-10, evolved models surpass a GD-trained NN by discovering multiplicative interactions, normalized gradients, weight averaging. When seeding the population with an existing algorithm, evolution adapts it to the task type: dropout-like ops emerge under little data, learning-rate decay emerges when faster convergence is needed (intro, results).

## Interpretation (flagged as inference)

- AutoML-Zero is the purest demonstration that self-improvement of a learning algorithm is REAL: evolution rediscovers backprop and invents real techniques (normalized gradient, weight averaging) without those concepts being in the primitive set. That is a genuine "machine improves the learning algorithm" result, not a tuning result.
- It is also the clearest demonstration that this is EXPENSIVE and search-space-sensitive: the generic space costs you a 1-in-10^12 needle, ~100-1000 CPU cores for 5 days, and bespoke speedup engineering (FEC, hurdles, proxy tasks) just to make evolution viable. Remove the human-designed structure and you pay for it in search cost. This is the central honest bound my lane lands.
- The proxy-task trick (project CIFAR-10 to 8-256 dims to get 10k evals/sec/core) is the same kind of move an autoresearch loop must make to afford search: cheap, lower-fidelity surrogates for the real metric.

## Mechanically useful vs research contribution

- **Mechanically useful (for this repo)**: the conceptual recipe for an autoresearch loop that improves the advisory-WAM TRAINING RECIPE (not the runtime): represent the "thing being improved" (a feature set, a reward shaping, a predictor head) as a small typed program/config; use a cheap proxy metric (here the runtime verifier on a small fixture of cycles) as the fitness; dedup equivalent candidates (FEC analog) to avoid re-spending the verifier. The "do not tune the meta-params, decide them small and freeze" discipline is a cheap-search hygiene worth copying.
- **Research contribution (do NOT reframe as ours)**: discovering complete ML algorithms from scratch is a Google-Brain-scale research program. The repo is not doing AutoML-Zero; it would at most run a tiny, verifier-grounded recipe search.
- **What to avoid / overclaim**: do not claim a small evolutionary loop will "discover" a social-WAM the way AutoML-Zero discovered backprop. The 1-in-10^12 sparsity result is the warning: a generic search over a social-prediction program space is astronomically expensive and the repo does not spend that compute. The repo's edge is a near-$0 verifier signal on a narrow space, not a big generic search.
