Source of truth / 기준 저장소

Actor workspace now carries owned runtime state actor workspace가 actor 소유 상태를 담는다

The runtime reads actor-owned active action skills from actor workspace records, persists actor profiles, and writes relationship edges from evidence-backed verification events.

runtime은 actor workspace record에서 actor-owned active action skill을 읽고, actor profile을 저장하며, 검증 evidence에서 relationship edge를 갱신한다.

Design decision / 설계 결정: current source has two action skill worlds. Canonical seed action skills live under probe/src/gameplay/seedSkills/ and are attached to actors through skills/ownership.ts. Exploratory generated TypeScript action skills are still saved under build/generated-skills by skillVillageCli.ts, not under data/actors/<actor>/action-skills/. For the current architecture, actor workspace is the only source of truth for actor-owned action skill lifecycle state. Legacy generated output must be archived, disabled, or migrated into validated actor workspace candidates before runtime selection can see it. 현재 source에는 두 개의 action skill 세계가 있다. canonical seed action skill은 probe/src/gameplay/seedSkills/에 있고 skills/ownership.ts를 통해 actor에 붙는다. 반면 exploratory generated TypeScript action skill은 아직 actor workspace가 아니라 build/generated-skills에 저장된다. 현재 아키텍처에서는 actor workspace가 actor-owned action skill lifecycle state의 유일한 source of truth다. legacy generated output은 runtime selection이 보기 전에 archived, disabled, 또는 validated actor workspace candidate로 migration되어야 한다.

Voyager model / Voyager 모델

Single-agent long-horizon autonomy 단일 agent의 장시간 자율 수행

Voyager can spend a long time planning, failing, generating code, retrying, and expanding its skill library because its goal is eventual task completion without social responsiveness.

Voyager는 사회적 반응성이 아니라 “언젠가 목표를 달성하는 것”이 핵심이다. 그래서 planning, failure, code generation, retry, skill library 확장에 긴 시간을 써도 구조적으로 문제가 덜하다.

This project / 이 프로젝트

Bounded turns with social pressure 사회적 압력이 있는 bounded turn

A social simulation seed needs actors to stay observable and responsive. Even without strict realtime, one NPC cannot block the world while a critic, generator, or long reflection loop runs.

우리 목표는 social simulation seed다. 완전한 realtime이 아니어도 actor가 관찰 가능하고 반응 가능해야 한다. 한 NPC가 critic, generator, 긴 reflection loop 때문에 world 전체를 막아서는 안 된다.

Design rule / 설계 규칙: keep the gameplay hot path small and bounded. Critic, reflection, and action-skill improvement should run asynchronously from immutable evidence and write suggestions into actor workspace, not block the current turn. gameplay hot path는 작고 bounded해야 한다. critic, reflection, action skill improvement는 immutable evidence를 읽는 비동기 작업이어야 하며, 현재 turn을 막지 않고 actor workspace에 suggestion/proposal을 써야 한다.
Per-NPC review ownership / NPC별 review ownership: each actor should have its own reviewer sidecar. A global reviewer can summarize cross-actor patterns, but it must not own actor memory, actor action-skill lifecycle, or actor-specific repair proposals. 각 actor에는 자기 reviewer sidecar가 하나씩 붙어야 한다. global reviewer는 actor 간 패턴을 요약할 수는 있지만, actor memory, actor action-skill lifecycle, actor-specific repair proposal의 owner가 되어서는 안 된다.

Async critic ownership 비동기 critic의 ownership

The critic should be actor-scoped by default: reviewer A reads actor A transcript slices, actor A evidence, provider snapshots for actor A, and actor A Langfuse traces. It writes review notes and candidate recipes under actor A workspace. The same pattern applies independently to every NPC.

critic은 기본적으로 actor-scoped여야 한다. reviewer A는 actor A의 transcript slice, actor A evidence, actor A provider snapshot, actor A Langfuse trace를 읽고 actor A workspace 아래 review note와 candidate recipe를 쓴다. 모든 NPC에 같은 구조가 독립적으로 붙는다.

A global reviewer is allowed only as a summarizer over already-written per-actor reviews. It can say "multiple NPCs drift away after failed log targeting," but the repair proposal still belongs to each affected actor workspace unless the team intentionally promotes a shared seed fix.

global reviewer는 이미 작성된 actor별 review를 요약하는 역할까지만 허용된다. 예를 들어 “여러 NPC가 log targeting 실패 뒤 멀리 이동한다”는 cross-actor pattern은 말할 수 있지만, repair proposal은 각 affected actor workspace에 남아야 한다. shared seed fix로 승격하려면 별도 의도적 promotion이 필요하다.

LLM Message Structure LLM 메시지 구조

Stable gameplay loop / 안정 gameplay loop: the default provider is deterministic, so no external LLM request is sent. The in-process provider input now carries { observation, lastResult, currentTask, goalStack, activeActionSkillContext, actorProviderContext } when actor workspace artifacts are enabled. This matters because actor workspace visibility cannot be judged from stable runProbe LLM prompts; there are none. 기본 provider는 deterministic이므로 외부 LLM request는 전송되지 않는다. actor workspace artifact가 켜진 경우 process 내부 provider input은 { observation, lastResult, currentTask, goalStack, activeActionSkillContext, actorProviderContext }를 담는다. 따라서 안정 runProbe 경로의 LLM prompt를 보고 actor workspace visibility를 판단할 수 없다. 그 prompt 자체가 없기 때문이다.

Actor Profiles, Goals, Relationships actor profile, goal, 관계 상태

Persona text is no longer the behavioral source of truth. The runtime uses a canonical ActorProfile, a categorical ActorGoalStack, and directional RelationshipEdge records. Numeric scores are projections from enum categories, not stored freeform values like trust: 0.73.

persona 문장은 행동의 source of truth가 아니다. runtime은 canonical ActorProfile, category 기반 ActorGoalStack, 방향성이 있는 RelationshipEdge record를 쓴다. 숫자 score는 enum category에서 파생되는 값이지 trust: 0.73 같은 임의 float가 아니다.

Ownership gate / ownership gate: a seed action skill is assigned to an actor only when it is implemented, the actor role is valid for it, and every primitive in the skill is allowed by that role's contract. This means the implemented/planned label alone is not enough; role permissions can still prevent ownership. seed action skill은 implemented이고, actor role이 해당 skill에 유효하며, skill의 모든 primitive가 그 role contract에서 허용될 때만 actor에 배정된다. 즉 implemented/planned label만으로는 충분하지 않고, role permission이 ownership을 막을 수 있다.

Expansion rule 확장 규칙

A new actor-owned action skill should not be generated code that immediately runs in the hot loop. It should first be a bounded recipe over existing trusted primitives. Generated TypeScript can be considered later as a reviewed patch-like artifact, but the actor workspace lifecycle should still hold the proposal, evidence, promotion, supersession, and retirement records.

새 actor-owned action skill은 hot loop에서 즉시 실행되는 generated code가 되어서는 안 된다. 먼저 신뢰 가능한 기존 primitive 위의 bounded recipe여야 한다. generated TypeScript는 나중에 reviewed patch 같은 artifact로 검토할 수 있지만, proposal, evidence, promotion, supersession, retirement record는 actor workspace lifecycle이 소유해야 한다.
